# ADR-043: Memory System Derived-Data GDPR Position

## Status

Proposed

## Context

### System Overview

The Rootweaver long-term memory system ingests Claude Code session transcripts and produces six categories of derived data across multiple storage backends:

1. **Rounds** (Postgres) — verbatim user and assistant message text, stored as structured rows with session foreign keys.
2. **Facts** (Postgres) — LLM-extracted decisions, problems, and learnings distilled from rounds. Each fact references its source round(s) but contains synthesised natural-language text, not verbatim quotes.
3. **Artifacts** (Postgres) — structured references to external objects created during sessions: git commit SHAs, Jira issue keys, file paths. Metadata-heavy, low free-text content.
4. **pgvector embeddings** (Postgres) — 1024-dimensional dense vectors generated from rounds and facts via Qwen3-Embedding-0.6B. Stored alongside their source row foreign keys.
5. **FalkorDB graph nodes and edges** — entities (Person, Project, Decision, Technology) and relationships extracted from facts and rounds, forming a knowledge graph queryable via Cypher.
6. **Assistant thinking content** (Postgres) — extended thinking blocks from Claude responses, stored in a separate redacted-access table for audit and debugging purposes only. Contains model reasoning traces that may reference user-provided personal data.

### Regulatory Landscape

GDPR Article 17 ("right to erasure") grants data subjects the right to request deletion of personal data. The question of whether derived and transformed data constitutes "personal data" under GDPR has been actively contested:

- **Hamburg DPA position (July 2024)**: The Hamburg Commissioner for Data Protection argued that parameters within trained LLM weights do not constitute personal data because individual data points cannot be meaningfully extracted from the model. This position was influential but non-binding and narrowly scoped to model weights, not to structured derived stores.

- **EDPB Opinion 28/2024 (December 2024)**: The European Data Protection Board rejected the Hamburg position's broad implications. EDPB held that if personal data can be extracted from a system — regardless of the transformation applied — the data remains personal data subject to GDPR. This opinion carries significant weight as EDPB guidance shapes enforcement across all EU member states.

- **OWASP AI Security Verification Standard (AISVS) C08**: Mandates hard-delete capability plus tombstoning for right-to-be-forgotten (RTBF) compliance in vector stores and memory systems. Specifically requires that deletion propagate to all derived representations, not merely the source record.

### Project Posture

The Rootweaver project owner (sole operator of private infrastructure) is not personally subject to GDPR enforcement — the system runs on private hardware, processes only the owner's data, and has no current EU data subjects beyond the owner. However, the project is planned for public release as an open-source platform. Downstream deployers operating within EU jurisdiction, or processing data of EU residents, will face full GDPR obligations. Architectural decisions made now determine whether GDPR compliance is achievable without a rewrite.

### The Core Tension

The six data layers form a derivation chain: transcripts → rounds → facts → embeddings/graph. Deleting only the source (rounds) while retaining derived representations (facts, embeddings, graph nodes) creates a situation where personal data persists in transformed form. Under EDPB Opinion 28/2024, if a fact reads "Ryan decided to migrate from ChromaDB to Qdrant on 2026-02-10," the name, decision, and date constitute personal data regardless of the derivation step that produced them. The embedding of that fact, while not human-readable, encodes the semantic content and may be reversible to approximate the original text via nearest-neighbour reconstruction attacks (Song & Raghunathan, 2020; Carlini et al., 2021).

## Decision

We adopt **EDPB Opinion 28/2024 as the governing interpretation** and implement cascading hard-delete across all six data layers, with the following specific positions:

### 1. All Six Layers Are In-Scope for Erasure

When a deletion request targets a session, user, or date range, the system MUST delete:

- **Rounds**: Hard-delete rows from Postgres. This is unambiguous — verbatim text is personal data.
- **Facts**: Hard-delete all facts whose provenance chain traces to the targeted rounds. Facts contain synthesised text that names individuals, references decisions, and encodes context — personal data under EDPB.
- **Artifacts**: Hard-delete artifact rows. While a git SHA is arguably not personal data in isolation, the associated metadata (who committed, when, what Jira issue, what file paths) constitutes personal data in combination. We delete the artifact record; we do NOT rewrite git history or delete Jira issues (those are governed by their own platforms' retention policies).
- **pgvector embeddings**: Hard-delete embedding rows whose foreign keys reference deleted rounds or facts. Embeddings are not practically reversible to exact source text, but (a) EDPB's standard is "extraction possible," and embedding inversion research demonstrates approximate reconstruction; (b) retaining embeddings of deleted text violates the spirit of erasure; (c) orphaned embeddings without source text serve no retrieval purpose anyway.
- **FalkorDB graph nodes and edges**: Delete nodes and edges whose provenance metadata traces to deleted sessions or rounds. Where a node aggregates information from multiple sessions (e.g., a Technology node for "Qdrant" referenced by many sessions), we remove edges from the deleted session but retain the node if other provenance remains. If all provenance is deleted, the node is deleted.
- **Assistant thinking content**: Hard-delete. This layer exists for audit only and may contain verbatim reproductions of user input within reasoning traces.

### 2. Tombstoning for Consistency

After deletion, a tombstone record is written to a `deletion_log` table recording:

- Deletion request ID (UUID)
- Timestamp of deletion
- Scope (session IDs, user identifier, date range)
- Layers affected and row counts deleted
- Requestor identity

Tombstones contain NO residual personal data — only the scope identifier and counts. This satisfies OWASP AISVS C08's tombstoning requirement and provides an audit trail proving deletion occurred, which is itself a GDPR accountability obligation (Article 5(2)).

### 3. Deletion Propagation Order

To avoid referential integrity violations and ensure atomicity:

```
1. FalkorDB graph edges (no FK constraints, delete first to avoid orphans)
2. FalkorDB graph nodes (where all provenance removed)
3. pgvector embeddings (FK to rounds/facts)
4. Facts (FK to rounds)
5. Artifacts (FK to sessions)
6. Assistant thinking (FK to rounds)
7. Rounds (source of truth)
8. Write tombstone record
```

Steps 3–7 execute within a single Postgres transaction. FalkorDB operations (steps 1–2) are best-effort with retry; if graph deletion fails, the tombstone records partial completion and a reconciliation job retries. This is acceptable because FalkorDB graph data is supplementary (query enrichment), not the system of record.

### 4. Retention Defaults

For the open-source release, the default configuration ships with:

- **No automatic retention limit** — the deployer must configure retention policy appropriate to their jurisdiction.
- **Deletion API endpoint** exposed on the memory service: `DELETE /api/v1/memory/sessions/{session_id}` and `DELETE /api/v1/memory/user/{user_id}`.
- **Bulk deletion by date range**: `DELETE /api/v1/memory/purge?before=YYYY-MM-DD`.
- **Documentation** in the deployment guide explicitly stating that EU deployers MUST configure retention policies and expose a mechanism for data subjects to request erasure.

### 5. Private Instance (Project Owner)

For the current private deployment, we implement the same deletion machinery but do not enforce automatic retention. The owner may run manual purges as desired. The value is architectural readiness — the system is built correctly now rather than retrofitted later.

## Consequences

### Positive

- **GDPR-ready for public release**: Downstream deployers in EU jurisdictions can satisfy Article 17 obligations without forking or rewriting the memory system.
- **Alignment with strongest regulatory interpretation**: By following EDPB rather than the more permissive Hamburg position, the project is robust against regulatory tightening. If Hamburg's view prevails in future enforcement, we lose nothing — deletion capability is strictly additive.
- **OWASP AISVS C08 compliance**: The tombstoning and hard-delete architecture satisfies the security verification standard, which is increasingly referenced in enterprise procurement evaluations.
- **Clean data lifecycle**: Cascading deletion prevents orphaned embeddings, dangling graph edges, and facts referencing nonexistent rounds — improving data quality independent of compliance motivations.
- **Trust signal for adoption**: Open-source projects with explicit GDPR positions and built-in deletion tooling see higher enterprise adoption in regulated industries.

### Negative

- **Implementation complexity**: Cascading deletion across four storage backends (Postgres, pgvector, FalkorDB, and the thinking-content table) requires careful transaction coordination. The FalkorDB leg is eventually-consistent, adding reconciliation job complexity.
- **Graph degradation**: Aggressive deletion of graph nodes whose provenance is fully removed may fragment the knowledge graph over time, reducing query quality for remaining data. Mitigation: graph compaction job that re-evaluates connectivity after bulk deletions.
- **No "anonymise instead of delete" path**: We chose hard-delete over anonymisation (see Alternatives). This means deleted sessions are irrecoverable — there is no "undo." For a personal knowledge system, this is acceptable; for multi-tenant deployments, it may be too aggressive. Downstream deployers can layer anonymisation on top if preferred.
- **Embedding deletion is conservative**: Deleting embeddings that may not be practically reversible is a cost paid for regulatory safety. The cost is low (re-embedding surviving content is cheap) but nonzero.
- **Performance impact of cascading deletes**: Bulk deletion (e.g., purge 6 months of data) will require batched execution to avoid long-running transactions and table locks. Must be designed as a background job, not a synchronous API call.

## Alternatives Considered

### Alternative 1: Hamburg Position — Embeddings and Graph Are Not Personal Data

Under Hamburg DPA's July 2024 interpretation, only the verbatim rounds and directly-identifying facts would require deletion. Embeddings and graph nodes, as transformed representations from which exact source text cannot be reliably recovered, would be retained.

**Why rejected**: The EDPB explicitly repudiated this interpretation in Opinion 28/2024. While Hamburg's position has academic support (particularly for model weights, which are far more transformed than our structured graph nodes), building on a rejected regulatory position creates legal risk for downstream deployers. Additionally, our graph nodes are not analogous to model weights — they contain human-readable entity names and relationship labels that plainly constitute personal data. A FalkorDB node labelled `(:Person {name: "Ryan Duffy"})` is not a diffuse statistical representation; it is a structured record.

### Alternative 2: Anonymise Rather Than Delete

Replace personal identifiers in facts, graph nodes, and artifact metadata with pseudonyms or placeholder tokens (e.g., "User_001") rather than deleting entire records. This preserves the knowledge graph structure and embedding space while removing identifying information.

**Why rejected**: (a) Anonymisation of free-text is unreliable — NER-based scrubbing misses oblique references, contextual identifiers, and rare names at a rate that varies by language and domain. Our facts contain highly specific technical context ("Ryan decided to use HAMi for GPU sharing on the K3s cluster") where removing "Ryan" still leaves a quasi-identifier given the project's uniqueness. (b) GDPR Article 17 grants a right to erasure, not a right to anonymisation. While anonymised data falls outside GDPR scope, the act of anonymisation must be provably complete — a bar that is difficult to meet for unstructured text. (c) Embeddings of anonymised text still encode semantic proximity to the original; the anonymisation would need to extend to re-embedding, at which point the cost advantage over deletion vanishes.

### Alternative 3: Soft-Delete with Access Controls

Mark records as deleted (a `deleted_at` timestamp) without physically removing them. Enforce access controls so that deleted records are excluded from all query paths. Physically purge on a deferred schedule (e.g., 30 days).

**Why not adopted as primary**: Soft-delete is a reasonable operational convenience (preventing accidental irreversible loss) but does not satisfy Article 17 on its own — the data still exists and is accessible to system administrators. We considered a hybrid: soft-delete for a 7-day grace period, then hard-delete via background job. This adds complexity for marginal benefit in a single-operator system. **Decision**: we implement hard-delete as the API contract. Deployers who want a soft-delete grace period can implement it in their application layer before calling our deletion API. The memory system itself does not retain data after deletion is confirmed.
