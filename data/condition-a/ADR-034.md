# ADR-034: Data Source Connector Architecture

## Status

Proposed

## Context

### Current State

Rootweaver ingests knowledge from a single source: the local Obsidian vault. The pipeline is event-driven — inotify watches the vault filesystem, emits change events to Kafka, and downstream consumers chunk, embed, and upsert documents into Qdrant. This gives sub-second indexing latency for vault changes and works well for its single-source scope.

Jira and Confluence are accessible via MCP tools (Atlassian MCP server), but these are live-query integrations — they execute API calls at query time and return results directly to the LLM. The data is never normalised, chunked, embedded, or stored in Qdrant. This means Jira and Confluence content cannot participate in vector search, hybrid BM25+vector retrieval, temporal filtering, or cross-source correlation.

### The Gap

The single-connector architecture creates a hard ceiling on the questions Rootweaver can answer. Consider:

- **Cross-source correlation**: "What Jira issues relate to search pipeline changes last week?" requires both vault documents and Jira issues in the same retrieval index. Today this needs two separate tool calls with manual reconciliation by the LLM, which is unreliable and token-expensive.
- **Unified timeline**: Temporal queries like "what happened in the last sprint?" should surface vault implementation reports, Jira transitions, Confluence page updates, and git commits in a single ranked result set. Currently only vault documents have temporal retrieval support.
- **Semantic search over structured sources**: Jira issues have rich text descriptions, comments, and linked context that would benefit from vector search. MCP live-query returns exact-match JQL results, missing semantic neighbours.
- **Staleness**: MCP live-query has zero pre-computation. Every query pays full API latency. For Jira searches this is 800ms-2s per call; for Confluence content retrieval, 1-3s. Pre-indexed content serves in <100ms from Qdrant.

### Competitive Landscape

Enterprise knowledge platforms have converged on a connector-based architecture:

- **Glean**: 100+ connectors, enterprise knowledge graph, unified search across all SaaS tools. Proprietary, $30+/user/month.
- **Onyx (formerly Danswer)**: 40+ open-source connectors, hybrid search (vector + keyword), polling-based ingestion with configurable intervals. Apache 2.0 licensed.
- **Rootweaver today**: 1 connector (vault/inotify) + 2 MCP live-query integrations (Jira, Confluence). No unified index.

Onyx's connector model is instructive. Each connector implements a `BaseConnector` interface with two methods: `load_from_state(checkpoint)` for incremental sync and `poll_source()` for periodic full/delta polling. Documents are normalised into a `Document` model with typed `Section` objects (title, text, link, metadata). A scheduler polls each connector at configured intervals, and a shared pipeline handles chunking, embedding, and upserting. This design has scaled to 40+ sources with consistent quality.

### Requirements

1. Support incremental ingestion from Jira, Confluence, Git (GitLab), and the existing vault source
2. Normalise all sources into a common document model compatible with the existing Qdrant schema
3. Preserve the existing vault inotify pipeline (low-latency, event-driven) — do not regress it to polling
4. Enable cross-source retrieval in a single query
5. Support source-specific metadata (Jira status, priority, sprint; Confluence space, labels; Git branch, author)
6. Minimise operational complexity — Rootweaver runs on a single-node K3s cluster, not a fleet

## Decision

We will implement a **connector framework** with a common interface, a shared ingestion pipeline, and a unified document model. The design draws from Onyx's proven patterns while adapting to Rootweaver's single-node K3s deployment and existing Kafka infrastructure.

### 1. Connector Interface

Each data source implements a `BaseConnector` abstract class:

```python
class BaseConnector(ABC):
    source_type: str                    # e.g. "jira", "confluence", "vault", "gitlab"
    
    @abstractmethod
    async def poll(self, checkpoint: SourceCheckpoint) -> ConnectorResult:
        """Fetch new/updated documents since checkpoint. Returns documents + new checkpoint."""
        ...

    @abstractmethod
    async def health_check(self) -> HealthStatus:
        """Verify connectivity and credentials."""
        ...
```

`SourceCheckpoint` is source-specific state (e.g., Jira's `updated > '2026-04-15 10:00'` JQL timestamp, Confluence's `lastModified` cursor, GitLab's commit SHA). Checkpoints are persisted in Postgres to survive restarts.

The existing vault connector is a special case: it remains event-driven (inotify → Kafka), but implements the same `BaseConnector` interface for health checks, metadata, and document normalisation. It does not use the polling scheduler.

### 2. Unified Document Model

All connectors emit a common `ConnectorDocument`:

```python
@dataclass
class ConnectorDocument:
    source_type: str                     # "jira", "confluence", "vault", "gitlab"
    source_id: str                       # unique within source (e.g., "VW-123", page ID, file path)
    canonical_id: str                    # globally unique: f"{source_type}:{source_id}"
    title: str
    sections: list[DocumentSection]      # typed content blocks
    url: str | None                      # web-accessible link
    created_at: datetime
    updated_at: datetime
    metadata: dict[str, Any]             # source-specific (sprint, labels, priority, space, branch...)
    access_control: list[str] | None     # future: permission groups
```

`DocumentSection` has a `section_type` enum (TITLE, BODY, COMMENT, DESCRIPTION, CODE, ATTACHMENT_TEXT) so downstream processing can weight or filter by section type. This mirrors Onyx's typed sections approach but with our own taxonomy.

The `canonical_id` format (`source_type:source_id`) ensures deduplication across sources. If a Jira issue is referenced in a vault document, the retriever can identify both as related without duplication.

### 3. Ingestion Pipeline

All connectors feed into the same pipeline:

```
Connector.poll()
    │
    ▼
Kafka topic: connector.documents.raw    (ConnectorDocument serialised as JSON)
    │
    ▼
Chunking Consumer                        (reuses existing vault chunking logic)
    │  - Source-aware chunking strategies (Jira comments as individual chunks,
    │    Confluence pages by heading, vault by markdown sections)
    │
    ▼
Kafka topic: connector.chunks.embedded   (chunks with embeddings)
    │
    ▼
Qdrant Upsert Consumer                   (upserts to rootweaver_vault collection)
    │  - Stores source_type, source_id in payload metadata
    │  - Enables filtered retrieval by source
    │
    ▼
FalkorDB Consumer (optional)             (updates document_knowledge graph with
                                          cross-source relationships)
```

The vault inotify path continues to emit directly to the existing Kafka topic (`vault.file.events`). A thin adapter translates vault events into `ConnectorDocument` format before they enter the shared pipeline. This preserves sub-second vault indexing latency while gaining unified document normalisation.

### 4. Polling Scheduler

A lightweight Prefect-based scheduler manages connector polling:

| Source     | Default Interval | Rationale |
|------------|-----------------|-----------|
| Jira       | 5 minutes       | Issue updates are frequent during work hours |
| Confluence | 15 minutes      | Page edits are less frequent |
| GitLab     | 10 minutes      | Commit/MR activity is bursty |
| Vault      | Event-driven    | inotify, no polling needed |

Intervals are configurable via Unleash feature flags, allowing runtime tuning without redeployment. Each poll run persists its checkpoint to Postgres, enabling clean recovery after pod restarts.

### 5. Retrieval Integration

The existing RAG retriever gains a `source_filter` parameter:

- `source_filter=None` (default): search all sources — enables cross-source correlation
- `source_filter=["jira", "vault"]`: restrict to specific sources
- Exposed via MCP bridge as an optional parameter on `auto_search_vault` and `hybrid_search_vault`

Source metadata is returned in search results, allowing the LLM to attribute findings ("from Jira VW-123" vs "from vault implementation report").

### 6. Implementation Phasing

- **Phase 1 (VW-TBD)**: Connector interface + unified document model + Jira connector. Jira is the highest-value second source (most cross-referenced with vault content).
- **Phase 2**: Confluence connector. Leverages same Atlassian auth.
- **Phase 3**: GitLab connector (commits, MRs, pipeline status).
- **Phase 4**: Cross-source graph relationships in FalkorDB (e.g., Jira issue → vault doc → git commit).

Each phase ships independently and adds value incrementally. The existing MCP live-query tools remain available as fallback during the transition and for operations that need real-time data (e.g., transitioning a Jira issue).

## Consequences

### Positive

- **Cross-source retrieval**: A single query can surface vault docs, Jira issues, and Confluence pages in one ranked result set. "What happened with the search pipeline last week?" returns implementation reports, related tickets, and design docs together.
- **Semantic search over Jira/Confluence**: Vector similarity finds related issues even when JQL keyword matching would miss them. "Memory leak incidents" finds issues titled "OOM in rag-gateway" without exact keyword overlap.
- **Temporal coherence**: All sources share the same `updated_at` field in Qdrant, enabling unified temporal filtering across sources.
- **Reduced query-time latency**: Pre-indexed Jira/Confluence content serves from Qdrant in <100ms vs 800ms-3s for live API calls.
- **Extensibility**: Adding a new source requires implementing one class with two methods. The pipeline, chunking, embedding, and storage are shared infrastructure.
- **Competitive parity**: Moves from 1 connector to 4, closing the gap with Onyx's model while keeping operational simplicity.

### Negative

- **Operational complexity**: Four connector polling loops, checkpoint persistence, and source-specific error handling add moving parts to a single-node cluster.
- **Data freshness lag**: Polling intervals mean Jira/Confluence data can be up to 5-15 minutes stale. The vault remains real-time via inotify. For time-critical queries, MCP live-query remains available.
- **Qdrant collection growth**: Jira and Confluence content will significantly increase the document count in `rootweaver_vault`. Rough estimate: ~5,000 Jira issue chunks + ~3,000 Confluence page chunks added to the current ~148,000 vault chunks. Manageable on current hardware but worth monitoring.
- **API rate limits**: Atlassian Cloud REST APIs have rate limits (basic: ~100 requests/minute). Connector polling must respect these with backoff and batching. GitLab's rate limit is more generous but still requires consideration.
- **Credential management**: Each connector needs API tokens/OAuth credentials. Currently managed via environment variables and K8s secrets, but the surface area increases.
- **Chunking strategy complexity**: Jira issues have a different structure (summary + description + comments + changelog) than vault markdown or Confluence pages. Source-aware chunking adds conditional logic to the pipeline.

### Risks

- **Scope creep**: The connector framework could become an integration platform. Mitigated by strict phasing and the rule that each connector must justify its inclusion by demonstrable cross-source query value.
- **Stale checkpoint recovery**: If a connector's checkpoint becomes corrupted or the source's data changes retroactively (e.g., Jira bulk import), a full re-sync mechanism is needed. Phase 1 should include a `--full-sync` flag per connector.

## Alternatives Considered

### Alternative 1: Expand MCP Live-Query (Status Quo+)

**Approach**: Keep Jira and Confluence as live-query-only via MCP tools. Improve the LLM's ability to synthesise across tool calls by adding a "multi-source query" MCP tool that fans out to vault search + Jira JQL + Confluence CQL in parallel and merges results.

**Pros**:
- Zero new infrastructure. No polling, no checkpoints, no additional Kafka topics.
- Always fresh — every query hits the live API.
- Simpler operational model — fewer pods, fewer failure modes.

**Cons**:
- No vector similarity for Jira/Confluence — limited to JQL/CQL keyword matching.
- Query latency scales with source count (3 parallel API calls + merge = 2-4s minimum).
- LLM must reconcile heterogeneous result formats in-context, consuming tokens and introducing synthesis errors.
- No temporal filtering for Jira/Confluence (JQL `updated` is keyword-based, not the same as Qdrant's date-range vector filtering).
- Fundamentally cannot do cross-source ranking — results from different sources have no comparable relevance score.

**Why rejected**: This approach has a hard ceiling. It can improve latency with parallelism and improve synthesis with better prompting, but it cannot achieve unified semantic search or cross-source ranking. These are architectural limitations, not implementation gaps.

### Alternative 2: Direct Qdrant Ingest (No Kafka)

**Approach**: Each connector directly chunks, embeds, and upserts to Qdrant without going through Kafka. A simpler pipeline: `Connector.poll() → chunk → embed → Qdrant.upsert()`.

**Pros**:
- Fewer moving parts — no additional Kafka topics or consumers.
- Lower latency from poll to indexed (skip Kafka serialisation/deserialisation).
- Simpler debugging — one process per connector, linear execution.

**Cons**:
- Loses Kafka's durability guarantees — if the connector crashes mid-batch, partially-embedded documents may be lost or duplicated.
- No replay capability — Kafka topics allow re-processing from offset if the embedding model changes or chunking logic is updated.
- Diverges from the existing vault pipeline pattern, creating two distinct ingestion architectures to maintain.
- Cannot feed the FalkorDB graph consumer or future consumers without duplicating the work.
- Embedding is the bottleneck (~50ms/chunk on CPU via Ollama); doing it synchronously in the connector process blocks polling.

**Why rejected**: The existing Kafka infrastructure is already deployed and operational. Using it for connectors adds marginal cost (new topics) but significant value (durability, replay, fan-out to multiple consumers). The direct-ingest approach trades resilience for simplicity we do not need — Kafka is already paid-for infrastructure.

### Alternative 3: Federated Search (Query-Time Fan-Out Without Indexing)

**Approach**: At query time, the RAG gateway fans out the query to multiple backends: Qdrant for vault content, Jira REST API for issues, Confluence REST API for pages. Results are normalised, re-ranked using a cross-encoder, and returned as a unified set.

**Pros**:
- Always fresh — no synchronisation lag.
- No storage growth in Qdrant.
- Source-native query capabilities preserved (JQL for Jira, CQL for Confluence).

**Cons**:
- Query latency is bounded by the slowest source (Confluence API: 1-3s).
- Re-ranking across heterogeneous result formats (vector scores vs API relevance) is an unsolved problem without a shared embedding space.
- Cannot do vector similarity on Jira/Confluence text — API search is keyword-based.
- Every query pays API cost, even for frequently-accessed content.
- Rate limit pressure scales with query volume, not data volume.

**Why rejected**: Federated search is architecturally elegant but practically limited by the inability to embed external source content. Without embeddings, cross-source ranking is heuristic at best. This approach is viable as a complementary layer (and the existing MCP tools effectively provide it) but insufficient as a primary architecture.
