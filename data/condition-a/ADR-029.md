# ADR-029: KEDA-Based GPU Swap Controller for Embed/Inference Coexistence

## Status

Accepted

## Context

The Rootweaver platform runs on a single-node K3s cluster with one NVIDIA RTX 4080 (16 GB VRAM). Three workloads require GPU access:

1. **vLLM inference** — serves a quantized 8B parameter model (GGUF/AWQ) for RAG synthesis and agent orchestration. Resident VRAM footprint is approximately 15.6 GB with KV-cache overhead, effectively pinning the entire GPU.
2. **TEI (Text Embeddings Inference)** — runs Qwen3-Embedding-4B for two use cases: (a) nightly full-vault reindex (~4,200 documents, ~148K chunks) and (b) on-demand delta ingestion triggered by the Jira connector, file-watcher Kafka events, and manual `vault-indexer` runs. Transient VRAM requirement is approximately 8 GB.
3. **Code-graph embedder** — generates embeddings for the PDG (Program Dependency Graph) code-knowledge FalkorDB graph. Runs on-demand after repository changes, similar resource profile to TEI.

The fundamental constraint is that vLLM and TEI cannot coexist on 16 GB VRAM. vLLM allocates the full GPU slice at startup via PyTorch's CUDA memory allocator (`gpu_memory_utilization=0.95` default), leaving no room for a second model. HAMi (Heterogeneous AI computing Management Interface) fractional GPU sharing was evaluated and ruled out: HAMi's memory partitioning relies on CUDA MPS or time-slicing, neither of which prevents OOM when two models each want >8 GB on a 16 GB card. The arithmetic simply does not work.

### Historical workaround and its failure modes

Until now, the operator (Ryan) manually scaled TEI to 1 replica before embedding runs and back to 0 afterwards, with vLLM scaled down during the swap window. This manual choreography has two failure modes that have occurred in production:

- **Stale scale-down**: TEI left at 1 replica after an embedding run, vLLM cannot start, RAG gateway returns 503s for all synthesis requests until the operator notices and intervenes.
- **Cold-start delay on ad-hoc ingest**: The Jira connector (VW-288) produces embedding requests in bursts whenever tickets are synced. With TEI scaled to 0, these requests queue indefinitely. The operator must notice the queue, scale down vLLM, scale up TEI, wait for model load (~45 seconds), process the batch, then reverse. Round-trip is 3-5 minutes of manual intervention per burst.

The introduction of the Jira connector and file-watcher Kafka pipeline (VW-288, VW-152) has changed the embedding workload from a predictable nightly batch to an unpredictable burst pattern. Manual orchestration is no longer tenable.

### Requirements

1. **Automatic**: No operator intervention for routine embedding bursts.
2. **Safe**: vLLM and TEI must never run simultaneously on the GPU. VRAM contention causes CUDA OOM, which crashes both processes.
3. **Bounded latency**: Embedding requests should begin processing within 90 seconds of arrival (swap window budget).
4. **Inference availability**: vLLM downtime during swaps must be minimized and observable. RAG gateway should gracefully degrade (return cached results or queue requests) during swap windows.
5. **Observable**: Swap events, durations, and failures must be visible in Prometheus/Grafana.

## Decision

We will implement a **KEDA-based GPU swap controller** that uses Kafka consumer-group lag and a CronJob schedule as scaling triggers to automatically orchestrate the mutually exclusive lifecycle of vLLM and TEI on the single GPU.

### Architecture

```
Kafka topic: vault.embedding.requests
        │
        ▼
KEDA ScaledObject (TEI)
  trigger: kafka-lag > 0 OR cron(02:00 UTC nightly)
        │
        ▼
gpu-swap-controller (pre-scale hook / Job)
  1. kubectl scale deployment/vllm --replicas=0
  2. Wait for vLLM pod termination (VRAM freed)
  3. kubectl scale deployment/tei --replicas=1
  4. Wait for TEI readiness probe (model loaded)
  5. Signal: TEI ready
        │
        ▼
TEI processes embedding queue
        │
        ▼
KEDA scales TEI to 0 (lag=0, cooldown elapsed)
        │
        ▼
gpu-swap-controller (post-scale hook / Job)
  1. Confirm TEI pod terminated
  2. kubectl scale deployment/vllm --replicas=1
  3. Wait for vLLM readiness probe
  4. Signal: vLLM ready
```

### Key design choices

**KEDA as the event-driven autoscaler.** KEDA is already available in the cluster (used by Prefect workers) and natively supports Kafka consumer-group lag as a trigger. A `ScaledObject` targeting the TEI Deployment with `minReplicaCount: 0` and `maxReplicaCount: 1` gives us event-driven scale-from-zero. The Kafka trigger fires when `vault.embedding.requests` consumer group has lag > 0.

**CronJob trigger for nightly reindex.** The nightly full-vault reindex is not Kafka-driven (it is a Prefect flow that calls the indexer directly). A KEDA `Cron` trigger at `02:00 UTC` handles this, ensuring TEI is warm before the Prefect flow starts at `02:05 UTC`.

**Pre/post lifecycle hooks as Kubernetes Jobs.** KEDA does not natively support "scale down workload A before scaling up workload B." We implement this as two short-lived Jobs triggered by KEDA's `ScaledObject` lifecycle:

- **Pre-activation Job** (`gpu-swap-activate`): Scales vLLM to 0, polls until the vLLM pod is fully terminated and GPU memory is released (verified via `nvidia-smi` query), then exits successfully. KEDA's TEI ScaledObject has a `fallback` policy that waits for this Job's completion before proceeding.
- **Post-deactivation Job** (`gpu-swap-deactivate`): After KEDA scales TEI to 0 (cooldown period elapsed, lag remains 0), this Job scales vLLM back to 1 and waits for its readiness probe.

Both Jobs use a minimal Python image with `kubectl` and the `kubernetes` Python client. They are idempotent — running them when the target is already in the desired state is a no-op.

**Mutual exclusion via resource quota.** As a safety net, a `ResourceQuota` on the `rootweaver` namespace limits `nvidia.com/gpu` to 1. If both Deployments somehow have replicas=1 simultaneously, the second pod stays `Pending` rather than causing CUDA OOM.

**Cooldown and debounce.** KEDA's `cooldownPeriod: 300` (5 minutes) prevents flapping when embedding requests arrive in closely-spaced bursts. After TEI processes a batch and lag returns to 0, it remains warm for 5 minutes to absorb follow-on bursts. Only after 5 minutes of zero lag does KEDA scale TEI down and the deactivation Job restores vLLM.

**Observability.** Both Jobs emit Prometheus metrics via pushgateway:
- `gpu_swap_total{direction="to_tei|to_vllm"}` — counter
- `gpu_swap_duration_seconds{direction}` — histogram
- `gpu_swap_failures_total{direction,reason}` — counter
- Grafana alert: `gpu_swap_duration_seconds > 120` fires a warning; `> 300` fires critical.

**RAG gateway graceful degradation.** The RAG gateway already implements a circuit breaker on the vLLM backend (ADR-012). During swap windows, synthesis requests receive a 503 with `Retry-After: 90` header. The MCP bridge surfaces this as a tool error with a human-readable message. For the Claude Code user experience, this means a brief pause rather than a cryptic failure.

### Code-graph embedder

The code-graph embedder shares the same swap mechanism. Its requests are published to `vault.embedding.requests` with a `source: code_graph` header. TEI processes them identically — the embedding model is the same. No separate swap path is needed.

### Swap budget

| Phase | Duration |
|-------|----------|
| vLLM graceful shutdown | ~10s |
| VRAM release confirmation | ~5s |
| TEI pod scheduling + pull | 0s (image pre-pulled) |
| TEI model load (Qwen3-Embedding-4B) | ~45s |
| **Total to-TEI swap** | **~60s** |
| TEI graceful shutdown | ~5s |
| vLLM pod scheduling + model load | ~30s |
| **Total to-vLLM swap** | **~35s** |

Round-trip (vLLM down → TEI up → process → TEI down → vLLM up) for a small burst: approximately 100 seconds plus processing time. This is within the 90-second activation budget for embedding requests.

## Consequences

### Positive

- **Eliminates manual intervention.** Embedding bursts from the Jira connector, file-watcher, and nightly reindex are handled automatically. The operator is removed from the critical path.
- **Prevents VRAM contention.** Mutual exclusion is enforced at two levels: orchestrated swap sequence and ResourceQuota safety net. CUDA OOM from concurrent models is structurally impossible.
- **Bounded inference downtime.** vLLM is unavailable only during active embedding work plus swap overhead. For typical Jira connector bursts (50-200 documents), this is 2-5 minutes. The 5-minute cooldown means small bursts arriving close together are batched.
- **Observable.** Swap events, durations, and failures are tracked in Prometheus and alertable in Grafana, replacing the previous "operator notices 503s" detection method.
- **Extensible.** Additional GPU workloads (e.g., fine-tuning jobs, GLiNER batch NER) can be integrated into the same swap controller by publishing to the Kafka topic and respecting the mutual-exclusion protocol.

### Negative

- **Inference unavailability during swaps.** Any RAG synthesis request during a swap window gets a 503. For a nightly reindex of ~148K chunks at ~2,000 chunks/minute, TEI occupies the GPU for approximately 75 minutes. vLLM is down for that entire window plus swap overhead. Mitigation: nightly reindex is scheduled at 02:00 UTC (03:00 BST) when interactive usage is near zero.
- **Complexity.** Two Kubernetes Jobs, a KEDA ScaledObject, a ResourceQuota, and a Kafka topic form a distributed state machine. Failure in any component (Job crash, KEDA scaler lag, Kafka unavailable) can leave the system in a degraded state. Mitigation: both Jobs are idempotent, a `gpu-swap-watchdog` CronJob runs every 10 minutes and forces convergence to vLLM-active if no embedding work is pending.
- **Cooldown tradeoff.** The 5-minute cooldown delays vLLM restoration after embedding work completes. If a user sends a synthesis request at minute 3 of cooldown, they wait 2 minutes plus vLLM swap time (~35 seconds). Mitigation: cooldown is tunable; 5 minutes is the initial value based on observed Jira connector burst patterns (median inter-burst gap: 12 minutes).
- **Single point of failure on KEDA.** If the KEDA operator pod is down, TEI never scales up and embedding requests queue indefinitely. Mitigation: Prometheus alert on KEDA operator pod health; manual `kubectl scale` remains available as fallback.

## Alternatives Considered

### Alternative 1: Second GPU (hardware solution)

Adding a second GPU (e.g., RTX 3060 12GB for embeddings) would eliminate the coexistence problem entirely. Each workload gets dedicated hardware.

**Rejected because:**
- The desktop is a single-slot consumer motherboard. A second GPU requires either a PCIe riser (thermal and stability risk) or a hardware upgrade.
- Budget constraint: the platform is a personal project; a second GPU is a disproportionate spend for a workload that needs GPU for at most 2 hours per day.
- Does not solve the architectural problem — adding a third GPU workload in the future would recreate the same issue.

### Alternative 2: CPU-only embedding with Ollama

Run embeddings on CPU via Ollama (already deployed in CPU-only mode) using a smaller embedding model. This avoids GPU contention entirely.

**Rejected because:**
- CPU embedding throughput on the desktop's Ryzen 7 5800X is approximately 15 chunks/second with `nomic-embed-text` (768-dim). The nightly reindex of 148K chunks would take ~2.7 hours. With Qwen3-Embedding-4B on GPU via TEI, the same workload completes in ~75 minutes.
- Embedding quality difference: Qwen3-Embedding-4B (1024-dim) significantly outperforms smaller CPU-feasible models on the platform's benchmark suite (Run-38: +0.12 context precision, +0.08 MRR vs nomic-embed-text).
- On-demand Jira connector ingestion latency would increase from ~2 seconds/document to ~8 seconds/document, making interactive workflows sluggish.

However, CPU embedding remains the **fallback path**: if the swap controller fails and TEI cannot acquire the GPU, the indexer falls back to Ollama CPU embedding with a quality/speed penalty rather than dropping documents.

### Alternative 3: NVIDIA MPS (Multi-Process Service) time-slicing

NVIDIA MPS allows multiple CUDA processes to share a GPU via time-slicing, with configurable memory limits per process.

**Rejected because:**
- MPS memory pinning is advisory, not enforced. Setting `CUDA_MPS_PINNED_DEVICE_MEMORY_LIMIT` to 8GB per process does not prevent PyTorch from allocating beyond that limit — it crashes with CUDA OOM instead of gracefully queuing.
- vLLM's memory allocator (`gpu_memory_utilization`) pre-allocates a contiguous block at startup. Even with MPS limiting vLLM to 8GB, the quantized 8B model plus KV-cache does not fit in 8GB — inference quality degrades catastrophically due to KV-cache eviction.
- Time-slicing introduces latency jitter (context switches between processes) that would degrade both vLLM token-generation latency (p99 target: <200ms) and TEI embedding throughput.
- HAMi (the K8s device plugin for fractional GPU) wraps MPS and inherits all these limitations. Evaluated and rejected during VW-180.

### Alternative 4: Prefect-orchestrated swap (no KEDA)

Use a Prefect flow to orchestrate the swap sequence: a `pre_embed` task scales down vLLM, a `run_embed` task processes the queue, a `post_embed` task restores vLLM.

**Rejected because:**
- Only handles scheduled workloads (nightly reindex). Ad-hoc Jira connector bursts would still require a separate trigger mechanism.
- Prefect work queues add 10-15 seconds of scheduling overhead before the swap even begins.
- Couples the swap logic to Prefect's availability — if the Prefect server is down, embeddings and inference are both stuck.
- KEDA's native Kafka trigger provides event-driven scale-from-zero without an intermediary orchestrator, which is architecturally cleaner for the burst pattern.
