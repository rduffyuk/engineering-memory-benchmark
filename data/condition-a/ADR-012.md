# ADR-012: Journal Pipeline Quality Overhaul — Model Selection & E2A Architecture

## Context

The journal streaming pipeline converts Claude Code session telemetry into automated daily journal entries via Kafka. The pipeline's mechanical infrastructure is sound — OTEL events flow from sessions through Kafka topics, a consumer deserialises them, an LLM generates prose, and the output lands in the Obsidian vault. The problem is that every journal entry produced by this pipeline is fiction.

A manual audit cross-referencing JSONL session transcripts against the generated journal entries found a 100% fabrication rate. The entries read convincingly — correct voice, plausible structure, appropriate technical vocabulary — but the specific claims (files edited, commands run, decisions made, bugs encountered) are invented. This is not a partial accuracy problem; it is a total grounding failure.

### Root Cause Analysis

Four compounding factors produce this outcome:

**1. Input poverty (richness 2/10).** The OTEL events reaching the journal consumer carry session-level metadata — start time, end time, session ID, a model identifier — but almost nothing about what actually happened. No file paths, no tool invocations, no conversation snippets, no error messages, no git operations. The LLM receives a prompt that effectively says "write a journal entry for a Claude Code session that lasted 47 minutes" and must hallucinate all substance.

**2. Model mismatch.** DeepSeek R1 8B (the current generation model, served via vLLM on the RTX 4080) is a reasoning-optimised model trained primarily for mathematical and logical tasks. When given a sparse prompt and asked to produce narrative text, it "overhelps" — filling gaps with plausible but fabricated technical details. Internal testing measured a ~14.3% hallucination rate even on well-grounded prompts; on near-empty prompts the rate approaches 100% because there is nothing to be faithful to. The model is not defective; it is misapplied.

**3. Single-shot generation.** The current architecture is fire-and-forget: one LLM call, no verification, no comparison against source material. There is no step that checks whether claims in the output are attested by the input. A hallucinated file path or a fabricated error message passes through unchallenged.

**4. Excessive temperature.** Generation runs at temperature 0.7, which is appropriate for creative writing but counterproductive for factual summarisation. Higher temperature increases lexical diversity at the cost of factual reliability — exactly the wrong trade-off for a grounded extraction task.

### The Stakes

The journal pipeline is not an isolated feature. It is the first stage of an intended content cascade: session → journal → weekly digest → blog post → LinkedIn/social. Every downstream artifact inherits upstream errors. Publishing fabricated technical claims under a real name, even on a personal blog, is a reputational risk. The pipeline must produce content that is verifiably grounded in what actually happened, or it must not produce content at all.

### Constraints

- **GPU budget is fixed.** A single RTX 4080 (16 GB VRAM) serves vLLM for all inference. The journal pipeline cannot monopolise the GPU or require a second model to be loaded concurrently with the primary serving model.
- **Latency is not critical.** Journal generation is a batch process running after sessions end. Minutes of processing time are acceptable; hours are not.
- **The vault is the source of truth.** Journal entries land in the Obsidian vault and are indexed into Qdrant. Fabricated entries pollute the knowledge base and degrade RAG search quality — a second-order harm beyond the journal itself.
- **JSONL transcripts exist.** Full session transcripts are stored as `.jsonl` files in `~/.claude/projects/`. These contain every user message, assistant response, and tool invocation. They are rich (richness ~9/10) but large (100 KB to 16 MB per session). The pipeline currently does not use them.

## Decision

We adopt a three-part overhaul: **Extract-Anchor-Author (E2A)** architecture, model replacement, and generation parameter tightening.

### 1. Extract-Anchor-Author (E2A) Pipeline

Replace the current single-shot generation with a three-stage pipeline:

**Stage 1 — Extract.** A dedicated extraction step parses the JSONL session transcript (not OTEL events) and produces a structured fact sheet: files touched, tools invoked, git operations, errors encountered, key decisions, approximate time allocation. This is a mechanical transformation, not a generative task. It can be implemented as deterministic Python code (regex + JSON parsing) with no LLM involvement, or as an LLM call with a tightly constrained extraction prompt and temperature 0.0. The output is a JSON document — the "anchor."

**Stage 2 — Anchor.** The anchor document is passed to the generation stage as the sole source of factual claims. The generation prompt is restructured with explicit grounding instructions: "Use ONLY the facts in the following anchor document. Do not add technical details, file paths, error messages, or decisions not present in the anchor. If the anchor is sparse, produce a shorter entry rather than inventing content."

**Stage 3 — Author.** The LLM generates the journal entry from the anchor. Post-generation, a verification pass (either rule-based or a second LLM call) checks that every specific claim in the output (file paths, commands, error messages, metric values) appears in the anchor document. Claims that fail verification are flagged or stripped. The entry is tagged with a confidence score reflecting anchor coverage.

This architecture makes hallucination structurally difficult rather than relying on the model to self-regulate. The extraction stage is grounded by construction (it reads the transcript). The generation stage is constrained by the anchor. The verification stage catches leakage.

### 2. Model Selection

Replace DeepSeek R1 8B with **Qwen2.5 7B Instruct** (or the latest Qwen instruct variant available at implementation time) for journal generation.

Rationale:

- **Instruction-following over reasoning.** Qwen2.5 Instruct is trained for instruction adherence, including "do not add information not present in the context" style constraints. DeepSeek R1's reasoning training actively works against grounding — it is rewarded for producing novel inferences, which manifests as hallucination when the task requires strict extraction.
- **Size compatibility.** At 7B parameters with 4-bit quantisation (~4 GB VRAM), Qwen2.5 fits comfortably on the RTX 4080 alongside or time-shared with the primary vLLM model. The journal pipeline runs as a batch job, so it can be scheduled during low-inference windows.
- **Multilingual and long-context capable.** Qwen2.5 supports 128K context, sufficient to ingest large session transcripts or their extracted anchors without truncation.
- **Community benchmarks.** Qwen2.5 7B Instruct shows lower hallucination rates on summarisation benchmarks (TruthfulQA, HaluEval) than comparably-sized reasoning-focused models. It is not hallucination-free — no model is — but it is better matched to the task.

If Qwen2.5 proves insufficient, the fallback is **Mistral 7B Instruct v0.3**, which has similar characteristics but slightly weaker instruction adherence in empirical testing.

### 3. Generation Parameter Tightening

- **Temperature: 0.1** (down from 0.7). Near-deterministic output for factual summarisation. Slight temperature avoids degenerate repetition while heavily penalising creative deviation.
- **Top-p: 0.9** (unchanged, but now meaningful given lower temperature).
- **Repetition penalty: 1.1** to prevent the low temperature from causing loops on longer entries.
- **Max tokens: 2048** with a soft target of 800-1200 for a typical daily entry.

### Implementation Sequence

1. Build the JSONL extraction stage (Python, no LLM) — this unblocks everything else.
2. Swap the model to Qwen2.5 7B Instruct on vLLM (or Ollama CPU for initial testing).
3. Rewrite the generation prompt with anchor-grounding instructions.
4. Implement the post-generation verification pass (start rule-based; add LLM-based later if needed).
5. Run parallel generation (old pipeline vs E2A) on 10 historical sessions, manually score both, confirm improvement before cutover.

## Consequences

### Positive

- **Grounding by construction.** The anchor document eliminates the "empty prompt" failure mode. The LLM cannot hallucinate file paths if the anchor contains the real file paths, and the prompt forbids invention.
- **Verifiable output.** Every claim in the journal entry traces back to the anchor, which traces back to the transcript. This chain of provenance enables automated quality checks and manual audits.
- **Content cascade unblocked.** With trustworthy journal entries, the weekly digest and blog pipeline can proceed. Fabrication risk moves from "certain" to "low and detectable."
- **Vault quality preserved.** Journal entries indexed into Qdrant will contain real facts about real sessions, improving rather than degrading RAG search quality.
- **Model-appropriate deployment.** DeepSeek R1 8B continues serving its designed purpose (reasoning tasks). Qwen2.5 7B handles its designed purpose (instruction-following generation). Neither is misapplied.

### Negative

- **Increased pipeline complexity.** Three stages instead of one. More code to maintain, more failure modes (extraction parsing errors, anchor schema drift, verification false positives).
- **Increased latency.** Extraction + generation + verification will take longer than single-shot generation. Estimated 30-90 seconds per session versus ~10 seconds currently. Acceptable for batch, but rules out real-time use.
- **JSONL dependency.** The pipeline now depends on session transcript availability. If transcripts are purged, moved, or formatted differently by a Claude Code update, the extraction stage breaks. A schema contract or normalisation layer may be needed.
- **Shorter entries when anchors are sparse.** By design, sessions with little extractable content produce short or empty journal entries. This is correct behaviour (no data → no fabrication) but may feel like a regression if users expect verbose output regardless of input richness.
- **Two models on one GPU.** Even time-shared, running Qwen2.5 alongside the primary vLLM model requires careful scheduling. OOM risk during overlap windows must be mitigated by explicit model loading/unloading or Ollama CPU fallback.

## Alternatives Considered

### Alternative A: Enrich OTEL Events (Keep Current Architecture)

Instrument Claude Code sessions to emit richer OTEL events — file paths, tool names, git operations, error counts — so the existing single-shot pipeline has adequate input.

**Pros:**
- No architectural change to the journal pipeline.
- OTEL enrichment benefits other consumers (monitoring, tracing dashboards).
- Keeps the pipeline simple (one stage, one model).

**Cons:**
- Claude Code's OTEL emission is not under our control — it is governed by Anthropic's client implementation. Custom instrumentation would require wrapping or patching the Claude Code binary, which is fragile and breaks on updates.
- Even with richer events, single-shot generation with no verification still allows hallucination. The model can still embellish beyond what the events contain.
- Does not address the model mismatch (DeepSeek R1 for summarisation) or temperature issues.
- Timeline: weeks to months for meaningful OTEL enrichment, versus days for JSONL extraction.

**Verdict:** Valuable as a complementary improvement but insufficient alone. OTEL enrichment should happen eventually, but cannot be the primary fix because it does not address the generation architecture or model selection problems.

### Alternative B: Use Claude API Directly (Cloud LLM)

Replace the local vLLM model with Claude 3.5 Sonnet (or equivalent) via the Anthropic API for journal generation. Claude models have strong instruction-following and lower hallucination rates on grounded tasks.

**Pros:**
- Best-in-class instruction adherence and grounding capability.
- No local GPU consumption — offloads entirely to API.
- Prompt engineering is simpler; the model is more likely to respect "do not fabricate" instructions without elaborate anchor architecture.
- Could potentially skip the verification stage, reducing complexity.

**Cons:**
- **Cost.** At ~$3-15 per million input tokens (depending on model tier), processing large session transcripts daily accumulates meaningful spend. A 5 MB transcript at $3/M input tokens costs ~$0.015 per session; at 5-10 sessions per day, this is $2-5/month — tolerable, but the vision includes scaling to multiple users and automated pipelines where costs compound.
- **Latency and availability.** API calls introduce network dependency. Rate limits, outages, or API changes could stall the pipeline.
- **Privacy.** Session transcripts contain file paths, code snippets, internal architecture details, and potentially sensitive information. Sending these to a cloud API may conflict with data handling preferences, especially for sessions involving employer-adjacent work.
- **Architectural dependency.** The platform's value proposition includes local-first, self-hosted AI. Depending on a cloud API for a core pipeline undermines this principle.

**Verdict:** The pragmatic choice if local models prove fundamentally inadequate. Worth keeping as a fallback, but the E2A architecture with a well-chosen local model should be attempted first. The extraction stage (Stage 1) is valuable regardless — even a Claude API call benefits from a structured anchor rather than a raw 16 MB transcript.

### Alternative C: Fine-Tune DeepSeek R1 8B on Grounded Summarisation

Create a training dataset of (transcript, journal entry) pairs, manually write high-quality journal entries for 50-100 sessions, and fine-tune DeepSeek R1 8B (or a LoRA adapter) specifically for this task.

**Pros:**
- Addresses the model mismatch directly by adapting the model to the task.
- Keeps the existing single-model deployment; no model swap needed.
- Could produce a highly specialised journal writer with consistent voice and format.

**Cons:**
- **Training data cost.** Manually writing 50-100 gold-standard journal entries is weeks of effort. The quality of the fine-tune is bounded by the quality of this dataset.
- **Overfitting risk.** At 50-100 examples, the model may memorise patterns rather than learn grounding. Session diversity (short debugging sessions vs long architecture discussions) compounds this.
- **Maintenance burden.** As session formats evolve, the fine-tune degrades. Periodic retraining is needed.
- **Does not fix input poverty.** A fine-tuned model receiving sparse OTEL events still lacks the raw material to produce grounded output. Fine-tuning might teach it to say "I don't know" more often, but it cannot teach it facts that are not in the input.

**Verdict:** Premature optimisation. The E2A architecture with a general-purpose instruct model should be validated first. If the architecture works but output voice/style needs refinement, fine-tuning the generation stage (Stage 3 only, with anchor-grounded inputs) becomes a targeted improvement rather than a hail-mary fix.
