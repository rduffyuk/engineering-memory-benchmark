# ADR-043: Memory System Derived-Data GDPR Position

## Status

Proposed

## Context

Rootweaver's memory system ingests Claude Code session transcripts (JSONL files containing verbatim user and assistant messages) and produces a layered graph of derived data across six storage backends:

1. **Postgres `memory.sessions`** -- session metadata: timestamps, branch, Jira refs, commits, files touched, summary embedding (pgvector 1024d), tombstone timestamp.
2. **Postgres `memory.rounds`** -- per-turn records: user text, assistant text, redacted assistant thinking, tool calls, raw message JSON, per-round embedding.
3. **Postgres `memory.facts`** -- LLM-extracted structured claims (type: decision, file_change, problem, learning, tool_use, collaboration, goal) with confidence scores, temporal validity windows (`t_valid`/`t_invalid`), and per-fact embeddings.
4. **Postgres `memory.artifacts`** -- links between rounds/facts and external objects (git commits, diff hunks, vault docs, Confluence pages, Jira issues, file operations, K8s applies).
5. **Postgres `memory.fact_corrections`** and **`memory.file_snapshots`** -- correction chains and captured file states at specific rounds.
6. **FalkorDB `code_knowledge` graph** -- nodes and edges derived from session data via `falkordb_fanout.py`, representing entities, decisions, and relationships extracted during memory processing.

The source transcripts contain personal data under UK GDPR Article 4(1): the user's natural-language instructions reveal working patterns, technical decisions, career context, employer details, and occasionally PII that the `redactor.py` module attempts to strip pre-insertion. The derived layers (facts, embeddings, graph nodes) are produced by LLM inference over that personal data and inherit its character.

The project is planned for public release. This forces a decision on three questions regulators and security auditors will ask:

**Q1: Is LLM-extracted derived data (facts, embeddings, graph nodes) personal data?**

The Hamburg DPA's January 2024 enforcement action against an AI chatbot provider held that "memories" derived from user conversations constitute personal data under GDPR even when the original conversation is deleted, because the derived data can re-identify the individual or reveal information about them. The EDPB Opinion 28/2024 on AI models and GDPR (adopted December 2024) does not fully resolve the question but establishes that where model outputs or stored inferences relate to an identifiable person, they fall within the material scope of the GDPR. For Rootweaver's case -- a single-user system where every fact, embedding, and graph node traces back to one identifiable individual via `session_id` foreign keys -- the answer is unambiguous: all derived layers are personal data.

**Q2: What does the right to erasure (Article 17) require for derived data?**

Article 17(1) mandates erasure of personal data when the data subject withdraws consent or the data is no longer necessary. The question is whether deleting the source transcript (JSONL) satisfies the obligation, or whether derived layers must also be purged. The Hamburg DPA position is that derived data must be independently deletable; retention of "memories" after conversation deletion violates Article 17. OWASP's AI Security Verification Standard (AISVS, v1.0 published 2025) control C08 requires that AI systems implement hard-delete capabilities for all personal data including derived inferences, and that deletion propagate to all downstream stores within a defined time window.

**Q3: How long may data be retained before deletion?**

The EDPB Opinion 28/2024 reaffirms the storage limitation principle (Article 5(1)(e)): personal data must be kept only as long as necessary for its purpose. For an engineering memory system whose purpose is decision recall across sessions, there is no fixed regulatory maximum, but the system must define and enforce a retention period, and deletion must be available on demand.

### Prior Art in the Codebase

The vault search surfaced the following relevant implementations:

- **VW-220** (Phase 2, 2026-04-21): Kafka-driven incremental indexer validated round-trip `create -> index -> delete -> CASCADE` for the vault-postgres schema. The CASCADE pattern is proven infrastructure.
- **VW-422** (Backlog): Provenance backfill ticket plans to mark existing 148K+ chunks with `provenance.trust_level=unknown-legacy`, establishing data lineage tracking.
- **VW-391** (Design, 2026-05-17): Long-term structured memory store design, the parent initiative that created the `memory.*` schema. The spec at `docs/superpowers/specs/2026-05-17-Long-Term-Memory-Postgres-Design.md` defined the six-table schema.
- **`memory_forget.py`** (operator-tools): Already implements tombstone-then-hard-delete with CASCADE through all Postgres child tables. The tool's docstring explicitly references this ADR.
- **`tables.py`** (rootweaver-memory): Every child table (`rounds`, `facts`, `artifacts`, `fact_corrections`, `file_snapshots`) declares `ForeignKey(..., ondelete="CASCADE")` back to `sessions.id`.
- **`test_operator_tools_memory_forget_cascade.py`**: Integration test using testcontainers + pgvector/pg16 that verifies both tombstone and immediate-delete paths cascade correctly.

The Postgres layer is therefore already compliant. The gap is FalkorDB: `falkordb_fanout.py` writes graph nodes but `memory_forget.py` does not yet delete them. The `reconcile_memory_graph.py` operator tool exists but is a consistency reconciler, not a deletion tool.

## Decision

We adopt the following position for Rootweaver's memory system:

### 1. All derived data is personal data

Every layer derived from session transcripts -- extracted facts, pgvector embeddings, FalkorDB graph nodes, assistant thinking signatures, fact corrections -- is classified as personal data under UK GDPR. This applies regardless of whether the derived data contains literal PII, because:

- It is produced by processing personal data (Article 4(2) definition of "processing").
- It relates to an identifiable individual (single-user system with `session_id` foreign keys).
- It can reveal information about the individual's working patterns, decisions, and technical choices.

### 2. Cascade-delete is mandatory for Article 17 compliance

When a session is deleted (via `memory-forget --session-id <uuid>`), all derived data across all storage backends must be purged:

| Layer | Delete mechanism | Timing |
|-------|-----------------|--------|
| `memory.sessions` | `DELETE` (Postgres) | Immediate or after tombstone grace period |
| `memory.rounds` | `ON DELETE CASCADE` from sessions | Automatic with session delete |
| `memory.facts` | `ON DELETE CASCADE` from sessions | Automatic with session delete |
| `memory.artifacts` | `ON DELETE CASCADE` from sessions/rounds/facts | Automatic with session delete |
| `memory.fact_corrections` | `ON DELETE CASCADE` from facts | Automatic with session delete |
| `memory.file_snapshots` | `ON DELETE CASCADE` from sessions | Automatic with session delete |
| pgvector embeddings | Embedded in rows above; deleted with row | Automatic with session delete |
| FalkorDB graph nodes | Explicit `MATCH (n {session_id: $sid}) DELETE n` | **Must be added to `memory_forget.py`** |

### 3. Tombstone-then-reap is the default path

The two-phase deletion model already implemented in `memory_forget.py` is adopted as the standard:

- **Phase 1 (tombstone)**: `UPDATE memory.sessions SET tombstoned_at = now()` marks the session for deletion. Tombstoned sessions are excluded from all query paths (MCP tools, `recall_*` functions, benchmark harness). This provides a grace period for accidental deletion recovery.
- **Phase 2 (hard delete)**: A nightly CronJob (`memory-tombstone-reaper`) executes `DELETE FROM memory.sessions WHERE tombstoned_at < now() - interval '7 days'`. The CASCADE propagates to all child tables. FalkorDB cleanup runs in the same job.
- **Immediate delete**: `memory-forget --immediate` bypasses the grace period for regulatory urgency (e.g., data subject access request with tight deadline).

### 4. Retention period: 24 months default, configurable

Session data is retained for 24 months from `sessions.ended_at` unless the user explicitly deletes earlier. After 24 months, the tombstone-reaper automatically marks sessions for deletion. Rationale: engineering decisions lose recall value beyond two years (codebase has typically evolved past relevance), while remaining long enough to support retrospective analysis and blog content generation.

The retention period is configurable via `MEMORY_RETENTION_MONTHS` environment variable to support different deployment contexts.

### 5. Redaction at ingestion, not just at deletion

The existing `redactor.py` module strips PII patterns (emails, API keys, file paths containing usernames, IP addresses) before insertion into Postgres. This defence-in-depth approach means that even if a deletion cascade has a gap, the stored data has reduced PII exposure. Redaction is not a substitute for deletion -- it is a complementary control.

### 6. Public release requires documentation

Before open-sourcing the memory system, the following must be present:

- A `PRIVACY.md` file documenting what data is stored, for how long, and how to delete it.
- The `memory-forget` CLI tool must be documented as the primary deletion interface.
- The default configuration must have retention enabled (not infinite by default).
- FalkorDB cascade-delete must be implemented and tested (see Consequences).

## Consequences

### Positive

- **Regulatory clarity**: The project has a documented, defensible position on derived-data classification and deletion. This removes ambiguity for contributors and users.
- **Existing infrastructure validates the approach**: Postgres CASCADE foreign keys are already in place and tested. The tombstone/reap pattern is implemented. The gap is small (FalkorDB only).
- **OWASP AISVS C08 compliance**: Hard-delete with cascade through all stores meets the AISVS control requirement for AI systems handling personal data.
- **Public release readiness**: Having a clear GDPR position documented before release is materially better than retrofitting post-release.

### Negative

- **FalkorDB deletion adds complexity**: Graph databases do not have foreign-key cascades. Deletion requires explicit Cypher queries scoped by `session_id`, and the reconciliation must handle partial failures (Postgres deletes but FalkorDB call fails). A compensating transaction or retry queue is needed.
- **24-month retention requires a reaper CronJob**: An additional K8s CronJob must be maintained. If it fails silently, data accumulates beyond the retention period. Monitoring (Prometheus metric for last-successful-reap timestamp) is required.
- **Tombstone exclusion must be enforced at every query path**: Every MCP tool, benchmark harness, and operator script that reads from `memory.*` must filter `WHERE tombstoned_at IS NULL`. A single missed query path leaks tombstoned data. A Postgres Row-Level Security policy could enforce this at the database layer as defence-in-depth.
- **Embedding deletion is irrecoverable**: Once a session's embeddings are deleted, similarity search against that session's content is permanently lost. The 7-day tombstone grace period mitigates accidental deletion, but the user must understand this is a destructive operation.

## Alternatives Considered

### Alternative A: Source-only deletion (delete JSONL, retain derived data)

Under this model, deleting the source transcript would satisfy Article 17 because the derived data (facts, embeddings) is "anonymised" by the extraction process -- the LLM summarises and restructures, removing literal personal data.

**Rejected because**:

- The Hamburg DPA explicitly rejected this argument in the January 2024 enforcement action. Derived "memories" were held to be personal data even after source conversation deletion.
- In Rootweaver's single-user architecture, every fact row has a `session_id` foreign key that links back to the individual. Even if the fact text is abstract ("decided to use Postgres over SQLite for memory store"), the link to a specific user session makes it personal data.
- OWASP AISVS C08 requires deletion of derived inferences, not just source data.
- The risk/reward is poor: retaining derived data after source deletion saves negligible storage while creating regulatory exposure for a project intended for public release.

### Alternative B: Anonymisation instead of deletion

Transform derived data on deletion rather than deleting it: strip `session_id` foreign keys, replace user-attributable content with generic placeholders, retain embeddings for aggregate analysis.

**Rejected because**:

- GDPR Recital 26 sets a high bar for anonymisation: data must be irreversibly de-identified such that the data subject is "no longer identifiable." In a single-user system with only one person's engineering decisions, even de-identified facts like "decided to use FalkorDB for graph storage" are re-identifiable from context.
- The engineering cost of building a robust anonymisation pipeline (that satisfies the "irreversibility" test) exceeds the cost of cascade-delete, which is already 90% implemented.
- Partial anonymisation (stripping session_id but retaining embeddings) fails the "no longer identifiable" test because embedding vectors can be matched against known content via cosine similarity.

### Alternative C: Shorter retention with no tombstone (immediate hard-delete only)

Delete data immediately on user request with no grace period. Reduce default retention to 6 months.

**Considered but not adopted because**:

- Immediate hard-delete with no recovery window is operationally risky. A single mistyped session ID permanently destroys data. The 7-day tombstone grace period is a standard pattern (cf. Google Workspace 25-day retention, AWS S3 Object Lock governance mode).
- 6-month retention is too short for retrospective blog content (Building-in-Public episodes reference decisions made 3-6 months prior) and for the engineering memory benchmark (ADR-045), which requires historical ground truth.
- However, the configurable `MEMORY_RETENTION_MONTHS` accommodates deployments that prefer shorter retention.

## Implementation Requirements

The following work items are needed to fully implement this ADR:

1. **FalkorDB cascade-delete in `memory_forget.py`**: Add Cypher deletion scoped by `session_id` to both tombstone-reap and immediate-delete paths. Add retry logic for FalkorDB connection failures.
2. **Retention reaper CronJob**: K8s CronJob that runs `DELETE FROM memory.sessions WHERE ended_at < now() - interval '$RETENTION months' AND tombstoned_at IS NULL`, then the tombstone-reap for anything past the 7-day grace. Deploy via rootweaver-gitops.
3. **Row-Level Security policy**: `CREATE POLICY hide_tombstoned ON memory.sessions FOR SELECT USING (tombstoned_at IS NULL)` as defence-in-depth. Apply to `memory_writer` role.
4. **Prometheus metric**: `memory_reaper_last_success_timestamp` gauge, scraped by existing monitoring stack.
5. **Integration test**: Extend `test_operator_tools_memory_forget_cascade.py` to verify FalkorDB node deletion (requires FalkorDB testcontainer or mock).
6. **`PRIVACY.md`**: Document data lifecycle for public release.
