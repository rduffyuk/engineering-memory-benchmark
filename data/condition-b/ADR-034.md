# ADR-034: Data Source Connector Architecture

## Status

Proposed

## Context

Rootweaver's RAG platform currently has a single data ingestion path: local Obsidian vault files are watched via inotify, published to Kafka, and indexed into Qdrant (148K+ chunks across ~4K documents, 1024-dimensional Qwen3-Embedding-0.6B vectors). Jira and Confluence data are accessible only through live MCP tool calls at query time --- they are never pre-indexed into the vector store. This means cross-source queries ("What Jira issues relate to this vault ADR?") cannot be answered by a single retrieval pass; the system either returns vault-only results or requires the caller to issue separate MCP calls and mentally fuse the answers.

Market comparators set a higher bar. Glean provides 100+ pre-built connectors as a managed SaaS offering. Onyx (formerly Danswer), the closest open-source analogue, ships 40+ connectors covering SaaS tools (Jira, Confluence, Slack, Google Drive, Notion, GitHub/GitLab, email) and uses a polling-based architecture with a normalised `Document` model, `BaseConnector.load_from_state()` / `poll_source()` interface, and Postgres-backed watermark checkpointing. For a personal-scale RAG platform aspiring to unified knowledge search, the absence of multi-source indexing is a functional gap, not a nice-to-have.

The immediate trigger for this ADR is VW-184 ("ADR: Data Source Connector Architecture for Multi-Source RAG Ingestion"), which scoped the problem and evaluated alternatives. Several follow-on issues already exist: VW-186 (source filter silently dropped in Qdrant --- since fixed in gateway path, commit `c392e56`), VW-188 (activate Jira connector --- producer side validated, 231 issues synced to Kafka with correct Postgres watermark), VW-232 (connector-indexer consumer has no `__main__` block --- KEDA ScaledJob spawns empty pods), and VW-185 (HashiCorp Vault for secrets, deferred until connector count exceeds 5--6 sources).

### Constraints

1. **Single-node K3s cluster** with 62GB RAM and one RTX 4080 GPU. Connectors must be lightweight (no persistent pods for idle sources).
2. **Existing Kafka infrastructure** (Strimzi, topic `vault.connector-docs`, KEDA ScaledJobs for scale-to-zero consumers). The vault file-watcher already proves this pattern.
3. **Qdrant collection `rootweaver_vault`** stores all content. Adding new sources must not require separate collections; source discrimination uses payload filtering on `source` and `source_id` fields.
4. **Credential management** via SealedSecrets today. OAuth token refresh and rotation are manual. HashiCorp Vault (port 30820 allocated) is a future dependency.
5. **Onyx's connector library** is Apache-2.0 licensed and provides tested implementations for 40+ sources. Writing connectors from scratch would duplicate validated API integration logic.

## Decision

We adopt a **polling-based connector framework that wraps Onyx's open-source connector implementations** behind an adapter layer, producing normalised documents into the existing Kafka pipeline for indexing into Qdrant.

### Architecture

```
                    ┌───────────────────────────────────────────┐
                    │         Connector Scheduler (CronJob)      │
                    │  packages/operator-tools/connector_sync.py │
                    └───────────┬───────────────────────────────┘
                                │ for each configured source
                                ▼
                    ┌───────────────────────────────────────────┐
                    │         ConnectorFactory.create(source)    │
                    │  rootweaver_connectors/factory.py          │
                    │  Registry: jira, confluence, gitlab,       │
                    │            slack, discord, gmail, web, ... │
                    └───────────┬───────────────────────────────┘
                                │ instantiates + loads credentials
                                ▼
                    ┌───────────────────────────────────────────┐
                    │         OnyxBridge.checkpoint_sync()       │
                    │  rootweaver_connectors/bridge.py           │
                    │  Loops connector.load_from_checkpoint()    │
                    │  Converts Onyx Document → ConnectorDoc     │
                    │  Produces to vault.connector-docs topic    │
                    └───────────┬───────────────────────────────┘
                                │ Kafka messages
                                ▼
                    ┌───────────────────────────────────────────┐
                    │    ConnectorIndexer (KEDA ScaledJob)       │
                    │  rootweaver_connectors/indexer_runner.py   │
                    │  Consumes from Kafka, chunks, embeds,     │
                    │  upserts into Qdrant rootweaver_vault     │
                    └───────────┬───────────────────────────────┘
                                │
                                ▼
                    ┌───────────────────────────────────────────┐
                    │         Qdrant (rootweaver_vault)          │
                    │  Payload fields: source, source_id, url,  │
                    │  author, date_created, date_updated        │
                    │  Indexed for faceted filtering             │
                    └───────────────────────────────────────────┘
```

### Key Components

**1. Onyx Shim (`rootweaver_connectors/onyx_shim/`)**
A vendored subset of Onyx's connector models and interfaces, providing `BaseConnector`, `Document`, `Section`, `ConnectorFailure`, and `HierarchyNode` types without pulling the entire Onyx application. This shim is injected onto `sys.path` at factory import time, allowing Onyx connector modules (under `onyx_shim/onyx/connectors/`) to import their own models as if running inside the full Onyx application.

**2. ConnectorDoc (`rootweaver_connectors/models.py`)**
A normalised dataclass representing a document from any external source:

```python
@dataclass
class ConnectorDoc:
    source: str          # e.g. "jira", "confluence", "gitlab"
    source_id: str       # unique within source
    title: str
    content: str
    url: Optional[str]
    author: Optional[str]
    created_at: Optional[datetime]
    updated_at: Optional[datetime]
    metadata: dict[str, Any]
```

This is serialised to JSON for Kafka transport and deserialised by the indexer consumer. The `source` and `source_id` fields are stored as Qdrant payload fields and registered in `FACET_INDEX_FIELDS` for efficient filtering.

**3. OnyxBridge (`rootweaver_connectors/bridge.py`)**
Drives any Onyx `CheckpointedConnector` to completion via `load_from_checkpoint()`, converting each yielded `Document` into a `ConnectorDoc` and producing it to Kafka. Handles `ConnectorFailure` items (logged, counted, document skipped) and `HierarchyNode` items (dropped --- no consumer yet). Flushes the Kafka producer in a `finally` block to prevent message loss on exception.

**4. ConnectorFactory (`rootweaver_connectors/factory.py`)**
Registry mapping source names to Onyx connector classes, configuration dicts, and credential environment variable names. Currently registers 9 sources: jira, confluence, gitlab, discord, slack, gmail, imap, web, file. `create_connector(source)` instantiates the class, loads JSON credentials from the environment, and returns a configured connector (or `None` if credentials are absent).

**5. Watermark Checkpointing (Postgres)**
The scheduler stores per-source `(last_run_at, last_run_status, docs_produced, watermark_ts)` in a `connector_sync_state` table. On each cycle, `start_ts` is set to the previous watermark; `end_ts` is `now()`. After a successful sync, the watermark advances to `max(doc.updated_at)` from the batch. This is a Postgres-backed variant of Onyx's `ConnectorCredentialPair` checkpoint, simplified for single-tenant operation.

**6. KEDA-Scaled Consumer**
The `ConnectorIndexer` consumes from `vault.connector-docs`, chunks documents, generates embeddings (same Qwen3 model as vault indexer), and upserts into Qdrant `rootweaver_vault` with `source`, `source_id`, and other metadata as payload fields. KEDA scales the consumer to zero when the topic has no lag, keeping the cluster idle-cost at zero between sync cycles.

### Connector Lifecycle

1. **CronJob fires** (e.g. every 6 hours for Jira, daily for Confluence).
2. **Factory creates connector**, loads credentials from SealedSecret-injected env vars.
3. **Bridge runs `checkpoint_sync()`**: polls from previous watermark, yields `ConnectorDoc` messages to Kafka.
4. **Postgres watermark advances** to max `updated_at` from the batch.
5. **KEDA detects lag** on `vault.connector-docs`, scales up indexer pod.
6. **Indexer consumes, embeds, upserts** into Qdrant with `source` payload metadata.
7. **KEDA scales indexer to zero** when lag clears.
8. **Search queries** can now filter by `source="jira"` or search across all sources.

### Source Filter Plumbing

The gateway's `SearchRequest` accepts `source` and `sources` filter parameters. The direct retriever path (HYBRID/TEMPORAL modes) forwards these to Qdrant via `build_filter()` (fixed in VW-186, commit `c392e56`). The agent path (AUTO/FAST/DEEP) does not yet forward source filters --- the base agent's `_execute_vector_search_async` supports a `filters` parameter, but the agent `search_async()` call sites do not pass it through. This is tracked as a follow-up (VW-186 part 2) and is not blocking: cross-source queries work via the hybrid/temporal modes today.

## Consequences

### Positive

- **Unified cross-source search**: A single Qdrant query retrieves relevant documents from vault, Jira, Confluence, GitLab, and any future source. No manual MCP call fusion required.
- **Incremental adoption**: Each source is independently configured via a SealedSecret and a CronJob schedule. Sources can be lit up one at a time with zero impact on others.
- **Zero idle cost**: KEDA scales the indexer to zero between sync cycles. CronJobs run only on schedule. No persistent connector pods.
- **Leverages tested implementations**: Onyx's connector library handles API pagination, rate limiting, OAuth token refresh, and error recovery for 40+ sources. We avoid reimplementing this logic.
- **Normalised metadata**: All sources share the same `ConnectorDoc` schema and Qdrant payload structure, enabling consistent filtering, freshness scoring, and attribution across sources.
- **Existing infrastructure reuse**: Kafka (Strimzi), KEDA, Qdrant, and Postgres are already deployed. No new infrastructure components required.

### Negative

- **Onyx shim maintenance**: The vendored shim must track upstream Onyx changes when connector APIs evolve. Breaking changes in Onyx's `Document` or `BaseConnector` model require shim updates. Mitigation: pin to a known Onyx release tag; update on a quarterly cadence or when adding new connectors.
- **Polling latency**: CronJob-based polling introduces inherent delay (minutes to hours) between a source update and its appearance in search results. Real-time ingestion (webhooks) is not in scope for this ADR. For Rootweaver's personal-scale usage, polling on a 6-hour cadence is acceptable.
- **Source filter gap in agent path**: The AUTO/FAST/DEEP agent `search_async()` methods do not forward source filters to Qdrant. Users must use HYBRID or TEMPORAL mode for source-filtered queries until VW-186 part 2 lands.
- **Credential sprawl**: Each source requires a SealedSecret with API credentials. At 9 registered sources, this is manageable; beyond 15--20, HashiCorp Vault (VW-185) becomes necessary for rotation and audit.
- **Consumer gap (VW-232)**: The `ConnectorIndexer` class exists but had no `__main__` block or Kafka consumer loop at the time of initial deployment. KEDA ScaledJobs spawned empty pods that exited silently. This has been identified and is being addressed; until resolved, produced messages queue in Kafka without being indexed.

### Neutral

- The existing vault file-watcher pipeline (inotify → Kafka → Qdrant) is unchanged. It continues to operate on its own topic (`vault.file-events`) independently of the connector pipeline.
- Qdrant collection schema does not change. New payload fields (`source`, `source_id`) are added to existing documents; old vault documents without these fields are unaffected (filter on `source` simply excludes them, which is correct behaviour).

## Alternatives Considered

### Alternative A: Direct API Calls at Query Time (MCP-Only, No Indexing)

Continue the current pattern: Jira and Confluence are queried live via MCP tools; no pre-indexing.

**Pros**: Zero infrastructure overhead. No sync state. Always returns latest data.
**Cons**: Cross-source queries impossible in a single retrieval pass. Latency spikes when MCP tools call external APIs. No semantic search over Jira/Confluence content --- only keyword matching via API query parameters. Does not scale to additional sources without adding more MCP tools and forcing callers to orchestrate result fusion.

**Rejected because**: The core use case --- "find everything relevant to X across all my knowledge" --- requires pre-indexed vectors from all sources in the same embedding space.

### Alternative B: Fork Onyx (Full Application)

Deploy Onyx as a standalone service alongside Rootweaver, using its built-in connector scheduling, Postgres state, and Vespa/typesense backend.

**Pros**: Get 40+ connectors immediately with battle-tested scheduling, monitoring, and admin UI. Full Onyx feature set including access control, document sets, and re-ranking.
**Cons**: Onyx is a full application (~2GB Docker image, requires its own Postgres, Redis, and either Vespa or typesense). Running it alongside Rootweaver's existing Qdrant + Kafka stack doubles infrastructure. Onyx's search backend would compete with Rootweaver's RAG agents for the same queries, creating a split-brain problem. Integration would require either migrating Rootweaver's search to Onyx's backend or building a federation layer. The operational burden of maintaining two search platforms on a single-node K3s cluster (62GB RAM) is disproportionate.

**Rejected because**: The infrastructure cost and architectural complexity of running a second search platform exceed the value of pre-built connectors, which can be obtained more cheaply via the shim approach.

### Alternative C: Build Connectors from Scratch (No Onyx Dependency)

Implement each connector as a standalone Python module with a Rootweaver-native `BaseConnector` interface, without any Onyx code.

**Pros**: No external dependency. Full control over connector interface. Simpler deployment (no shim, no vendored code).
**Cons**: Each connector requires implementing API pagination, rate limiting, OAuth flows, error handling, and data model mapping from scratch. Jira alone has 6 API endpoints (issues, comments, worklogs, attachments, changelogs, fields). Confluence has 4. GitLab has 8+. The development cost for 9 sources is estimated at 2--4 weeks per connector, totalling 4--9 months of effort. Onyx's implementations have been tested against production APIs by a community of contributors; reimplementing them risks subtle API compatibility bugs.

**Rejected because**: The build cost is prohibitive for a solo operator. Onyx's Apache-2.0 licensed connectors are free to vendor, and the shim approach isolates Rootweaver from Onyx's application-level complexity while reusing the connector logic.
