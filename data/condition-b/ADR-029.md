# ADR-029: KEDA-Based GPU Swap Controller for Embed/Inference Coexistence

**Status**: Accepted
**Date**: 2026-04-29
**Author**: Ryan Duffy + Claude
**Jira**: VW-233
**Supersedes**: Hybrid GPU Embedding Architecture (VW-98, 2026-02-18) — MacBook MPS primary path retired after VW-203 decommission; desktop-only GPU swap replaces hybrid approach.

---

## Context

### The Hardware Constraint

Rootweaver runs on a single-node K3s cluster with one NVIDIA RTX 4080 (16 GB VRAM). Three GPU workloads compete for that memory:

| Workload | Pod | VRAM Required | Duty Cycle |
|----------|-----|---------------|------------|
| LLM inference | `vllm-0` (namespace: `vllm`) | ~15.6 GB at 95% utilisation | Always-on (default tenant) |
| Production embedding | `tei-embedding` (namespace: `rootweaver`) | ~8 GB transient | On-demand (Kafka lag or nightly cron) |
| Code-graph embedding | `vllm-embedding` (namespace: `vllm`) | ~8 GB transient | On-demand (PDG rebuild) |

vLLM serves DeepSeek-R1-0528-Qwen3-8B-AWQ at 114K context with YaRN RoPE scaling and FP8 KV cache. At `--gpu-memory-utilization 0.95` it claims ~15.6 GB, leaving under 1 GB free — far short of the ~8 GB TEI needs for Qwen3-Embedding-4B (2560d, matching the `rootweaver_vault` Qdrant collection). **Both workloads cannot coexist on the GPU simultaneously** (HLD-03 AI Layer §1, verified 2026-05-19).

### Why This Changed

The predecessor ADR (VW-98, 2026-02-18) solved this via a hybrid architecture: MacBook M4 Pro MPS as primary embedder, desktop TEI as nightly fallback during a vLLM sleep window. That architecture worked but was retired when the MacBook embedding agent was decommissioned (VW-203, 2026-04-18) because:

1. **Memory leak on MPS**: The MacBook agent consumed 24 GB unified memory due to stacking PyTorch MPS bugs (VW-107). The fix worked but added operational fragility.
2. **Syncthing dependency**: Embedding required the MacBook to be awake with Syncthing active. Lid-closed mode was unreliable for long vault indexing runs.
3. **Workload consolidation**: Moving all compute to the desktop K3s node simplified operations and eliminated a cross-device failure mode.

With the MacBook path gone, the desktop must handle both inference and embedding on a single GPU. Manual swap (scaling vLLM down, scaling TEI up, reversing after embedding completes) was attempted for the VW-96 Kafka delta-indexing pipeline but proved untenable: the Jira connector (VW-288) makes ingest bursts routine — every Jira sync produces dozens of Kafka events on `vault.file-events`, and each burst would require manual GPU orchestration.

### HAMi Ruled Out

HAMi (Heterogeneous AI computing Management Interface) is installed in the cluster (`hami-system` namespace) and theoretically supports fractional GPU allocation via `nvidia.com/gpumem` resource requests. However, vLLM requires `nvidia.com/gpu: "1"` — a full device claim — because it manages its own KV-cache memory pool and cannot operate within a HAMi memory partition. TEI similarly takes the full device when scheduled. HAMi fractional sharing is incompatible with both workloads (HLD-03 §1, "HAMi — Installed but Bypassed").

---

## Decision

Implement a **KEDA-driven GPU swap controller** using Kubernetes PriorityClass preemption to yield the GPU between inference and embedding workloads on demand, with no manual intervention.

### Mechanism

The core insight is that Kubernetes PriorityClass preemption provides a native, scheduler-level GPU swap without requiring custom controllers or vLLM sleep-mode orchestration:

1. **PriorityClass separation**: TEI embedding runs at `vllm-embedding-priority` (value: 1000, high). vLLM inference runs at `vllm-llm-priority` (value: 100, low). When both request the GPU, the scheduler preempts the lower-priority pod.

2. **KEDA ScaledObject for TEI**: A KEDA ScaledObject watches two triggers:
   - **Kafka trigger**: Consumer lag on `vault.file-events` topic (lagThreshold=1, pollingInterval=30s). Any pending file event — whether from the inotify file-watcher or a Jira connector sync burst — triggers TEI scale-up from 0 to 1.
   - **Cron trigger**: Daily 03:00–05:00 UTC window for the nightly consistency check (hash comparison + full re-embed of drifted documents).

3. **Preemption cascade**: When TEI scales from 0 to 1 and requests the GPU, the Kubernetes scheduler sees that TEI (priority 1000) outranks vllm-0 (priority 100). It preempts vllm-0, which enters a graceful termination → eviction cycle. TEI starts on the freed GPU, processes queued Kafka events, and completes.

4. **Automatic recovery**: When the KEDA trigger clears (Kafka lag returns to 0 and the cron window closes), KEDA scales TEI back to 0 after a 5-minute cooldown (`cooldownPeriod: 300`). The scheduler then reschedules vllm-0, which cold-starts in ~60 seconds (model load from local `/models/` volume, not network pull).

5. **Code-graph embedder**: `vllm-embedding` (jina-v4-code for PDG rebuilds) uses the same `vllm-embedding-priority` PriorityClass, so it follows the identical preemption pattern when triggered.

### Key Configuration

```yaml
# PriorityClasses
apiVersion: scheduling.k8s.io/v1
kind: PriorityClass
metadata:
  name: vllm-llm-priority
value: 100
preemptionPolicy: PreemptLowerPriority
description: "Default GPU tenant — evictable by embedding workloads"
---
apiVersion: scheduling.k8s.io/v1
kind: PriorityClass
metadata:
  name: vllm-embedding-priority
value: 1000
preemptionPolicy: PreemptLowerPriority
description: "On-demand embedding — preempts LLM inference"
```

```yaml
# KEDA ScaledObject for TEI (simplified)
apiVersion: keda.sh/v1alpha1
kind: ScaledObject
metadata:
  name: tei-embedding
  namespace: rootweaver
spec:
  scaleTargetRef:
    name: tei-embedding
  minReplicaCount: 0
  maxReplicaCount: 1
  cooldownPeriod: 300
  triggers:
    - type: kafka
      metadata:
        bootstrapServers: kafka-controller-0.kafka-controller-headless.kafka.svc:9092
        consumerGroup: tei-embedding-consumer
        topic: vault.file-events
        lagThreshold: "1"
    - type: cron
      metadata:
        timezone: UTC
        start: "0 3 * * *"
        end: "0 5 * * *"
        desiredReplicas: "1"
```

### Retry Coupling

The `connector-indexer` (and other Kafka consumers that call TEI for embedding) uses a retry budget of 5 attempts with 2-second exponential backoff base (~30 seconds total). This budget is deliberately sized to absorb TEI's cold-start time. **Tuning either the retry budget or TEI's startup time requires considering the other** — they are coupled parameters.

## Consequences

### Positive

- **Zero manual intervention**: Jira connector syncs, file-watcher events, and nightly consistency checks all trigger GPU swap automatically. The system handles bursty ingest patterns without operator involvement.
- **No custom controller code**: The entire mechanism uses standard Kubernetes primitives (PriorityClass, KEDA ScaledObject). No vLLM sleep-mode orchestration, no custom webhook, no sidecar — reducing the surface area for bugs.
- **Preserves embedding quality**: Qwen3-Embedding-4B at 2560d on GPU remains the production embedder. No model downgrade, no Matryoshka dimension reduction, no collection migration. Qdrant `rootweaver_vault` stays at native resolution.
- **Single-node simplicity**: Eliminates the cross-device dependency on the MacBook (retired VW-203). All GPU compute is local, deterministic, and observable via `nvidia-smi` / DCGM metrics on the desktop.
- **Scale-to-zero cost**: When embedding is idle (the majority of the time), TEI runs at 0 replicas — no GPU memory consumed, no CPU overhead. Only the GPU reservation for vLLM exists during steady state.
- **Composable with future workloads**: The PriorityClass pattern extends naturally to additional GPU consumers (e.g., Scout Fleet dispatching on GPU-idle windows — see VW-TBD Scout Fleet plan, which triggers on DCGM < 10% for 5 minutes).

### Negative

- **~60-second LLM cold start**: Every GPU swap cycle incurs a vLLM restart. Model weights load from the local NVMe volume (not network), but the ~60-second init includes CUDA context setup, KV-cache allocation, and warmup. During this window, LLM inference requests from the MCP bridge / RAG gateway return errors or queue.
- **Inference unavailability during embedding**: Unlike vLLM sleep mode (Level 1 wake in 0.1–6 seconds), preemption-based swap has no "fast resume" path. The LLM is fully terminated while TEI runs. At ~50 vault edits/day consolidated into 1–3 Kafka batches, this means 5–15 minutes of daily LLM downtime.
- **StatefulSet restart ordering**: vLLM runs as a StatefulSet. After preemption, the pod must re-enter the ordinal-0 scheduling path. If the GPU is not freed cleanly (e.g., TEI hangs), vllm-0 stays in Pending state until the GPU is released or TEI is manually deleted.
- **Retry budget coupling**: The connector-indexer's retry budget and TEI's cold-start time are coupled. Changes to either require coordinated tuning — a non-obvious dependency that must be documented and tested together.
- **No graceful request draining**: vLLM pods receive SIGTERM on preemption. In-flight inference requests are dropped (vLLM does not currently support graceful drain on preemption). For the single-user homelab workload this is acceptable; for multi-tenant production it would not be.

## Alternatives Considered

### Alternative 1: vLLM Sleep Mode Orchestration (Rejected)

**Concept**: Use vLLM's native sleep mode (`POST /sleep`, `POST /wake_up`) to offload model weights to CPU RAM, freeing GPU VRAM for TEI, then wake vLLM after embedding completes. This was the original Option A evaluated in the VW-98 ADR.

**Why rejected**:

1. **Requires `VLLM_SERVER_DEV_MODE=1`**: The sleep endpoint is behind a development flag with no stability guarantees. Running a dev-mode flag in a production inference server introduces an unquantified risk surface.
2. **Orchestration complexity**: Requires a custom sidecar or init-container to coordinate the sleep → embed → wake sequence. If the consumer crashes mid-batch, the LLM stays asleep — a stuck state requiring manual recovery. The KEDA + PriorityClass approach has no such stuck state: pod termination is handled by the kubelet, and vLLM simply reschedules when the GPU is free.
3. **CPU RAM pressure**: Level 1 sleep offloads ~15.6 GB of model weights to CPU RAM. The desktop has 62 GB total with ~36–54 GB used depending on BM25 index state. Under heavy load, the offloaded weights compete with BM25, Qdrant mmap, and OS page cache. Level 2 (discard weights) avoids this but requires a full model reload on wake — equivalent to the cold start in the preemption approach, without the scheduling simplicity.
4. **Marginal wake-time advantage eroded**: Level 1 sleep promises 0.1–6 second wake (weights already in CPU RAM). But the orchestration overhead (detect Kafka lag → call sleep API → wait for sleep confirmation → start TEI → embed all events → stop TEI → call wake API → wait for wake confirmation) adds 10–30 seconds of plumbing around the fast wake. The net improvement over a 60-second cold start is not significant enough to justify the custom controller.

**Verdict**: The theoretical advantage (fast wake) does not survive contact with operational reality (dev-mode flag, stuck-state risk, RAM contention, orchestration overhead).

### Alternative 2: Smaller Embedding Model to Enable GPU Coexistence (Rejected)

**Concept**: Downgrade from Qwen3-Embedding-4B (2560d, ~8 GB VRAM) to Qwen3-Embedding-0.6B (1024d, ~1.2 GB VRAM). At 1.2 GB the embedding model fits alongside vLLM's ~15.6 GB within the 16 GB VRAM budget, eliminating the need for GPU swapping entirely.

**Why rejected**:

1. **Quality regression**: The 4B model was a deliberate upgrade (VW-39 evaluation) specifically for technical document retrieval accuracy. The MTEB retrieval gap between 4B and 0.6B is 4.38 points — significant for a RAG system where retrieval precision directly determines answer quality. This was explicitly called out in the VW-98 ADR revision as the reason to keep the 4B model.
2. **Collection migration required**: The Qdrant `rootweaver_vault` collection is configured for 2560d vectors. Switching to 1024d requires re-embedding all 148K+ chunks and rebuilding the collection — a multi-hour operation with data loss risk during the migration window.
3. **Locks in a constraint**: Choosing a smaller model to fit the current GPU creates a ceiling. If vLLM's memory footprint grows (larger LLM, longer context, more concurrent requests), the coexistence margin disappears and the problem resurfaces. The swap approach is invariant to model sizes.

**Verdict**: Optimising for coexistence by degrading embedding quality inverts the value hierarchy. The system exists to serve high-quality search; the GPU swap controller exists to make that possible within hardware constraints.

### Alternative 3: Second GPU (Deferred)

**Concept**: Add a second GPU (e.g., RTX 4060 Ti 16GB or a used RTX 3090 24GB) dedicated to embedding workloads, eliminating contention entirely.

**Why deferred, not rejected**: This is the correct long-term solution. A second GPU with HAMi fractional allocation (or simply dedicated to TEI) eliminates all swap overhead, cold-start latency, and inference unavailability. However:

- **Cost**: £300–600 for a suitable card, plus possible PSU upgrade (RTX 4080 + second GPU may exceed 750W PSU capacity).
- **Physical constraints**: The desktop case and motherboard have a second PCIe x16 slot, but thermal and power delivery need verification.
- **Diminishing returns at current scale**: At ~50 vault edits/day and a single user, the 5–15 minutes of daily LLM downtime from the swap controller is acceptable. The second GPU becomes justified when either the workload grows (multi-user, continuous ingest) or the downtime becomes intolerable (real-time inference SLO).

**Trigger to revisit**: If daily LLM downtime from GPU swaps exceeds 30 minutes, or if a second concurrent GPU workload (e.g., fine-tuning, continuous code-graph embedding) is needed.
