# ADR-012: Journal Pipeline Quality Overhaul — Model Selection & E2A Architecture

## Context

The journal streaming pipeline (VW-6, VW-30) produces automated journal entries from Claude Code sessions via Kafka. The pipeline functions end-to-end — Kafka topics flow correctly, triggers fire, entries get written to vault. However, **content quality is not publication-ready**.

The vision for this pipeline is a fully automated content flywheel: session activity flows through Kafka, gets distilled into daily journals, which feed weekly reviews, blog drafts, and social media posts. This vision is blocked by quality. Entries that reach the vault are fabricated — the voice and structure match Ryan's style, but the factual content is fiction. An automated journal that invents what happened is worse than no journal at all: it poisons the vault's status as a single source of truth, undermines downstream RAG search, and makes the blog pipeline untenable.

### Root Causes

Four independent failures converge to produce unreliable output:

**1. OTEL Event Sparsity (2 of 10 sessions have usable telemetry)**

The synthesizer component relies on OpenTelemetry events emitted by Claude Code sessions to provide grounding context — tool calls, file edits, error traces, MCP invocations. In practice, only ~20% of sessions emit enough OTEL data to ground a journal entry. The remaining 80% force the LLM to generate from session metadata alone (start time, duration, token count), which provides insufficient factual basis. The VW-68 rearchitecture (2026-02-12) lowered `MIN_OTEL_EVENTS` from an unreachable 999 to 50, which was a necessary fix but did not address the fundamental sparsity problem — most sessions still have fewer than 50 events.

**2. Model Hallucination Rate (14.3% measured on DeepSeek R1 8B)**

The pipeline currently uses DeepSeek R1 8B via local vLLM inference. While cost-effective and fast (~4.2 tokens/ms at batch 1 on RTX 4080), this model exhibits a 14.3% hallucination rate on the journal generation task as measured by LLM-as-judge evaluation. This is consistent with broader findings from the platform's hallucination testing (vault reference: `2025-12-29-AI-Memory-System-Enhancements-Complete.md`), which measured a 20% hallucination rate for RAG-grounded generation and 43% without RAG. An 8B parameter model operating on sparse context sits firmly in the high-hallucination regime.

For comparison, the October 2025 model evaluation matrix (vault reference: `01-Guides-Demos-Consolidated-Documentation.md`) established 13B as the sweet spot for the RTX 4080, achieving 80%+ code correctness with acceptable VRAM usage. The 8B model falls below this threshold for tasks requiring factual faithfulness — it can narrate plausibly but cannot reliably distinguish what actually happened from what might have happened.

**3. Single-Shot Generation (no extraction-before-synthesis step)**

The current pipeline generates journal prose in a single LLM call: raw session data goes in, narrative text comes out. This conflates two fundamentally different tasks — structured fact extraction and creative synthesis — into one prompt. The LLM must simultaneously decide what happened, what matters, and how to express it. When the input context is sparse (root cause 1), the model fills gaps with plausible fabrications rather than admitting uncertainty. The VW-68 rearchitecture introduced a two-pass consolidation strategy for the daily consolidator (pass 1: structured technical log, pass 2: narrative grounded in the log), but this only operates at the daily aggregation layer, not at the per-session streaming layer where the initial quality problem originates.

**4. Temperature Configuration (0.7 — too high for factual extraction)**

The generation temperature is set to 0.7, which is appropriate for creative writing but actively harmful for factual extraction. At this temperature, the model's probability distribution is flattened enough that low-probability continuations (i.e., fabricated facts) are sampled with non-trivial frequency. For a task where faithfulness to source material is the primary requirement, this setting amplifies every other failure mode.

### Prior Decisions and Incidents

- **ADR-010** (Streaming Journal System architecture): Established the Kafka-based pipeline topology with session parser, chunk router, vLLM workers, and daily aggregator.
- **ADR-011** (Journal LLM Context Handling for Large Sessions): Addressed context window management for sessions exceeding model limits, referencing DeepSeek and GPU allocation. Related to VW-7 (tool context extraction from OTEL).
- **VW-62** (2026-02-11): Kafka consumer crash-loop fix — the journal-session-parser accumulated 837 restarts from infinite reprocessing due to TTL-based dedup. Fixed with persistent file-backed dedup.
- **VW-68** (2026-02-12): Journal system rearchitecture — fixed five independent silent failures, introduced the two-pass daily consolidator using Claude Haiku 4.5 with fallback chain (Haiku → Groq Llama 3.3 70B → local vLLM). Established that two-pass generation prevents hallucination at the consolidation layer.
- **Platform hallucination baseline**: RAG-grounded generation halves hallucination rate from 43% to 20% (vault reference: `AI-Memory-System-Enhancements-Complete.md`). This establishes that grounding is necessary but insufficient alone — model capability matters.
- **GPU allocation constraint**: vLLM holds exclusive GPU access (~15GB VRAM on RTX 4080). Ollama runs CPU-only. Any model selection must fit within this envelope or use external API fallback.

## Decision

We adopt a two-part architectural change: (1) replace the streaming-layer model and (2) introduce an Extract-Aggregate-Articulate (E2A) pipeline that separates fact extraction from prose generation.

### Part 1: Model Selection — Tiered Inference with Qwen3-14B Primary

**Replace DeepSeek R1 8B with Qwen3-14B-AWQ as the primary local model for journal generation.**

Rationale:
- The October 2025 model evaluation established 13B as the RTX 4080 sweet spot: 50-56 tok/s, ~8GB VRAM, 80%+ correctness. Qwen3-14B-AWQ (4-bit quantized) fits comfortably within the 15GB VRAM budget alongside vLLM's overhead.
- AWQ quantization preserves faithfulness better than GPTQ for instruction-following tasks, which is the critical dimension for journal extraction.
- Qwen3 models demonstrate strong structured-output compliance (JSON mode), which the E2A pipeline requires for the extraction phase.
- The HLD-Aware Architecture Review Pipeline design (vault reference: `2026-03-09-HLD-Aware-Architecture-Review-Pipeline-Design.md`) already validated Qwen3-8B-AWQ at 57K context with YaRN RoPE scaling on the same vLLM instance, establishing operational precedent for Qwen3 on this infrastructure.

**Retain the VW-68 fallback chain for the daily consolidator:**
1. Claude Haiku 4.5 (external API — highest quality, cost-controlled)
2. Groq Llama 3.3 70B (external API — fast, free tier)
3. Local vLLM Qwen3-14B (on-premise — always available)

The streaming layer uses local vLLM exclusively (no external API calls per-session — cost would be prohibitive at session volume). The daily consolidator, running once at 23:55, can afford external API quality for the final synthesis pass.

### Part 2: Extract-Aggregate-Articulate (E2A) Architecture

Replace single-shot generation with a three-phase pipeline at the streaming layer:

**Phase 1 — Extract (temperature 0.1, structured JSON output)**

For each session, the LLM extracts structured facts from available evidence (OTEL events, git diffs, file paths, tool calls, session metadata). Output is a JSON schema with mandatory fields:

```json
{
  "session_id": "uuid",
  "timestamp": "ISO-8601",
  "facts_extracted": [
    {
      "claim": "Modified rag_gateway/search.py to add BM25 fallback",
      "evidence_type": "git_diff | otel_event | file_path | session_metadata",
      "evidence_ref": "commit abc1234 | span xyz",
      "confidence": 0.95
    }
  ],
  "decisions_made": [],
  "problems_encountered": [],
  "tools_used": [],
  "files_modified": [],
  "metrics": {},
  "gaps": ["No OTEL events for MCP tool calls — inferred from file paths only"]
}
```

Critical design properties:
- **Temperature 0.1**: Minimises sampling variance. For extraction, we want the modal output, not creative alternatives.
- **Structured JSON output**: Forces the model to decompose its output into discrete, attributable claims rather than flowing prose where fabrications hide.
- **Confidence scores**: Each extracted fact carries a self-assessed confidence. Facts below 0.7 are tagged as inferred rather than observed.
- **Explicit gaps field**: The model must declare what it does not know. This is the anti-hallucination mechanism — gaps in the extraction phase become "not mentioned" in the synthesis phase rather than "plausibly fabricated."

**Phase 2 — Aggregate (deterministic, no LLM)**

A pure Python aggregation step merges extractions from multiple sessions within the same day. This is code, not inference:
- Deduplicate facts by `evidence_ref`
- Merge `files_modified` lists
- Resolve conflicting claims by highest confidence
- Compute daily summary metrics (total commits, unique files, tool call counts)
- Build a structured daily context document

This phase introduces no hallucination risk because it involves no generation.

**Phase 3 — Articulate (temperature 0.4, prose from structured input)**

The daily consolidator (already running at 23:55 via CronJob) receives the aggregated structured context and generates the two vault files:
- `YYYY-MM-DD-technical-log.md`: Structured, factual, enumerated
- `YYYY-MM-DD-reflection-journal.md`: Narrative, first-person, grounded

Temperature 0.4 is a deliberate compromise: low enough to suppress fabrication, high enough to produce natural prose rather than mechanical enumeration. The VW-68 implementation proved that generating narrative from a structured log (not raw fragments) prevents hallucination — E2A extends this principle to the per-session layer.

**The articulation prompt includes a hard constraint**: "You may only reference facts present in the structured context. If the context has gaps, acknowledge them. Do not infer, speculate, or fill in missing information."

### Configuration Changes

| Parameter | Current | New | Rationale |
|-----------|---------|-----|-----------|
| Model (streaming) | DeepSeek R1 8B | Qwen3-14B-AWQ | 13B+ required for faithful extraction |
| Model (consolidator) | Claude Haiku 4.5 | Claude Haiku 4.5 (unchanged) | Already validated in VW-68 |
| Temperature (extraction) | 0.7 | 0.1 | Factual extraction requires low variance |
| Temperature (articulation) | 0.7 | 0.4 | Prose needs some variance, not fabrication-level |
| Pipeline pattern | Single-shot | E2A (3-phase) | Separates extraction from synthesis |
| Output format (extraction) | Free-text markdown | Structured JSON | Forces attributable claims |
| MIN_OTEL_EVENTS | 50 | 10 (with confidence tagging) | Allow sparse sessions but flag low-evidence entries |

## Consequences

### Positive

- **Hallucination reduction**: The combination of a more capable model, lower extraction temperature, structured output, and explicit gap declaration should reduce the hallucination rate from 14.3% to below 5%. The two-pass pattern at the consolidation layer (VW-68) already demonstrated this principle; E2A extends it to the session layer.
- **Auditable pipeline**: Every claim in the final journal traces back to an `evidence_ref` in the extraction JSON. This makes quality failures debuggable — was the extraction wrong, or was the articulation unfaithful to the extraction?
- **Vault integrity**: Journal entries grounded in structured facts protect the vault's status as a reliable source for RAG search. Fabricated journal entries would contaminate Qdrant with false information, degrading search quality across the entire platform.
- **Blog pipeline unblocked**: Publication-ready journal entries enable the downstream automation: journal to weekly review to blog draft to social post. Each downstream step inherits the quality of its input.
- **Graceful degradation**: Sessions with sparse OTEL data produce short, honest entries ("3 files modified, no tool call telemetry available") rather than plausible fiction. This is a feature, not a limitation.

### Negative

- **Increased latency**: Three LLM calls per session (extract + articulate at streaming layer, plus daily consolidation) versus one. Expected increase: ~2-3x per session at the streaming layer. Acceptable for an asynchronous pipeline with no real-time SLA.
- **Increased VRAM usage**: Qwen3-14B-AWQ requires ~8GB VRAM versus ~5GB for DeepSeek R1 8B. This remains within the 15GB budget but reduces headroom. If vLLM needs to serve concurrent requests (e.g., architecture review pipeline), contention may occur. Mitigation: the streaming journal pipeline is low-throughput (single-digit sessions per hour) and can queue behind higher-priority inference.
- **Increased token consumption**: Structured JSON extraction output is more verbose than free-text. Estimated 40% increase in per-session token generation. Cost impact is zero for local vLLM (no per-token charge) but increases Kafka message sizes.
- **Migration complexity**: Requires updating the synthesizer container image (new model, new prompt templates, new output parsing), the aggregator logic, and the daily consolidator input format. The Kafka topic schema (`journal.sessions`) may need a version bump if the message format changes.

### Neutral

- **No change to Kafka topology**: The existing topics (`journal.triggers`, `journal.sessions`, `journal.chunks`, `journal.summaries`, `journal.output`) remain. E2A adds processing logic within existing consumers, not new topics.
- **No change to daily consolidator schedule**: 23:55 CronJob remains. It receives richer input (structured aggregated context vs. raw fragments) but the scheduling and fallback chain are unchanged.

## Alternatives Considered

### Alternative 1: External API for All Generation (Claude Haiku per-session)

Use Claude Haiku 4.5 (or Sonnet) for every session extraction, not just daily consolidation.

**Pros**:
- Highest quality generation available
- No local VRAM allocation required
- Proven faithful in VW-68 daily consolidator

**Rejected because**:
- **Cost**: At 10-20 sessions/day, ~4K tokens/session for extraction + ~2K for synthesis, this would consume approximately 60-120K tokens/day. At Haiku pricing ($0.25/MTok input, $1.25/MTok output), annual cost would be $15-30. Affordable, but introduces an external dependency for a pipeline that should run autonomously on-premise. The philosophy of this platform is self-hosted-first with external API as fallback (established in VW-68 fallback chain design).
- **Latency and availability**: External API calls introduce network dependency. If Anthropic has an outage, the entire journal pipeline stalls. The VW-68 rearchitecture specifically learned that silent failures compound — adding another failure mode to an already fragile pipeline is unwise.
- **Data sovereignty**: Session data contains file paths, code snippets, and tool call details. Sending this to an external API, even one with favourable data handling policies, is a decision that should be explicit and opt-in, not the default path.

### Alternative 2: Keep DeepSeek R1 8B with Enhanced Prompting

Retain the current model but invest in prompt engineering: chain-of-thought extraction, few-shot examples of faithful journal entries, explicit anti-hallucination instructions.

**Pros**:
- No infrastructure changes
- No VRAM impact
- Fast iteration cycle (prompt changes, not model swaps)

**Rejected because**:
- **Capability ceiling**: The 14.3% hallucination rate is a model capability issue, not a prompt engineering issue. The October 2025 model evaluation demonstrated that 8B models achieve <80% correctness on structured tasks regardless of prompting strategy. Prompt engineering can reduce hallucination by 2-4 percentage points but cannot close a 9+ point gap to reach the <5% target.
- **Evidence from platform**: The RAG hallucination baseline showed that even with RAG grounding (which provides stronger evidence than OTEL events), the hallucination rate for smaller models remains at 20%. Prompting alone cannot compensate for insufficient model capacity.
- **Structured output reliability**: 8B models produce unreliable JSON — missing fields, malformed arrays, inconsistent schemas. The E2A architecture requires structured output as a first-class capability, which 14B+ models handle significantly better.

### Alternative 3: Fine-Tune DeepSeek R1 8B on Journal-Specific Data

Create a training dataset from the existing journal entries (manually curated for quality) and fine-tune the 8B model specifically for the journal extraction task.

**Pros**:
- Preserves VRAM budget of the smaller model
- Domain-specific optimisation could close the quality gap
- One-time effort with long-term benefit

**Rejected because**:
- **Insufficient training data**: The pipeline has produced approximately 30 journal entries to date, of which perhaps 5-10 are high enough quality to serve as training examples. Fine-tuning requires hundreds to thousands of examples to produce meaningful improvement.
- **Maintenance burden**: Every change to the journal format, voice guide, or content requirements would require retraining. The pipeline is still evolving (this ADR is evidence of that); locking in a fine-tuned model creates rigidity.
- **Infrastructure overhead**: Fine-tuning requires a training pipeline (data preparation, training runs, evaluation, model versioning) that does not currently exist on the platform. Building this for a single use case is disproportionate.
- **Deferred, not rejected permanently**: Once the pipeline is stable and producing consistent high-quality output (post-E2A), fine-tuning a smaller model on that output becomes viable as a cost optimisation. This is a Phase 2 consideration, not a Phase 1 approach.
