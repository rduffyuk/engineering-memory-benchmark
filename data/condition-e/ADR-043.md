# ADR-043 — Memory System Derived-Data GDPR Position

**Status**: Proposed
**Date**: 2026-05-17
**Jira**: [VW-391](https://ryanduffyuk.atlassian.net/browse/VW-391)
**Related design**: [[2026-05-17-Long-Term-Memory-Postgres-Design]]

---

## Context

The Rootweaver long-term memory system (VW-391) ingests Claude Code session JSONL transcripts and produces multiple layers of derived data stored across three distinct persistence technologies:

### Layer 1 — Postgres relational tables (`memory.*` schema)

The canonical store. Seven tables under the `memory` schema namespace, all anchored to `memory.sessions` via foreign keys with `ON DELETE CASCADE`:

| Table | Content | Personal-data classification |
|-------|---------|------------------------------|
| `sessions` | Session metadata: UUID, timestamps, branch, Jira refs, commits, file paths, summary text, summary embedding (pgvector 1024d) | **Direct identifiers** — file paths, Jira refs, branch names, summary text reveal the operator's work patterns |
| `rounds` | Per-turn user text, assistant text, assistant thinking (redacted at insert), tool calls, per-round embedding, message UUIDs | **Special category** — user_text is the operator's own words; assistant_thinking has documented 19-78% sensitive-value leakage rate per arXiv:2506.15674 |
| `facts` | LLM-extracted structured decisions, problems, learnings with reasoning chains, entities, per-fact embedding, temporal validity window | **Derived personal data** — entities array may contain names, project identifiers, org-specific terminology |
| `fact_corrections` | Negation/supersession records linked to facts | Inherits classification from parent fact |
| `artifacts` | References to git commits, vault docs, Confluence pages, Jira issues, PRs, file creates/modifies/deletes, K8s applies | **Behavioral data** — artifact graph reconstructs the operator's complete work history |
| `file_snapshots` | Point-in-time captures of file state during a session | **Direct content** — may contain source code, configuration, credentials |
| `backfill_progress` | Bookkeeping for the ingestion pipeline | Not personal data; operational metadata only |

All tables are defined in `packages/rootweaver-memory/src/rootweaver_memory/tables.py` using SQLAlchemy Core (not ORM) with explicit `ForeignKey(..., ondelete="CASCADE")` declarations. The `ExtractionResult` Pydantic model in `packages/rootweaver-memory/src/rootweaver_memory/models.py` enforces atomic write transactions per session — partial writes are structurally impossible, which is a prerequisite for cascade-delete correctness.

### Layer 2 — pgvector embeddings (co-located on Postgres)

Three embedding columns exist across the schema:

- `sessions.summary_embedding` — Vector(1024), summary of the whole session
- `rounds.embedding` — Vector(1024), per-turn semantic representation
- `facts.embedding` — Vector(1024), per-fact semantic representation

These are columns ON the relational tables, not a separate vector database. This is a deliberate architectural decision: co-locating embeddings with their source text means `ON DELETE CASCADE` automatically removes vectors when parent rows are deleted. The alternative — splitting vectors into a separate store as langchain+pgvector does (see langchain discussion #17499) — would have broken cascade-delete and required coordinating multi-system deletes.

### Layer 3 — FalkorDB knowledge graph

Entity-relationship triples derived from `facts` rows are projected into FalkorDB's `document_knowledge` graph via Graphiti. These represent abstracted knowledge: "Entity X relates-to Entity Y as-of timestamp T." The graph is a projection, not a source of truth — it can be fully reconstructed from Postgres at any time.

### The regulatory question

When an operator invokes the right to erasure (GDPR Article 17), which of these three layers must be deleted? The answer depends on whether derived data retains its classification as personal data after extraction.

### Regulatory landscape (as of May 2026)

**Hamburg DPA discussion paper (July 2024)** argued that LLM model parameters are not personal data — deletion of training inputs should not require deletion of weights. This was a significant statement from a major EU data protection authority, and it opened a window for the position that derived/transformed data might not carry forward the personal-data classification of its source.

**EDPB Opinion 28/2024 (December 2024)** rejected Hamburg's position. The European Data Protection Board held that anonymity requires that extraction or regurgitation of personal data from the model be impossible. For fact-extraction pipelines like ours — where the `facts.text` column literally contains LLM-extracted claims about the operator's decisions, including entity names and project identifiers — the extraction capability is not theoretical. It is the pipeline's explicit purpose. Under EDPB's reading, the derived facts remain personal data.

**EDPB Coordinated Enforcement Framework 2025** launched a coordinated enforcement action across all EU supervisory authorities specifically targeting the right to erasure. The report was due February 2026. This signals that erasure compliance is an active enforcement priority, not a theoretical concern.

**OWASP AI Security Verification Standard (AISVS) C08** provides the practitioner-level control framework. Section C08 on Memory, Embeddings, and Vector Databases explicitly mandates hard-delete plus tombstoning for right-to-be-forgotten requests. Soft-delete (setting a `deleted_at` flag without ever purging) is classified as non-compliant. The standard further requires that deletion propagate to all derived stores, including vector embeddings and knowledge graph projections.

**Practitioner consensus** (aicompetence.org, Mem0 security analysis, Brandur Leach's analysis of soft-deletion antipatterns) converges on hard-delete as the defensible default. The production PKM-plus-LLM-memory systems surveyed — Khoj, Letta, Cognee, Memori, Open Brain, Smart Connections, Notion — all co-locate source and derived data, and none have published postmortems attributing compliance failures to co-location.

### Why this matters for Rootweaver specifically

Ryan personally is not subject to GDPR for his own private notes on his own infrastructure (single-user, no third-party data processing). However, the project is planned for public release. Downstream users may operate it in multi-user, team, or service-with-customers contexts where GDPR applies. Defaults shape behaviour. Shipping with soft-delete-by-default would create GDPR exposure for every deployer who does not actively revisit the retention strategy. The conservative read costs little at implementation time and avoids a class of regulatory and reputational risk that is difficult to retrofit.

## Decision

**We treat all derived data in the memory system as personal data when the source transcript contains personal data, and we cascade-delete it on session erasure.**

### Implementation (seven controls)

**1. `ON DELETE CASCADE` from `sessions` propagates to all child tables.** A single `DELETE FROM memory.sessions WHERE id = ?` removes the complete derived graph: `rounds`, `facts`, `fact_corrections`, `artifacts`, `file_snapshots`. This is enforced in the SQLAlchemy table definitions (`tables.py`) and verified by integration tests in `packages/operator-tools/tests/test_operator_tools_memory_forget_cascade.py`, which uses testcontainers with pgvector/pgvector:pg16 to exercise the full cascade through a real Postgres instance.

**2. pgvector embeddings are deleted alongside their parent rows.** Because the three embedding columns (`sessions.summary_embedding`, `rounds.embedding`, `facts.embedding`) live ON the same tables as the source text, cascade-delete is automatic. We deliberately did not split embeddings into a separate table or a dedicated vector store. This is the single most consequential design choice for erasure compliance: it eliminates the vector-orphan problem that plagues systems using external vector databases (Milvus, Pinecone, Weaviate) alongside relational stores.

**3. FalkorDB knowledge graph nodes are reconciled weekly from Postgres source-of-truth.** The `document_knowledge` graph is a projection. Once Postgres cascade-deletes remove source rows, the next reconciliation pass drops FalkorDB nodes that have no corresponding Postgres parent. The reconciliation operates on a no-orphan policy: any node without a live Postgres source is removed.

**4. Seven-day grace tombstone before hard-delete.** The `memory-forget` operator tool (at `packages/operator-tools/src/operator_tools/memory_forget.py`) defaults to setting `sessions.tombstoned_at = now()`. A nightly cron job (`memory-tombstone-reaper`) executes `DELETE WHERE tombstoned_at < now() - interval '7 days'`, triggering the cascade. All MCP query tools and dashboard views apply an implicit `WHERE tombstoned_at IS NULL` filter, so tombstoned sessions are functionally invisible from the moment of marking.

**5. `--immediate` flag bypasses the tombstone grace period.** For actual GDPR-deadline requests (Article 12 allows one calendar month for response, but some operators may need faster turnaround), `memory-forget --immediate --session-id <uuid>` executes the hard `DELETE` in the same transaction. The `--dry-run` flag previews cascade counts (sessions, rounds, facts, artifacts, snapshots, corrections) without making changes.

**6. Redaction at insert for `assistant_thinking`.** The `rounds.assistant_thinking` column stores raw extended-thinking content from Claude's reasoning chain. Research documents a 19-78% rate of sensitive-value leakage in thinking traces (arXiv:2506.15674, "Leaky Thoughts"). NER-based scrubbing via the existing `ner-consumer` infrastructure removes API keys, email addresses, phone numbers, named persons, and credit card numbers before the row is persisted. This column is never embedded (marked in the schema comments, enforced by the writer).

**7. Source JSONL files are NOT auto-deleted by `memory-forget`.** The raw conversation archives under `conversation-archives/raw-jsonl/` are user-owned files on the operator's filesystem. Deleting them is an explicit operator action, documented in the operator handbook, separate from memory-system erasure. This separation is deliberate: the memory system should not reach outside its database to modify the operator's filesystem without explicit consent.

## Consequences

### Positive

- **Defensible against regulator inquiry.** The implementation aligns with EDPB Opinion 28/2024 (the most authoritative current interpretation) and OWASP AISVS C08 (the industry-standard AI security control framework). In a regulatory review, the project can demonstrate cascade-delete mechanics, tombstone-to-hard-delete workflow, and integration test evidence — not just a policy document.
- **Public release is GDPR-safe by default.** Downstream deployers inherit the strict position without having to audit or modify the retention strategy. This reduces the liability surface for the open-source project.
- **Single command erases a session completely.** The `memory-forget` tool provides `--dry-run` (preview), default (7-day tombstone), and `--immediate` (hard-delete) modes, covering operator-error recovery, standard requests, and deadline-driven requests respectively.
- **No orphaned vectors.** Co-locating pgvector embeddings on the same tables as source text makes vector cleanup automatic and verifiable via standard SQL introspection. There is no second store to coordinate with, no eventual-consistency window, no separate garbage-collection process.
- **Tombstone pattern protects against typo errors** without softening the eventual deletion guarantee. The 7-day window is long enough for an operator to notice a mistake and short enough to satisfy Article 17 timelines.
- **Co-location simplifies rather than complicates erasure.** The intuition that "separate databases give cleaner GDPR isolation" is wrong for this use case. Federated stores require coordinating multi-system DELETEs across Postgres, a vector database, and a graph database — three transactions, three failure modes, three consistency windows. A single `DELETE FROM memory.sessions WHERE id = ?` against a co-located schema is one transaction with deterministic cascade. The `memory` schema namespace provides logical isolation within `vault_index` while preserving cross-schema JOINs (`memory.artifacts.artifact_ref <-> public.notes.path`) that power `vault_doc_history` and `recall_origin` as single-query operations.

### Negative

- **Cascade can be expensive at scale.** A large session may produce hundreds of facts and artifacts. CASCADE through pgvector-indexed columns triggers index updates proportional to the number of deleted vectors. At the current single-user scale this is negligible (sub-second). At production scale with millions of facts, partition-aware deletion strategies may be needed (out of scope for v1).
- **Implicit tombstone filter on every query.** Every MCP tool and dashboard panel must apply `WHERE sessions.tombstoned_at IS NULL`. This is enforced by always routing through prepared tool wrappers and dashboard views rather than raw SQL, but operators writing ad-hoc queries must remember the filter.
- **Re-ingestion of a deleted session produces a new UUID.** Session IDs are never reused. Forensic continuity from old commit trailers (VW-393) to the new ingestion is broken by design — this is correct behaviour for GDPR (the erasure was meant to be permanent) but is a forensic-recovery cost.
- **Source JSONL not auto-deleted.** Some strict interpretations of GDPR would require deletion of the original record alongside derived data. We surface this as a documented operator responsibility rather than automating filesystem-level deletion from a database tool.

### Open questions / acknowledged limitations

- **Entity-level cascade is out of v1 scope.** "Forget everyone named X across all sessions" requires an entity-tag index that does not yet exist. The current granularity is session-level only.
- **Model weights are not in scope.** The memory system does not fine-tune models on ingested data. If a future capability adds fine-tuning from this corpus, this ADR must be revisited per the "What Should LLMs Forget?" literature (arXiv:2507.11128).
- **Query audit log is not in v1.** Single-user deployment defers this cleanly. Multi-user deployments will need to revisit for GDPR Article 30 (records of processing activities).
- **Backup retention.** Postgres backups (Velero snapshots, pg_dump CronJobs) may retain deleted sessions in older snapshots. Full erasure requires either backup-rotation policy alignment or selective restore-and-redelete. Documented in operator handbook.

## Alternatives Considered

### Alt 1: Soft delete with `deleted_at` flag, never purge

**Rejected.** Soft-delete without eventual hard-delete is explicitly classified as non-compliant by OWASP AISVS C08. Brandur Leach's analysis ("Soft Deletion Probably Isn't Worth It") documents the operational burden: soft-deleted rows accumulate indefinitely, bloating tables and indexes, while creating a false sense of deletion that does not survive regulatory scrutiny. The tombstone pattern we use is superficially similar but fundamentally different: it is a 7-day grace window before guaranteed hard-delete, not permanent retention behind a flag.

### Alt 2: Cascade-delete source rows but retain derived facts (Hamburg DPA position)

**Rejected.** Hamburg DPA's July 2024 position — that derived/transformed data does not inherit the personal-data classification of its source — was directly contradicted by EDPB Opinion 28/2024 five months later. The EDPB holds the more authoritative position, and the CEF 2025 coordinated enforcement action specifically targeting erasure compliance signals that the EDPB view will be operationalized across member states. Building on the contested-and-losing interpretation is an unforced error, especially for a project targeting public release.

### Alt 3: Redaction-at-query rather than redaction-at-insert for assistant_thinking

**Rejected.** Query-time redaction requires trusting the query layer to consistently apply the redactor — a trust boundary that does not survive prompt injection or misconfigured query paths. An attacker (or a buggy tool wrapper) could bypass the redactor and access raw thinking content containing leaked credentials or PII. Insert-time redaction permanently removes sensitive substrings before they reach persistent storage. The trade-off is that insert-time loses the original content; query-time risks leaking it. The leakage risk dominates.

### Alt 4: No `memory-forget` tool; use raw SQL

**Rejected** after discussion (2026-05-17). For personal single-user use, `psql -c "DELETE FROM memory.sessions WHERE id = '<uuid>'"` would be functionally sufficient. For public release, a typed wrapper script with `--dry-run` preview, `--immediate` flag, and the tombstone safety net is a meaningful safety improvement over expecting every operator to write correct SQL. The implementation cost is low: one Python file (133 lines), one cron job, one column addition.

## How to validate this ADR is being followed

1. **Schema cascade check** — every child table has `ON DELETE CASCADE` back to `memory.sessions.id`:
   ```sql
   SELECT conname, conrelid::regclass FROM pg_constraint
   WHERE contype = 'f' AND confrelid = 'memory.sessions'::regclass
     AND confdeltype = 'c';
   ```

2. **Embedding co-location check** — every pgvector column lives on a `memory.*` table with a `session_id` FK:
   ```sql
   SELECT table_schema, table_name, column_name
   FROM information_schema.columns
   WHERE udt_name = 'vector' AND table_schema = 'memory';
   ```

3. **Schema namespace check** — every VW-391 table lives in the `memory` schema:
   ```sql
   SELECT table_name FROM information_schema.tables
   WHERE table_schema = 'memory';
   ```

4. **Tombstone filter audit** — every MCP tool query and dashboard panel applies `WHERE sessions.tombstoned_at IS NULL`. Enforced via code review checklist.

5. **Redactor regression test** — `tests/regression/test_redactor_pii_seed.py` runs in CI on every PR touching `redactor.py`.

6. **Cron evidence** — `memory-tombstone-reaper` CronJob runs daily; Prometheus metric `memory_tombstone_reaper_rows_deleted_total` is non-zero whenever tombstoned sessions exist.

## When to revisit this ADR

- **CEF 2025 report publishes** — adjust position based on coordinated enforcement outcomes
- **Project moves to multi-user deployment** — entity-level cascade + query audit log become mandatory
- **Project adds fine-tuning from the memory corpus** — model weights become in-scope per arXiv:2507.11128
- **EDPB issues clarifying opinion on derived-data deletion** — re-anchor against latest guidance
- **A regulator inquiry occurs** in any deployment of this code — escalate to ADR revision regardless of outcome
