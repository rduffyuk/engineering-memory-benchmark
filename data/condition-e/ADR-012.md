# ADR-012: Journal Pipeline Quality Overhaul — Model Selection & E2A Architecture

**Status**: Proposed
**Date**: 2026-02-08
**Jira**: VW-33
**Supersedes**: Partial aspects of ADR-010 (Streaming Journal System)

## Context

The journal streaming pipeline, operational since late January 2026 (VW-6, VW-30), produces automated journal entries from Claude Code sessions. The pipeline works end-to-end: OTEL events flow into Kafka, triggers fire correctly, session parsing routes transcripts, and entries land in the vault. However, the content generated is not publication-ready, and investigation has revealed that a significant fraction of it is outright fabricated.

### Evidence of Quality Failure

Cross-referencing JSONL transcripts against OTEL-path journal entries for session `session_20260207_163955` revealed systematic fabrication:

| Journal Claims | Actual Session Content (JSONL) |
|---|---|
| `deploy_watchdog.sh` JSON conversion work | Writing Season 3 Episode 1 blog post |
| CI pipeline integration and Docker troubleshooting | Generating retro anime images via HuggingFace |
| Exit code debugging across 4 consecutive entries | Zero mentions of `deploy_watchdog.sh` anywhere in transcript |

Every OTEL-path entry was entirely fabricated. The voice and formatting were correct (the v8 prompt redesign had nailed tone), but the factual content was fiction. This was not a marginal quality issue but a categorical failure: the pipeline was generating plausible-sounding narratives about work that never happened.

### Root Causes Identified

Four compounding root causes were identified through systematic investigation:

**1. OTEL events are structurally too sparse (richness score: 2/10).** Claude Code's OTEL telemetry is privacy-stripped by design. The wire format carries `event.name`, `tool_name`, `decision`, `duration_ms`, `cost_usd`, `model`, and `token_counts` -- but not file paths, command text, prompt content, or tool result content. When the synthesizer receives 50 OTEL events, it has event names and timestamps but zero semantic grounding. No amount of prompt engineering can compensate for absent source data. The boundary detector first identified this sparsity, but the synthesis path continued to operate on it regardless.

**2. DeepSeek R1 8B is the wrong model for summarization.** DeepSeek R1 was trained via reinforcement learning for mathematical reasoning and chain-of-thought tasks. When given sparse input, it "overhelps" by filling gaps with plausible-sounding fiction -- a well-documented behavior pattern for RL-trained reasoning models. Vectara's Hallucination Leaderboard places R1's hallucination rate at approximately 14.3% for the full model; the 8B distillation performs worse. For comparison, general-purpose models like DeepSeek V3 hallucinate at roughly 4x lower rates. For journal entries that must be factually grounded in session transcripts, a 14%+ hallucination rate is catastrophic.

**3. Single-shot generation with no verification.** The pipeline generates content in a single LLM pass, writes it to the vault, and moves on. There is no quality gate, no check against source material, and no mechanism to detect when the model has fabricated content. Production content systems universally employ multi-stage pipelines with verification steps for this reason.

**4. Temperature too high (0.7).** The generation temperature of 0.7 encourages creative variation, which is desirable for conversational or artistic tasks but actively harmful for factual summarization. Research on extractive and abstractive summarization consistently shows that temperatures in the 0.3-0.5 range produce the best balance of factual grounding and readable prose.

### Pipeline Vision

The target architecture is a fully automated content pipeline: Claude Code session -> journal entry -> weekly blog post -> LinkedIn/social media distribution. This pipeline is currently blocked at the first stage because journal quality requires manual correction, which defeats the automation purpose entirely.

## Decision

We adopt a five-part architectural change to the journal synthesis pipeline, replacing the single-shot OTEL-dependent generation with a multi-stage Extract-then-Abstract (E2A) architecture using appropriate models at each stage.

### 1. Deprecate OTEL Synthesis Path

The OTEL synthesis path is structurally incapable of producing faithful journal entries because the input data lacks semantic content. No model improvement or prompt refinement can fix this fundamental data sparsity problem.

**Action**: Effectively disable OTEL synthesis by setting `MIN_OTEL_EVENTS = 999`, adding a deprecation log message. The JSONL path becomes the sole synthesis source. The OTEL boundary detector is preserved -- it remains useful for detecting session boundaries, milestones, and activity thresholds that trigger JSONL processing. It just must not attempt narrative synthesis from OTEL events.

**Rationale from typed memory**: A prior decision (session `ec1691bb`, 2026-05-04) explicitly confirmed "Disable the OTEL synthesis path" because "the OTEL path was producing fabricated outputs due to insufficient grounding" and "Use JSONL transcripts for replay instead of the OTEL stream" because "the OTEL stream lacks content payload." VW-276 later confirmed that 3,872 hallucinated entries (77% of `01-Journal/Streaming/`) accumulated over 3 months before the path was actually enforced as disabled, not merely documented as deprecated.

```
OTEL events -> boundary_detector -> journal.triggers -> session_parser -> JSONL processing
                                   (trigger only,       (finds JSONL file,
                                    no synthesis)         routes to E2A pipeline)
```

### 2. Replace DeepSeek R1 8B with Qwen3-14B for Extraction

The extraction stage requires a model that excels at reading comprehension and structured data extraction from long transcripts, not at mathematical reasoning. Qwen3-14B is a general-purpose model with strong summarization capability that fits within the RTX 4080's 16GB VRAM budget.

| Attribute | DeepSeek R1 8B | Qwen3-14B |
|---|---|---|
| Hallucination rate | ~14%+ (RL reasoning model) | ~4-6% estimated (general purpose) |
| Writing quality | Poor (math/reasoning optimized) | Good (broad training mix) |
| VRAM (Q4 quantization) | ~5GB | ~9GB |
| Fits RTX 4080 (16GB) | Yes | Yes, with KV cache headroom |
| Summarization strength | Weak | Strong |

A prior decision (session `af5afaf7`, 2026-03-08) confirmed the model swap: "Swapped from Qwen3-14B-AWQ to Qwen3-8B-AWQ" for context-capacity reasons, indicating that the Qwen3 family was already validated for this workload by March 2026. The extraction model runs locally on vLLM at temperature 0.3.

### 3. Implement Extract-then-Abstract (E2A) Two-Stage Pipeline

This is the single most impactful architectural change. The E2A pattern, documented in summarization research (arXiv:2410.06520v1), demonstrates dramatically reduced hallucination when the synthesis model operates from structured, verified facts rather than raw transcripts.

```
Stage 1: EXTRACT (Local -- Qwen3-14B on vLLM)
  Input:  JSONL session transcript (formatted)
  Output: Structured JSON via guided_json (vLLM response_format)
  Schema: ExtractedFacts {
    session_date, duration_estimate,
    primary_goals[],
    technical_decisions[{decision, reasoning}],
    files_modified[{path, change_type, description}],
    problems_encountered[{problem, solution, resolved}],
    tools_and_services[],
    key_learnings[],
    collaboration_highlights[]
  }
  Temperature: 0.3
  Purpose: Extract only facts verifiably present in transcript

Stage 2: SYNTHESIZE (API -- Claude Sonnet 4.5 Batch)
  Input:  Extracted facts JSON + style prompt + previous entry context
  Output: Publication-ready journal entry (markdown)
  Temperature: 0.5
  Purpose: Transform verified facts into Ryan's voice and narrative style
```

**Why E2A prevents hallucination**: The synthesis model never sees the raw, noisy transcript. It works exclusively from the structured `ExtractedFacts` JSON. If a fact was not extracted in Stage 1, it cannot appear in Stage 2. The extraction schema (verified in `scripts/journal/streaming/synthesizer/extraction_schema.py`, VW-33) constrains Stage 1 output via vLLM's `guided_json` parameter, ensuring the model produces valid structured data rather than free-form text that might drift into fabrication.

The `facts_to_prompt_context()` function in `extraction_schema.py` formats the structured facts into a clean, readable block that replaces the raw transcript in the synthesis prompt. This boundary between stages is the key architectural guarantee against hallucination.

### 4. Add LLM-as-Judge Quality Gate

A third-stage verification step evaluates the synthesized entry against the extracted facts before publication:

```
Stage 3: JUDGE (API -- Claude Haiku 4.5)
  Input:  Draft entry + extracted facts JSON
  Output: JudgeVerdict {
    factual_grounding (0-10),
    completeness (0-10),
    hallucination_detected (boolean),
    tone_style (0-10),
    coherence (0-10),
    overall_pass (computed),
    feedback (free text),
    revision_suggestions[]
  }
  Threshold: All scores >= 7 AND hallucination_detected == false
  Actions:
    Pass -> auto-publish to vault
    Marginal (min_score >= 5, no hallucination) -> revision loop (max 1 attempt)
    Fail -> flag for human review
```

The judge schema (`scripts/journal/streaming/synthesizer/judge_schema.py`, VW-33) recomputes `overall_pass` from actual scores rather than trusting the model's self-assessment, clamps all scores to the 0-10 range, and provides structured revision suggestions for the marginal case.

Research on LLM-as-judge approaches shows approximately 80% agreement with human evaluators at roughly 98% lower cost than human annotation. For an automated pipeline where the alternative is no quality gate at all, this represents a substantial improvement.

### 5. Adjust Inference Parameters

| Parameter | Before | After | Rationale |
|---|---|---|---|
| Temperature | 0.7 | 0.3 (extraction) / 0.5 (synthesis) / 0.1 (judge) | Lower for factual accuracy; judge near-deterministic |
| Max tokens | 2048 | 1500 (extraction) / 2048 (synthesis) | Extraction should be concise structured data |
| Model | deepseek-r1-0528-qwen3-8b | Qwen3-14B local + Claude Sonnet 4.5 API | Quality uplift at each stage |

## Consequences

### Positive

- **Hallucination eliminated by architecture, not by hope.** The E2A boundary ensures the synthesis model only sees verified facts, not raw noisy transcripts. This is a structural guarantee rather than a prompt-engineering aspiration.
- **Publication-quality output at negligible cost.** At ~$0.78/month, the hybrid pipeline enables the full automated content chain (session -> journal -> blog -> social) that was the original design goal of ADR-010.
- **Measurable quality via LLM-as-Judge.** The judge stage produces structured scores per entry, enabling quality tracking over time and regression detection across model or prompt changes.
- **Composable extraction output.** The `ExtractedFacts` JSON produced by Stage 1 can feed multiple downstream consumers: daily journal synthesis, weekly blog aggregation, LinkedIn post formatting, and Jira-linked session summaries. The extraction work is done once and reused.
- **No new hardware required.** Qwen3-14B fits on the existing RTX 4080 with room for KV cache. The GPU allocation model (vLLM exclusive, Ollama CPU-only) documented in the platform guardrails is preserved.

### Negative

- **API dependency for synthesis and judge stages.** Network connectivity and Anthropic API availability become requirements for the synthesis and judge stages. Mitigation: the extraction stage runs locally and can buffer extracted facts during API outages.
- **Increased pipeline complexity.** Three stages (extract, synthesize, judge) plus an optional revision loop versus the current single-shot approach. Each stage introduces a potential failure point. Mitigation: the Kafka-based architecture provides natural retry and dead-letter-queue support.
- **Slightly higher latency.** API calls add 5-15 seconds per entry versus 2-5 seconds for local-only inference. For a daily journal pipeline, this latency is negligible. If same-day publishing urgency arises, standard API pricing (2x batch) provides lower latency.
- **Model download required.** Qwen3-14B is approximately 9GB. This is a one-time cost on the existing vLLM infrastructure.

### Risks

- **Claude API pricing changes.** Currently negligible at ~$0.78/month; would require a 100x+ price increase to become material. The local-only fallback (quality 5/10) remains available if needed.
- **Qwen3-14B extraction quality untested on actual transcripts.** Mitigation: A/B test the first 10 sessions against R1 8B output, measuring extraction completeness and hallucination count. The guided_json constraint reduces model-specific risk by forcing structured output.
- **Batch API latency.** Anthropic's Batch API can take up to 24 hours for processing. If same-day publishing is required, standard API pricing provides near-instant responses.

## Alternatives Considered

### A. Keep DeepSeek R1 8B, Improve Prompts Only

Rejected. The model's architecture is fundamentally misaligned with summarization tasks. R1 was trained via reinforcement learning to reason about math and logic problems; when given sparse input, it fills gaps with plausible fiction. VW-30 already applied grounding blocks and prompt hardening (v8/v9 prompt redesigns) which improved formatting but could not fix the underlying hallucination rate because the model's training objective conflicts with faithful summarization. A 14%+ hallucination rate cannot be prompt-engineered away.

### B. Upgrade to 70B+ Local Model

Rejected for now. Models like Qwen 2.5 72B or Llama 3.3 70B would provide substantially better extraction quality, but require approximately 40GB VRAM at Q4 quantization. The RTX 4080 has 16GB. This would require either multi-GPU inference (not supported in the current K3s deployment) or a new GPU. Revisit if Qwen3-14B extraction quality proves insufficient -- the E2A architecture is model-agnostic at the extraction stage.

### C. Fully API-Based Pipeline (All Stages via Claude API)

Viable but unnecessarily expensive and architecturally simpler than needed. Local extraction is free, fast (2-5 seconds on vLLM), and keeps the GPU meaningfully utilized. The hybrid approach saves roughly $1.63/month versus full API while keeping the quality-critical synthesis and judge stages on the most capable available models. If the local extraction stage proved inadequate in practice, migrating it to API would be a configuration change, not an architectural one.

### D. Fine-Tune a Small Model on Ryan's Writing Style

Interesting but premature. Fine-tuning requires a curated dataset of "good" journal entries, which do not yet exist because the pipeline has not yet produced publication-quality output. After 3-6 months of E2A-generated entries at quality 8/10, the accumulated corpus could serve as training data for a style-specific model that might reduce or eliminate the API dependency for synthesis. This is a natural Phase 2 optimization, not a prerequisite.
