# ADR-034: Data Source Connector Architecture

**Status**: Proposed
**Date**: 2026-04-15
**Jira**: VW-184
**Deciders**: Ryan Duffy

## Context

Rootweaver ingests data from a single source: the local Obsidian vault. A file-watcher detects changes, publishes events to Kafka (`vault.file-events`), and a KEDA ScaledJob consumes them through a chunker-embedder-Qdrant pipeline. The result is a 148K-chunk vector store powering multi-agent RAG search with temporal filtering, hybrid BM25+vector retrieval, and knowledge-graph enrichment via FalkorDB.

Jira and Confluence are accessible through the Atlassian MCP server, but their content lives outside the search index. They are available only as synchronous API calls during active Claude Code sessions. When a session ends, that context vanishes. GitLab CI/CD data, Slack conversations, and email threads are not accessible at all.

This single-source limitation blocks three capabilities that the platform's architecture otherwise supports:

1. **Cross-source search**. A query like "what happened on the search pipeline last week" should return vault notes, Jira issues, GitLab MR discussions, and Slack threads in a single ranked result set. Today it returns vault content only; the user must separately invoke MCP tools for each external source and mentally fuse the results.

2. **Temporal filtering on external data**. The temporal search implementation (ADR date-aware retrieval, `start_date`/`end_date` parameters on `auto_search_vault`) works only on vault documents because external data has no `date_created` or `updated_at` payload fields in Qdrant.

3. **Knowledge graph linking across sources**. The NER pipeline (spaCy + EntityRuler + YAKE) and the `document_knowledge` FalkorDB graph only see vault documents. Entities mentioned in Jira issues that also appear in vault notes are not linked, preventing cross-source entity resolution.

Market comparison underscores the gap. Glean indexes 100+ data sources into a unified enterprise graph. Onyx (formerly Danswer), the closest open-source architectural peer, ships 40+ connectors with a `BaseConnector` interface, incremental sync via `load_from_state()`/`poll_source()`, and a normalised `Document` model. Rootweaver has one connector (vault) and two live-query integrations (Jira, Confluence via MCP).

## Decision

### Kafka-Native Connector Framework with Onyx Shim Layer

Build connectors as Kafka producers emitting normalised `ConnectorDoc` messages to a `vault.connector-docs` topic, consumed by the existing chunker-embedder-Qdrant pipeline. Adapt Onyx's MIT-licensed connector implementations through a thin shim layer rather than adopting Onyx wholesale or writing connectors from scratch.

The architecture has four layers:

**Layer 1 -- Onyx Shim** (`packages/rootweaver-connectors/src/rootweaver_connectors/onyx_shim/`). A Python namespace package that makes Onyx connector imports resolve to local stubs. It replaces five Onyx internal dependencies (Vespa client, enterprise access control, file store, DB models, indexing heartbeat) with no-ops or minimal implementations. This lets Onyx connector code run unmodified except for a single global substitution (`retry` replaced by `tenacity`). The shim is 20+ files totalling approximately 1,400 lines. `sys.path` manipulation in `factory.py` ensures the shim resolves before any global packages.

**Layer 2 -- Connector Sources** (`packages/rootweaver-connectors/src/rootweaver_connectors/sources/`). Nine Onyx connectors copied under MIT license, one directory per source. Total: 9,209 lines. Each connector implements Onyx's `CheckpointedConnector` interface with `load_from_checkpoint()` for incremental sync.

| Connector   | Lines | What Gets Indexed                      |
|-------------|-------|----------------------------------------|
| Jira        | 1,394 | Issues, comments, worklogs             |
| Confluence  | 2,883 | Pages, comments, spaces                |
| GitLab      | 270   | Issues, MRs, code files                |
| Discord     | 362   | Messages, threads, channels            |
| Slack       | 1,995 | Messages, threads, channels            |
| Gmail       | 634   | Emails, attachments                    |
| IMAP        | 559   | Generic email (any IMAP server)        |
| Web         | 801   | Any URL via crawl + extract            |
| File        | 326   | Local file upload (PDF, DOCX)          |

**Layer 3 -- Bridge and Models** (`bridge.py`, `models.py`, 202 lines combined). `ConnectorDoc` is a dataclass with JSON serialisation for Kafka production/consumption. `OnyxBridge` drives any Onyx connector through its checkpoint loop, converts Onyx `Document` objects to `ConnectorDoc`, and produces them to the `vault.connector-docs` Kafka topic. The `checkpoint_sync()` method handles `Document`, `ConnectorFailure`, and `HierarchyNode` item types, flushes the Kafka producer even on exceptions, and returns `(produced, failures, max_updated_at)` for watermark advancement.

**Layer 4 -- Factory, Indexer, and Scheduling** (`factory.py`, `indexer.py`, `indexer_runner.py`, 261+ lines combined). The factory maps source names to Onyx connector classes with per-connector config and a credentials environment variable. If the credential env var is unset, the connector is silently skipped -- enabling a connector requires only creating its SealedSecret. The indexer consumes `ConnectorDoc` messages from Kafka, chunks at 750-word windows with 75-word overlap, embeds via the configured model, and upserts to Qdrant's `rootweaver_vault` collection through `QdrantAdapter` (VW-235 dual-write for v1+v2 collection migration). Point IDs are deterministic UUID5s keyed on `source/source_id/chunk_index`, ensuring upsert idempotency.

### Qdrant Collection Strategy

Single collection (`rootweaver_vault`) with `source` and `source_id` added to `FACET_INDEX_FIELDS` for payload filtering. This was chosen over per-source collections because:

- Hybrid search (BM25 + vector) works unchanged -- no federation logic needed.
- Temporal filtering applies uniformly across sources.
- The existing `vault_section` facet pattern extends naturally to a `source` facet.
- For a single-user platform, collection-per-source adds complexity without benefit.

The `source` filter is plumbed through `SearchRequest` to Qdrant's `build_filter()` helper (fixed in VW-186, commit `c392e56`). A known follow-up: the agent path (`FastSearchAgent`, `DeepResearchAgent`, `OracleAgent`) does not yet forward the `source` filter through its `search_async()` signatures (noted in VW-186 comments as a deferred plumbing change).

### Scheduling and State Management

Connector syncs are deployed as K8s CronJobs. The Jira connector scheduler (`connector-jira-scheduler`) runs on a configurable cron, invokes `JiraConnector.load_from_checkpoint()` via `OnyxBridge.checkpoint_sync()`, and advances a Postgres watermark in `connector_sync_state`. KEDA's `connector-indexer` ScaledJob consumes the Kafka topic and scales 0-to-2 based on consumer lag, scaling back to zero when drained.

VW-188 (Jira connector activation) validated the producer side end-to-end: 231 Jira issues synced in one cycle, watermark advanced correctly, all 231 messages confirmed on the Kafka topic. The consumer side was blocked by VW-232 (the indexer had no `__main__` block, so KEDA-spawned pods exited silently). VW-243 designed a DLQ replay tool for messages that fail embedding (the expected path when vLLM-embedding is offline).

### Why Onyx Fork Over Alternatives

**vs. Airbyte**: Airbyte connectors are Docker containers with full network access, no source code auditability, and ELv2 licensing constraints. Onyx connectors are readable Python under MIT license.

**vs. writing from scratch**: Onyx's Slack connector alone is 1,995 lines of battle-tested edge-case handling (pagination, rate limiting, thread resolution, channel filtering). Rewriting this from scratch would take 2-3x the effort for equivalent reliability.

**vs. adopting Onyx wholesale**: Onyx uses Vespa for search (we use Qdrant), has no Kafka pipeline (we need event-driven architecture), no temporal search strategies, and no benchmark regression suite. Deploying Onyx alongside Rootweaver would double infrastructure without integrating into our existing pipeline.

**vs. keeping MCP-only**: The current MCP integration provides bidirectional access (read + write Jira/Confluence) but data is available only during active sessions, not indexed for search, not temporally filterable, and not linked in the knowledge graph.

The decision preserves the MCP tools for write operations (create/update Jira issues, edit Confluence pages) while connectors handle read-at-scale ingestion. The two complement rather than replace each other.

## Consequences

### Positive

- **Cross-source search**: A single `auto_search_vault` query returns results from vault, Jira, Confluence, GitLab, and other indexed sources, ranked by relevance via hybrid search with RRF fusion.
- **Temporal filtering across sources**: The `start_date`/`end_date` parameters work on connector documents because the indexer populates `date_created`, `timestamp`, `year`, `month`, and `day` payload fields from `ConnectorDoc.updated_at` or `created_at`.
- **Knowledge graph enrichment**: Connector documents flow through the existing NER pipeline and `document_knowledge` graph, enabling cross-source entity linking (e.g., a Jira issue key mentioned in a vault note is linked to the indexed Jira issue).
- **Benchmark evaluation**: The benchmark suite can evaluate cross-source retrieval quality with a new `cross_source` query category.
- **Infrastructure reuse**: Kafka, KEDA, Qdrant, Prometheus, and the chunker-embedder pipeline are all existing components. The framework adds a Kafka topic and a KEDA ScaledJob.
- **Zero-cost idle**: KEDA scales the indexer to zero when no messages are pending. Connectors without configured credentials are silently skipped.
- **Incremental sync**: Checkpoint-based watermarks in Postgres ensure only changed documents are re-indexed on each sync cycle.

### Negative

- **Increased Qdrant storage**: Connector documents increase collection size by 2-5x depending on volume. At current scale (148K vault chunks), this remains within single-node Qdrant capacity.
- **Credential management overhead**: Each connector requires a SealedSecret. Nine connectors means nine secrets to create, rotate, and monitor for expiry.
- **Connector maintenance burden**: Upstream Onyx changes (breaking API changes in data sources, new pagination patterns) require manual merge into the forked connector code. The fork is pinned to a specific commit to avoid surprise breakage.
- **Source filter plumbing incomplete**: The agent path does not yet forward `source` filters to Qdrant. Until VW-186-part-2 lands, source-specific queries via `auto_search_vault(source="jira")` work only on the direct retriever path, not through agent routing.
- **Consumer-side gap**: VW-232 (indexer `__main__` block) blocked the KEDA consumer. Until that fix ships, Jira documents sit on the Kafka topic but are not embedded into Qdrant. The DLQ replay tool (VW-243) is designed but not yet deployed for handling embedding failures.

### Risks and Mitigations

| Risk | Mitigation |
|------|------------|
| High-volume source (Slack) dilutes search quality | Per-channel filtering in connector config; message-length thresholds; `source` facet enables weighted retrieval in future |
| Silent connector failure produces stale data | `ConnectorSyncStale` Prometheus alert fires after 30 minutes of missed sync; `connector_sync_state` Postgres table tracks `last_run_status` per source |
| OAuth token expiry | SealedSecrets pattern (ADR-013) for current credentials; HashiCorp Vault (VW-185) for automated rotation in future |
| Onyx upstream breaking changes | Connector sources pinned to specific commit hash at fork time; manual merge only after audit |
| Embedding model change invalidates connector vectors | Kafka topic is replayable; replay re-embeds all connector documents with the new model |
| Single-collection noise at scale | Collection-per-source is a future migration path if quality degrades; current single-user scale does not warrant the federation complexity |

## Alternatives Considered

### Alternative 1: Adopt Onyx Wholesale

Deploy the full Onyx stack alongside Rootweaver for its connector ecosystem.

**Rejected** because it adds a Vespa dependency (second vector store), loses the Kafka/KEDA event-driven pipeline, provides no temporal search or benchmark integration, and doubles infrastructure for a single-user platform.

### Alternative 2: MCP-Only (Status Quo)

Continue using Atlassian MCP tools for live Jira/Confluence queries without indexing.

**Rejected** because it does not support cross-source search, temporal filtering on external data, knowledge graph linking, or offline access. Context is lost when Claude Code sessions end.

### Alternative 3: Build Connectors as MCP Tools

Add tools like `index_jira_to_qdrant` that a user manually invokes.

**Rejected** because it creates a manual-trigger model that does not scale, has no scheduled sync, and places the scheduling burden on the user rather than the platform.

### Alternative 4: Airbyte Connectors

Use Airbyte's 300+ connector catalogue with its Docker-based execution model.

**Rejected** because Airbyte connectors are opaque Docker containers with no source code auditability, require a separate Airbyte server deployment, use ELv2 licensing, and do not integrate with Kafka-native pipelines without additional adapter infrastructure.
