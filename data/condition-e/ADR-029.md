# ADR-029: KEDA-Based GPU Swap Controller for Embed/Inference Coexistence

**Status**: Accepted (retroactive -- ratifies the deployed VW-233 design)
**Date**: 2026-04-29
**Jira**: [VW-233](https://ryanduffyuk.atlassian.net/browse/VW-233) (swap controller), [VW-237](https://ryanduffyuk.atlassian.net/browse/VW-237) (GPU swap alerts), [VW-311](https://ryanduffyuk.atlassian.net/browse/VW-311) (vault-delta-indexer trigger), [VW-232](https://ryanduffyuk.atlassian.net/browse/VW-232) (connector-indexer rewire)

---

## Context

### The Single-GPU Constraint

The Rootweaver platform runs on a single NVIDIA RTX 4080 with 16 GB of VRAM. Three distinct workloads need GPU access:

1. **vLLM inference** (DeepSeek-R1-0528-Qwen3-8B-AWQ, ~15.6 GB VRAM at 95% utilisation). This is the primary LLM serving the AgentRouter, OracleAgent, SRE Brain, and the architecture-review pipeline. It runs as a StatefulSet in the `vllm` namespace with `nvidia.com/gpu: "1"` resource claims and `gpu-memory-utilization=0.95`.

2. **TEI embedding** (Qwen3-Embedding-4B, 2560-dimensional vectors, ~8 GB transient GPU footprint). This is the production embedder whose vector dimension matches the Qdrant `rootweaver_vault` collection. It serves two distinct demand patterns: a deterministic nightly bulk re-embedding window (03:00--05:00 UTC via `vault-nightly-reindex` CronJob) and ad-hoc bursts driven by the Jira connector pipeline (VW-232) landing batches of issue updates on the `vault.connector-docs` Kafka topic.

3. **vllm-embedding** (jina-v4-code). A separate embedder StatefulSet retained for code-graph workflows, sharing the same GPU contention characteristics.

The first two cannot coexist on 16 GB. vLLM's KV-cache allocator pins effectively the full card (`gpu-memory-utilization=0.95` translates to ~15.2 GB committed), leaving no headroom for TEI's model load. This is not a soft limit that can be tuned away -- DeepSeek-R1-0528-Qwen3-8B-AWQ's weight footprint plus KV cache at the configured `max-model-len=32768` and `max-num-seqs=2` simply consumes the available VRAM.

### HAMi Fractional Sharing: Investigated and Ruled Out

HAMi (Heterogeneous AI computing Management Interface) is installed in the cluster (`hami-system` namespace) and can partition GPU memory into fractional allocations via `nvidia.com/gpumem` resource requests. However, vLLM requires a full device claim (`nvidia.com/gpu: "1"`) and is incompatible with HAMi's fractional allocation model. More fundamentally, there is no fraction to give: 15.6 GB consumed out of 16 GB total leaves no slice for TEI regardless of the scheduling mechanism. HLD-03 (AI Layer) documents this as "HAMi: Installed but Bypassed" with the note that re-evaluation is warranted only after a GPU upgrade or a material reduction in vLLM's VRAM footprint.

Source: `obsidian-vault/09-System/Architecture/HLD/HLD-03-AI-Layer.md`, section "HAMi (Installed but Bypassed)"; typed-memory fact_id 2131 (GPU contention problem, session `642c9d97`); Jira VW-233 comment 2026-04-27 ("HAMi virtual GPU sharing investigated and ruled out -- vllm-0 uses 15.6/16GB GPU; no headroom for HAMi to slice").

### The Manual Swap Problem

Before VW-233, GPU swap was an operator pattern: manually scale TEI up before the nightly reindex, scale it down afterwards. This had two structural failure modes:

- **Ad-hoc Jira ingest spikes** (routine since VW-188 Jira connector go-live, 2026-04-26) needed TEI immediately, but TEI was scaled to zero outside the nightly window. Messages accumulated in `vault.connector-docs` and eventually DLQ'd -- approximately 600 messages were stranded in `vault.connector-docs.dlq` during the initial VW-232 deployment because the embedder was scaled to 0 the entire time the runner was processing messages.

- **Forgotten scale-down** left TEI holding the GPU, which caused vLLM to stay in `Pending` state. This was observed multiple times (typed-memory fact_id 2183: "vLLM pod failed to become ready due to GPU contention -- scaled down the TEI embedding to free the GPU for the vLLM pod"). During these windows, agent routing fell back to Ollama on CPU with confidence scores dropping from ~0.85 (LLM-driven) to ~0.3 (rules-fallback).

An additional discovery on 2026-04-27 (documented in VW-233 comments) revealed that FluxCD's 5-minute reconciliation interval was silently reverting any manual `kubectl scale` operations on `vllm-embedding`, because the GitOps manifest hardcoded `replicas: 0`. This meant the manual swap window was at most 5 minutes wide before FluxCD pushed it back -- far too short for any embedding workload to complete.

### Why Now

VW-188's Jira connector go-live transformed embedding demand from a predictable nightly batch into a pattern with routine daytime bursts. The manual operator model was untenable. A workload-aware, automated swap mechanism was needed.

## Decision

Deploy a **KEDA + PriorityClass GPU swap controller** that uses Kubernetes-native scheduling primitives -- no custom code -- to time-share the single GPU between inference and embedding workloads based on actual demand signals.

### Components

| Object | Kind | Manifest | Role |
|--------|------|----------|------|
| `vllm-embedding-priority` | PriorityClass (value=1000) | `infrastructure/vllm/vllm-priorityclasses.yaml` | `PreemptLowerPriority`. Applied to the TEI Deployment so that when KEDA scales TEI from 0 to 1, the K8s scheduler evicts vLLM (lower priority) to free the GPU. |
| `vllm-llm-priority` | PriorityClass (value=100) | (same file) | `PreemptLowerPriority`. Applied to the `vllm-0` StatefulSet. Lower value means it yields the GPU when a higher-priority pod needs scheduling. |
| `tei-embedding` | KEDA ScaledObject | `infrastructure/rootweaver/tei-embedding-scaledobject.yaml` | Scales TEI Deployment 0 <-> 1 based on three OR-semantics triggers. `pollingInterval=30s`, `cooldownPeriod=300s`, `fallback.replicas=0` on 3 consecutive trigger failures. |
| GPU swap health alerts | PrometheusRule | `infrastructure/monitoring/prometheus-rules.yaml` | VW-237: `GPUSwapStuck`, `GPUSwapMissedScaleDown`, `vLLMColdStartChurn`. |

### Triggers (OR Semantics)

The `tei-embedding` ScaledObject activates (scales to 1) when **any** of three triggers fires:

1. **Kafka lag: connector-indexer** -- consumer-group `connector-indexer` on topic `vault.connector-docs` exceeds `lagThreshold: 50`. This covers ad-hoc bursts when `connector-jira-scheduler` produces batches of issue updates. The threshold of 50 was chosen to avoid thrash on small trickles while activating promptly on meaningful batches.

2. **Kafka lag: vault-delta-indexer** (added VW-311) -- consumer-group `vault-delta-indexer` on topic `vault.file-events` exceeds `lagThreshold: 1`. This trigger was added after the initial VW-233 deployment when the vault-delta-indexer was migrated from in-process CPU SentenceTransformer embedding to GPU TEI via HTTP (a ~1700x speedup per VW-311). Without this trigger, TEI would scale to 0 during delta-indexer bursts, breaking embedding mid-stream.

3. **Cron window** -- daily 03:00--05:00 UTC, matching the `vault-nightly-reindex` CronJob's historical window. Guarantees TEI is available during the deterministic bulk re-embedding pass even if no Kafka lag is present at that moment.

### Swap Mechanics

```
Idle state:
  vllm-0 (priority=100) holds GPU. TEI scaled to 0 replicas.

Trigger fires (any of: connector-indexer lag > 50, vault-delta-indexer lag > 0, cron 03:00 UTC):
  KEDA scales TEI Deployment 0 -> 1.
  TEI pod has priorityClassName: vllm-embedding-priority (value=1000).
  Scheduler sees vllm-0 (priority=100) occupying the GPU.
  Preemption: vllm-0 evicted, GPU released.
  TEI starts, loads Qwen3-Embedding-4B (~30s on GPU), serves /v1/embeddings.

All triggers clear (lag returns to 0, cron window ends):
  cooldownPeriod=300s elapses with no further activation signal.
  KEDA scales TEI 1 -> 0. GPU released.
  vllm-0 rescheduled by K8s, reloads DeepSeek-R1-0528-Qwen3-8B-AWQ (~60s cold start).
  Inference serving resumes.
```

### Retry Coupling

The `connector-indexer` ScaledJob is configured with `RETRY_TRIES=5` and `RETRY_BASE_DELAY=2.0` (exponential backoff schedule: 2s, 4s, 8s, 16s = ~30s total). This is deliberately sized to absorb TEI's model load time after a cold KEDA scale-up, so that the first embedding requests arriving during the swap window retry through the warm-up period rather than immediately DLQ. Any future change to TEI's start-up latency or the retry budget must consider the coupling.

### KEDA Configuration Rationale

- **`minReplicaCount: 0`** -- GPU returns to vLLM during quiet windows. This was temporarily set to 1 during early VW-311 work, then reverted once the third trigger (vault-delta-indexer-lag) was in place, following the VW-296 SLO principle that "embedding throughput > LLM uptime" only when there is actual embedding work.

- **`cooldownPeriod: 300`** -- 5 minutes of post-trigger quiet before scale-down. Prevents thrash on bursty Kafka traffic and lets a vault-nightly-reindex job complete a batch without premature termination.

- **`fallback.replicas: 0`** -- If the KEDA trigger backend (Kafka) is unreachable for 3 consecutive polling intervals, TEI scales to 0 rather than holding the GPU indefinitely. This is a safety valve: inference availability is more valuable than speculative embedding readiness.

- **`paused-replicas` annotation vs `fallback.replicas`** (typed-memory fact_id 2384, session `8111ef83`): During implementation, the team discovered that KEDA's `fallback.replicas` does not force-scale when the scaler reports 0 lag. The `paused-replicas` annotation was used instead to gate TEI in specific maintenance scenarios.

## Consequences

### Positive

- **Zero idle embedding cost.** TEI runs at 0 replicas when there is no Kafka lag and no cron window, reclaiming ~8 GB VRAM headroom for inference. In observed production, TEI is active less than 10% of the day.

- **Bounded inference outage.** vLLM's cold-start is ~60 seconds. During this window, agent routing gracefully degrades to Ollama (CPU-only, ClusterIP 11434). The system stays nominally available -- search queries route through FastSearchAgent or the rules-fallback path.

- **Workload-driven, not operator-driven.** Eliminates the two manual failure modes (forgotten scale-down, delayed scale-up). The Kafka lag signal fires within 30 seconds of new work arriving.

- **Compatible with existing KEDA patterns.** The ScaledObject extends a pattern already established across 8 other KEDA-managed workloads in the cluster: `connector-indexer`, `vault-delta-indexer`, `ner-consumer`, `vault-postgres-delta-indexer`, `arch-review-batcher`, `arch-review-extractor`, `arch-review-scorer`, `arch-review-vault-writer`. No new mechanism introduced.

- **Zero custom code.** The entire swap mechanism is declarative YAML: two PriorityClasses, one ScaledObject, one Deployment annotation. No Go controllers, no Python daemons, no sidecar processes.

### Negative / Accepted Tradeoffs

- **vLLM cold-start latency on every swap (~60 seconds).** This is the cost of time-sharing a single GPU. Mitigated by: (a) embedding bursts cluster temporally (one swap per burst, not per request), (b) the nightly window swaps at most twice per day, (c) Ollama provides a CPU fallback during the gap.

- **Single point of GPU contention by design.** Only one of {vLLM, TEI, vllm-embedding} can hold the GPU at any instant. Adding a fourth GPU workload would require either a second physical GPU or a fundamentally different scheduling approach.

- **Observable but not silent eviction.** The swap produces visible events: pod eviction, cold starts, temporary routing degradation. The VW-237 alert family (`GPUSwapStuck`, `GPUSwapMissedScaleDown`, `vLLMColdStartChurn`) was deployed alongside to detect stuck states. However, if the monitoring stack itself is degraded (e.g., `dcgm-exporter` in CrashLoopBackOff, as observed 2026-04-29), swap observability reduces to Kafka consumer lag monitoring only.

- **NetworkPolicy surface area.** TEI's embedding endpoint (port 8090) must be reachable from every consumer: `connector-indexer`, `vault-delta-indexer`, `arch-review-extractor`, and `vault-nightly-reindex`. Each required its own egress NetworkPolicy update (VW-233 added explicit allow rules). Future embedding consumers must also be added to this allowlist.

- **Model alignment is load-bearing.** TEI must serve the same embedding model as the Qdrant `rootweaver_vault` collection (Qwen3-Embedding-4B, 2560 dimensions). Changing one without the other silently produces dimensionality mismatch errors. This coupling is why TEI replaced the prior `vllm-embedding` ScaledObject for VW-232 -- `vllm-embedding` was loading jina-v4-code, which is the wrong model for the general-text Qdrant collection.

### Operational Notes

- **Alerts to watch (VW-237)**: `GPUSwapStuck` fires if TEI is scaled up but vLLM is not evicted within 90 seconds. `GPUSwapMissedScaleDown` fires if TEI replicas > 0 with no Kafka lag and outside the cron window for more than 10 minutes. `vLLMColdStartChurn` fires on more than 3 evictions per hour, indicating thrash.

- **Manual override**: `kubectl scale` on the TEI Deployment is ineffective -- KEDA will revert the replica count within one `pollingInterval` (30 seconds). To force TEI up for an extended maintenance window, either widen the cron trigger in the ScaledObject manifest or use `kubectl annotate scaledobject tei-embedding autoscaling.keda.sh/paused-replicas="1"`.

- **Deployment strategy**: The TEI Deployment uses `strategy: Recreate` (not `RollingUpdate`) because its 18Gi memory limit cannot fit two pods simultaneously on the single node. This was a VW-389 finding -- `RollingUpdate` with `maxSurge=25%` rounds up to 1 on a 1-replica Deployment, causing a FailedScheduling event for the new pod.

- **Probe architecture**: TEI uses a three-probe pattern (VW-389, VW-412). `startupProbe` allows 25 minutes for cold-cache model download and 4B-parameter warmup. `livenessProbe` is TCP-only (not HTTP) because CPU-bound embedding work starves the HTTP `/health` handler past the timeout threshold, causing false liveness kills. `readinessProbe` remains HTTP to properly de-route traffic when TEI is genuinely wedged.

## Alternatives Considered

### A. HAMi Fractional GPU Sharing

**Rejected.** vLLM consumes ~15.6 of 16 GB VRAM. TEI needs ~8 GB to load Qwen3-Embedding-4B. The two cannot coexist regardless of slicing. HAMi can partition GPU memory but cannot manufacture it. Source: HLD-03 "HAMi (Installed but Bypassed)"; Jira VW-233 architectural discovery comment 2026-04-27. Re-evaluate after GPU upgrade or vLLM footprint reduction.

### B. Pure Cron Scheduling (No Kafka Trigger)

**Rejected.** Adequate for the deterministic nightly reindex (03:00--05:00 UTC), but cannot serve the ad-hoc demand pattern introduced by VW-232's Jira connector and VW-311's vault-delta-indexer. A 24-hour cron grain would leave embedding requests queuing for hours during daytime Jira update bursts. The OR-trigger design (cron + Kafka lag) captures both deterministic and event-driven demand.

### C. Run TEI Permanently on CPU

**Deferred (subsequently accepted as VW-337).** At the time of this decision (2026-04-29), TEI on CPU was measured at ~5-10x slower than GPU for embedding throughput. The vault-nightly-reindex window would not fit in 2 hours. Additionally, the vault-delta-indexer had just been migrated from CPU SentenceTransformer to GPU TEI (VW-311) for a measured ~1700x speedup. Reverting to CPU would sacrifice this gain.

*Note*: VW-337 (filed 2026-05-11) subsequently revisited this alternative and proposed moving TEI to a CPU-only image (`ghcr.io/huggingface/text-embeddings-inference:cpu-1.9`) with reduced batch sizes (`--max-batch-tokens=2048`, `--max-client-batch-size=4`) and float16 weights. The rationale shifted: since vault-delta-indexer is a background job (vault file changes trickle in), the latency hit is acceptable; the user-facing latency that matters is LLM triage during incidents, which improves when vLLM has uncontested GPU access. The current deployed state reflects VW-337's CPU migration, but this ADR documents the original KEDA+PriorityClass design that proved the swap mechanism viable and whose scaffolding (ScaledObject, triggers, cooldown, alerts) remains in production regardless of TEI's compute backend.

### D. Add a Second GPU

**Rejected (capex-bound).** The natural long-term solution, but requires hardware procurement. The KEDA+PriorityClass design is intentionally reversible: when a second GPU lands, the priority annotations can be dropped and TEI can be pinned to the dedicated embedding GPU. The mechanism has zero lock-in.

### E. Inline Embedding Inside vLLM (Model Hot-Swap)

**Rejected.** vLLM's runtime is optimised for serving a single model with pre-allocated KV cache. Hot-swapping model weights would require tearing down the KV cache, loading new weights, and reallocating -- a process unlikely to be materially faster than the ~60-second cold start of a separate pod, and one that couples two workload lifecycles in a way that makes independent scaling impossible.
