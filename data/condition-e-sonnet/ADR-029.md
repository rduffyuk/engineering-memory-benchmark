# ADR-029: KEDA-Based GPU Swap Controller for Embed/Inference Coexistence

**Status**: Accepted (retroactive — ratifies the deployed VW-233 design)
**Date**: 2026-04-29
**Jira**: [VW-233](https://ryanduffyuk.atlassian.net/browse/VW-233) (controller), [VW-237](https://ryanduffyuk.atlassian.net/browse/VW-237) (alerts), [VW-232](https://ryanduffyuk.atlassian.net/browse/VW-232) (scaling target rewire), [VW-311](https://ryanduffyuk.atlassian.net/browse/VW-311) (third trigger: vault-delta-indexer-lag), [VW-296](https://ryanduffyuk.atlassian.net/browse/VW-296) (GPU contention incident, closed WAI)

**Related ADRs**:
- ADR-018 (Data Source Connector Architecture) — defines the connector pipeline that produces the Kafka lag signal driving this controller
- ADR-028 (Vault Structured Index Postgres) — establishes the vault-postgres tier; embedding model alignment with the Qdrant `rootweaver_vault` collection drove the choice of TEI (not the legacy `vllm-embedding` StatefulSet) as the scaling target

---

## Context

The platform runs on a single NVIDIA RTX 4080 (16 GB VRAM), hosted on a one-node K3s cluster. At minimum three distinct workloads compete for that GPU:

1. **vLLM serving** — DeepSeek-R1-0528-Qwen3-8B-AWQ, consuming ~15.6 GB at `gpu_memory_utilization=0.95`. Serves the OracleAgent, SRE Brain, architecture-review pipeline, and direct LLM calls from the MCP bridge.
2. **TEI embedding** — HuggingFace Text Embeddings Inference running Qwen3-Embedding-4B (2560-dimensional), producing vectors that match the active Qdrant collection `rootweaver_vault`. Used by `vault-nightly-reindex` (daily 03:00–05:00 UTC) and by `connector-indexer` (VW-232) when Jira deltas land on the `vault.connector-docs` Kafka topic.
3. **vllm-embedding StatefulSet** — a separate TEI-family container serving jina-v4-code, retained for code-graph ingestion workflows. Also carries `priorityClassName: vllm-embedding-priority`.

None of these workloads can timeshare the GPU: vLLM pins its slice at pod startup via CUDA context allocation, and TEI requires exclusive GPU access to load Qwen3-Embedding-4B in workable time. HAMi fractional allocation was evaluated and rejected for this pair (see Alternatives). The physical constraint is binary: at most one workload holds the GPU at any moment.

### Why the manual pattern broke down

Before VW-233, the operator workflow was:

1. Manually `kubectl scale deployment/tei-embedding --replicas=1` before the nightly reindex window.
2. Wait for vLLM to be evicted (it had no PriorityClass at this point, so eviction was not guaranteed).
3. After the reindex, manually scale TEI back to 0.

This produced two persistent failure modes:

- **Ad-hoc ingest bursts** introduced by VW-232 (April 2026) required TEI immediately upon Kafka lag appearing, with no human in the loop. The old pattern had no mechanism to respond to a Kafka signal.
- **Forgotten scale-down** left TEI running after the nightly window, keeping vLLM permanently evicted. This was observed multiple times before VW-233; each instance required operator triage.

VW-188's go-live (Jira connector, 2026-04-26) converted what had been a quarterly manual event into a routine several-times-per-day automation. The manual model was untenable.

Evidence from VW-296 (filed 2026-05-07) confirmed the ambient risk: vllm-0 cycled three times in a single working day. Forensics captured via a live watcher script (`/tmp/vllm-watcher-2026-05-07.log`) revealed the pod events:

```
22:30:07Z  FailedScheduling: 0/1 nodes available: 1 Insufficient nvidia.com/gpu
22:35:06Z  Scheduled to rduffy
22:45:05Z  Preempted: Preempted by pod ce186681-... on node rduffy
22:45:16Z  FailedScheduling: 0/1 nodes available: 1 Insufficient nvidia.com/gpu
22:50:06Z  Scheduled to rduffy (TEI scaled down → GPU freed)
```

The preemption was correct and intentional under VW-233's PriorityClass design; VW-296 was closed as Working-As-Intended. The SLO confirmed at that time: daily Kafka throughput across data pipelines (vault-delta-indexer, scout fleet, NER, journal automation, connector ingest) must clear by end-of-day; vLLM uptime is not in the SLO. Brief vLLM unavailability is acceptable when embedding workloads need the GPU.

---

## Decision

Deploy a **KEDA + PriorityClass GPU swap controller** that automates the embedding/inference GPU handoff without operator intervention.

The design has three components:

1. **Two PriorityClass objects** that give the K8s scheduler an unambiguous, deterministic preemption order between embedding pods (high priority) and the inference pod (low priority).
2. **A KEDA ScaledObject** (`tei-embedding`) that replaces the manual scale operator. KEDA watches Kafka consumer-group lag and a cron schedule; it scales TEI 0↔1 based on workload signal rather than operator timing.
3. **Prometheus alerting** (VW-237) that detects stuck swap states, missed scale-down, and eviction churn — making the swap observable without requiring constant operator attention.

### Priority class manifest

`rootweaver-gitops/infrastructure/vllm/vllm-priorityclasses.yaml` — deployed live, verified against the cluster:

```yaml
apiVersion: scheduling.k8s.io/v1
kind: PriorityClass
metadata:
  name: vllm-embedding-priority
value: 1000
globalDefault: false
preemptionPolicy: PreemptLowerPriority
description: >-
  High priority for embedding workloads. KEDA-driven scale-up preempts
  vllm-0 to free the GPU. VW-233.
---
apiVersion: scheduling.k8s.io/v1
kind: PriorityClass
metadata:
  name: vllm-llm-priority
value: 100
globalDefault: false
preemptionPolicy: PreemptLowerPriority
description: >-
  Lower priority for vllm-0 (LLM serving). Gets preempted when an
  embedding workload is scheduled by KEDA. VW-233.
```

Applied to pods:
- `vllm-0` StatefulSet: `priorityClassName: vllm-llm-priority` (value=100)
- `tei-embedding` Deployment: `priorityClassName: vllm-embedding-priority` (value=1000)
- `vllm-embedding` StatefulSet (jina-v4-code): same high-priority class as TEI

### KEDA ScaledObject

`rootweaver-gitops/infrastructure/rootweaver/tei-embedding-scaledobject.yaml`:

```yaml
apiVersion: keda.sh/v1alpha1
kind: ScaledObject
metadata:
  name: tei-embedding
  namespace: rootweaver
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: tei-embedding
  minReplicaCount: 0
  maxReplicaCount: 1
  pollingInterval: 30
  cooldownPeriod: 300
  fallback:
    failureThreshold: 3
    replicas: 0
  triggers:
    - type: kafka
      name: connector-indexer-lag
      metadata:
        bootstrapServers: "kafka-cluster-kafka-bootstrap.kafka:9092"
        consumerGroup: "connector-indexer"
        topic: "vault.connector-docs"
        lagThreshold: "50"
        offsetResetPolicy: earliest
    - type: kafka
      name: vault-delta-indexer-lag         # added VW-311
      metadata:
        bootstrapServers: "kafka-cluster-kafka-bootstrap.kafka:9092"
        consumerGroup: "vault-delta-indexer"
        topic: "vault.file-events"
        lagThreshold: "1"
        offsetResetPolicy: earliest
    - type: cron
      name: vault-nightly-reindex-window
      metadata:
        timezone: "UTC"
        start: "0 3 * * *"
        end: "0 5 * * *"
        desiredReplicas: "1"
```

Triggers use OR semantics: TEI scales to 1 if **any** trigger fires.

| Trigger | Signal | Threshold | Purpose |
|---------|--------|-----------|---------|
| `connector-indexer-lag` | Kafka consumer-group lag | >50 messages | Ad-hoc Jira connector batches (VW-232) |
| `vault-delta-indexer-lag` | Kafka consumer-group lag | ≥1 message | Any vault file event (VW-311) |
| `vault-nightly-reindex-window` | Cron (UTC) | 03:00–05:00 | Deterministic nightly bulk re-embedding |

### Swap mechanics

```
IDLE STATE
  vllm-0 (priority=100) holds GPU.
  tei-embedding Deployment: 0 replicas.

TRIGGER FIRES (any Kafka lag OR cron window opens)
  KEDA scales tei-embedding Deployment: 0 → 1.
  TEI pod requires nvidia.com/gpu: 1.
  K8s scheduler: vllm-0 (priority=100) < tei-embedding (priority=1000).
  Preemption: vllm-0 evicted. GPU freed.
  TEI loads Qwen3-Embedding-4B (~30s cold start). Serves embedding traffic.

TRIGGER CLEARS (lag returns to 0 / cron window closes)
  cooldownPeriod=300s elapses with no further lag.
  KEDA scales tei-embedding: 1 → 0. GPU released.
  vllm-0 rescheduled. Reloads model (~60s cold start).
  SRE Brain and AgentRouter fall back to Ollama (CPU, llama3.2:3b) during the gap.
```

### Retry coupling

`connector-indexer` is configured with `RETRY_TRIES=5` and `RETRY_BASE_DELAY=2s`, producing approximately 30 seconds of cumulative exponential backoff. This is not coincidental: the backoff window is deliberately sized to absorb TEI's model load time (~30s), so the first few embedding requests arriving after a cold scale-up do not fail the consumer immediately. Any future tuning of TEI startup time or consumer retry budget must treat these as coupled parameters.

### Observability (VW-237)

Three alert rules in `infrastructure/monitoring/prometheus-rules.yaml`:

| Alert | Condition | Rationale |
|-------|-----------|-----------|
| `GPUSwapStuck` | TEI replicas > 0 but vllm-0 not evicted within 90s | Detects PriorityClass misconfiguration or scheduler failure |
| `GPUSwapMissedScaleDown` | TEI replicas > 0 with no Kafka lag and outside cron window for >10 min | Detects stuck ScaledObject or KEDA broker connectivity loss |
| `vLLMColdStartChurn` | >3 vllm-0 evictions per hour | Detects thrash from bursty Kafka traffic overwhelming the cooldown |

Note: As of 2026-04-29, `dcgm-exporter` is in CrashLoopBackOff. In that state, GPU occupancy is observable only via Kafka consumer-group lag metrics and vllm-0 pod scheduling events. The VW-237 alerts remain functional because they key off kube-state-metrics pod counts and KEDA ScaledObject status, not DCGM.

## Consequences

### Positive

- **No idle embedding cost** — TEI runs at 0 replicas during quiet windows. This reclaims ~8 GB of VRAM headroom for inference across approximately 90% of the day.
- **Bounded inference outage on swap** — vLLM cold start (~60s) is the only unavoidable cost when an embedding burst arrives. Ollama on CPU provides a routing fallback during that gap, keeping the system nominally available.
- **Workload-driven, not operator-driven** — eliminates the "forgot to scale down" failure mode and removes the human from the hot path of ingest latency.
- **Aligns with SLO** — the SLO is daily Kafka throughput; brief vLLM unavailability is explicitly acceptable. The design enforces the SLO automatically rather than relying on operator timing.
- **No new mechanism** — extends the existing KEDA ScaledJob pattern already used by `connector-indexer`, `vault-delta-indexer`, `ner-consumer`, and `arch-review-*`. Operators already understand KEDA semantics.
- **Reversible** — when a second GPU is available, the PriorityClass annotations can be dropped and each workload pinned to its own device. The ScaledObject Kafka triggers remain useful regardless.

### Negative / accepted tradeoffs

- **vLLM cold-start latency on every swap** (~60s). Burst traffic from Kafka means the inference pod can experience multiple evictions per day during heavy ingest periods. Under VW-296's confirmed SLO this is acceptable; it would not be acceptable if LLM latency were a primary SLO metric.
- **Single GPU contention ceiling** — this design manages exactly three workloads on one GPU. Adding a fourth GPU workload (e.g. a future fine-tuning job) requires either extending the PriorityClass hierarchy with a new tier or acquiring additional GPU capacity. The current design does not anticipate that expansion.
- **Eviction is observable only via Kafka and kube-state-metrics when DCGM is down** — the `dcgm-exporter` CrashLoopBackOff leaves a gap in GPU telemetry. This is a separate operational issue; the swap controller functions correctly regardless, but full observability requires DCGM to be healthy.
- **NetworkPolicy surface** — TEI ingress (port 8090) must be reachable from `connector-indexer`, `arch-review-extractor`, and `vault-nightly-reindex`. Each consumer required an explicit egress rule update in VW-233 (commit `19d74f0`). New consumers that need GPU embedding must update their NetPol before embedding calls will succeed.
- **Manual override is discouraged** — `kubectl scale` on the TEI Deployment is reverted by KEDA within one `pollingInterval` (30s). Operators who need to hold TEI up for an extended maintenance window must edit the ScaledObject's `fallback.replicas` or temporarily widen the cron trigger, not kubectl-scale the Deployment directly.
- **Model alignment is load-bearing** — TEI must serve the same model as the Qdrant `rootweaver_vault` collection (currently Qwen3-Embedding-4B, 2560d). Changing the TEI model without a corresponding Qdrant collection migration will silently produce dimensionality mismatches. This dependency is not enforced by any automated check.

## Alternatives Considered

### A. HAMi fractional GPU sharing

**Rejected.** HAMi is the current allocator that grants vLLM its 16 GB slice at pod startup. However, fractional sharing between vLLM and TEI is not viable: vLLM requests essentially the full card (~15.6 GB at 0.95 utilization), leaving fewer than 500 MB of addressable VRAM. TEI requires approximately 8 GB to load Qwen3-Embedding-4B with workable batch sizes. HAMi can partition a GPU among multiple smaller tenants; it cannot partition a GPU where one tenant occupies 97% of it.

### B. Pure cron scheduling (no Kafka triggers)

**Rejected.** Adequate for the deterministic nightly reindex, but insufficient for the VW-232 connector-indexer workload. Jira ingest events are bursty and unpredictable; they can arrive any time of day when a Jira webhook fires or a scheduled connector-jira-scheduler run produces issue updates. A 24-hour cron grain would leave embedding requests queued for hours during off-window bursts.

### C. TEI on CPU only

**Rejected at design time.** Benchmarked during the 2026-03-18 model migration: Qwen3-Embedding-4B at 2560d embeds approximately 10× slower on CPU than on GPU. The vault-nightly-reindex window (~2 hours) would not fit within its budget at CPU throughput. This alternative was revisited in May 2026 as VW-337 (filed 2026-05-11) with the rationale that the background indexing latency hit is acceptable given that LLM uptime is not in the SLO. VW-337 remains in Backlog; this ADR reflects the current deployed decision.

### D. Add a second GPU

**Rejected (capex-bound).** The natural long-term resolution, but requires hardware acquisition. KEDA+PriorityClass is the interim solution. The design is explicitly built to be dropped when a second GPU makes workload separation trivial: remove PriorityClass annotations from all pods, pin TEI to `nvidia.com/gpu: 1` on the new card, and vLLM keeps its existing allocation. The Kafka triggers in the ScaledObject remain independently useful.

### E. Inline embedding inside vLLM (in-process model swap)

**Rejected.** vLLM's serving runtime is not designed for hot-swapping model weights. In-process swaps would produce similar cold-start costs to the pod eviction approach (~60s) while coupling two unrelated workload lifecycles inside a single process. Operational complexity would increase significantly, and the failure blast radius of a TEI embedding error would extend to the LLM serving path.
