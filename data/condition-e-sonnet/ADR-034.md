# ADR-034: Data Source Connector Architecture

**Status**: Proposed
**Date**: 2026-04-15
**Jira**: VW-188
**Relates to**: VW-186, VW-232, VW-235

---

## Context

Rootweaver's RAG pipeline was built with the Obsidian vault as its sole data source. The vault indexer (neural-vault) uses inotify to watch for file changes, chunks Markdown, and upserts into Qdrant collection `rootweaver_vault`. This covers the local knowledge base well but leaves two categories of high-value content entirely outside the search corpus:

1. **Operational data**: Jira issues and Confluence pages, which are the primary record of engineering decisions, bug investigations, sprint work, and architectural discussions. At the time of this ADR, 231+ Jira VW-project issues existed and were not searchable via any RAG tool.
2. **Team communication and code**: GitLab commits and MRs, Slack messages, Discord, Gmail, and web content — all produced continuously and relevant to engineering queries.

The consequence is cross-source search failure. A query such as "what architecture decisions were made about the connector framework?" (observed in benchmark Run 40, VW-208) retrieves only vault documents even when the most recent and authoritative answer lives in a Jira ticket or Confluence page. The source filter introduced in VW-186 (`source="jira"` on SearchRequest) was implemented at the gateway boundary but silently dropped before reaching Qdrant — confirmed by smoke test on 2026-04-16 showing identical results with and without the filter.

A second driver is the absence of a structured ingestion path for connector-sourced content. The file watcher → Kafka → Qdrant pipeline is purpose-built for vault files. Connector documents (Issue, Page, MR, Message) require normalisation, chunking, metadata standardisation, and deduplication before they can share the same Qdrant collection and be returned alongside vault results.

This ADR records the design decisions made during VW-188 (first connector activation) to address both the ingestion gap and the cross-source retrieval gap.

---

## Decision

### 1. Onyx connector shim as the data extraction layer

Rather than write custom API clients for each source (Jira REST API, Confluence REST API, GitLab API, etc.), Rootweaver reuses the open-source [Onyx](https://github.com/onyx-dot-app/onyx) connector implementations via a vendored shim at `packages/rootweaver-connectors/src/rootweaver_connectors/onyx_shim/`.

The shim provides the Onyx `BaseConnector`, `CheckpointedConnector`, and `ConnectorCheckpoint` interfaces plus the `Document`, `ConnectorFailure`, and `HierarchyNode` output types. Each upstream connector (`JiraConnector`, `ConfluenceConnector`, `GitlabConnector`, `SlackConnector`, `GmailConnector`, `ImapConnector`, `WebConnector`, `FileConnector`, `DiscordConnector`) is imported directly through the shim path added to `sys.path` in `factory.py`.

This gives nine connectors with production-grade checkpoint/resume logic, rate-limit handling, and incremental sync (time-window based) without maintaining that logic ourselves. The shim surface we own is narrow: `interfaces.py`, `models.py`, `exceptions.py`, and four `cross_connector_utils` modules.

**Registry** (`factory.py`, `CONNECTOR_REGISTRY`):

| Source     | Class                | Credentials env var               |
|------------|----------------------|-----------------------------------|
| jira       | JiraConnector        | `CONNECTOR_JIRA_CREDENTIALS`      |
| confluence | ConfluenceConnector  | `CONNECTOR_CONFLUENCE_CREDENTIALS`|
| gitlab     | GitlabConnector      | `CONNECTOR_GITLAB_CREDENTIALS`    |
| discord    | DiscordConnector     | `CONNECTOR_DISCORD_CREDENTIALS`   |
| slack      | SlackConnector       | `CONNECTOR_SLACK_CREDENTIALS`     |
| gmail      | GmailConnector       | `CONNECTOR_GMAIL_CREDENTIALS`     |
| imap       | ImapConnector        | `CONNECTOR_IMAP_CREDENTIALS`      |
| web        | WebConnector         | `CONNECTOR_WEB_CONFIG`            |
| file       | FileConnector        | `CONNECTOR_FILE_CONFIG`           |

Connectors are instantiated lazily: `create_connector(source)` returns `None` if the credentials env var is not set, so unconfigured sources are silently skipped rather than failing startup.

### 2. Kafka as the normalisation boundary between producer and consumer

The `OnyxBridge` class (`bridge.py`) drives any `CheckpointedConnector` via `checkpoint_sync(start_ts, end_ts)` and converts each yielded `Document` into a `ConnectorDoc` (the Rootweaver canonical shape) before producing it to Kafka topic `vault.connector-docs`.

The `ConnectorDoc` dataclass (`models.py`) is the stable interface between producer and consumer:

```python
@dataclass
class ConnectorDoc:
    source: str        # "jira", "confluence", "gitlab", …
    source_id: str     # canonical ID within source (e.g. "VW-188")
    title: str
    content: str
    url: Optional[str]
    author: Optional[str]
    created_at: Optional[datetime]
    updated_at: Optional[datetime]
    metadata: dict[str, Any]
```

Kafka message key is `{source}:{source_id}` (e.g. `jira:VW-188`), enabling log-compaction deduplication. The producer calls `producer.flush()` in a `finally` block after the checkpoint loop so buffered messages are not lost if an exception interrupts mid-sync.

`HierarchyNode` items are intentionally dropped at the bridge layer (no consumer for hierarchy yet). `ConnectorFailure` items are logged and counted; the producer continues rather than aborting the sync.

### 3. Watermark-based incremental sync with Postgres state

Each connector source has a row in the `connector_sync_state` Postgres table with columns `source`, `scope`, `last_watermark` (timestamp), `last_run_status`, and `docs_produced`. The scheduler advances `last_watermark` only when `max_updated_at` returned from `OnyxBridge.checkpoint_sync()` is not `None` — i.e. when at least one document was yielded.

On the first sync, the watermark defaults to `2000-01-01T00:00:00+00:00` (effectively "sync everything"). Subsequent syncs pass `prev_watermark` as `start_ts` and `now()` as `end_ts`, achieving incremental delivery with no duplicate production risk (Kafka key compaction handles the consumer side).

Verified end-to-end on 2026-04-26 (VW-188 session wrap-up): 231 Jira issues produced in a single cycle, Postgres row showing `last_run_status=ok`, `docs_produced=231`, `new_watermark=2026-04-26T19:57:15.665+00:00`.

### 4. K3s CronJob scheduler (Path B, not Prefect)

Two deployment paths were considered during VW-188 brainstorming:

- **Path A**: Prefect scheduled flow — uses existing Prefect infrastructure, familiar pattern (used for journal automation, vault indexing). A Prefect flow was prototyped on 2026-04-15 (fact recorded in memory: session 5f5aba56, "Implemented a Prefect scheduled flow for Jira and Confluence integration").
- **Path B**: K3s CronJob — new dedicated scheduler pod, invokes `JiraConnector.load_from_checkpoint()` in a loop, produces to Kafka, KEDA `connector-indexer` ScaledJob consumes.

**Path B was selected** because:
1. The connector sync is a data pipeline step, not a workflow orchestration concern. Prefect's overhead (work-pool scheduling, worker assignment, flow-run state management) is unnecessary for a fixed-cadence, single-step job.
2. The existing KEDA `connector-indexer` ScaledJob was already deployed and waiting for `vault.connector-docs` Kafka messages. Path B lets the producer (CronJob) and consumer (ScaledJob) remain decoupled, consistent with the file-watcher → Kafka → vault-indexer pattern already in production.
3. Prefect scheduler reliability in this cluster has had prior issues (journal-watcher service stopped per VW decision 2026-05-10). A lightweight CronJob has fewer failure modes.

### 5. ConnectorIndexer as the consumer, routing through QdrantAdapter

The `ConnectorIndexer` class (`indexer.py`) consumes `ConnectorDoc` messages from Kafka, chunks content (750-word windows with 75-word overlap), embeds via the shared embedder, and upserts into Qdrant via `QdrantAdapter.add_documents()`.

A critical architectural constraint documented in VW-235: the indexer MUST use `QdrantAdapter` (not raw `QdrantClient`) so that the v1+v2 dual-write fires automatically. Earlier versions called `qdrant.upsert()` on a raw client and silently bypassed `_upsert_to_v2`, leaving the v2 collection stale for all connector data. The comment in `indexer.py` preserves this rationale permanently:

```python
# VW-235: Routes through QdrantAdapter (not raw qdrant_client) so v2 dual-write
# fires automatically.
```

Point IDs are deterministic UUIDs (`uuid5` with namespace `12345678-1234-5678-1234-567812345678`): `uuid5(NAMESPACE, f"{source}/{source_id}/{chunk_index}")`. This ensures upsert idempotency: re-indexing the same document replaces rather than duplicates.

Metadata schema for each indexed chunk includes `source`, `source_id`, `source_url`, `vault_section` (set to `source` for routing), `type="connector"`, `semantic_chunk=True`, plus timestamp fields and `meta_*` prefixed source-specific fields (e.g. `meta_priority`, `meta_status`, `meta_key` for Jira).

### 6. Source filter plumbing — gateway fixed, agent path deferred

VW-186 identified that `source` and `sources` fields added to `SearchRequest` were accepted at the gateway boundary but not forwarded to Qdrant. The fix (committed `c392e56`) added `rag_platform.core.vector_stores.qdrant.build_filter()` and repaired three call sites in the retriever chain.

The agent path (AUTO/FAST/DEEP agents in `rag_platform.agents.*`) was left as a deferred follow-up at the time of this ADR. The `_execute_vector_search_async(query, n_results, filters)` base method supports filters, but `search_async()` signatures across all three agents do not accept a `filters` dict and do not forward `request.source`. This means cross-source filtering via the agent path (`mode="auto"`) does not yet work — only direct retriever calls respect the source filter. Tracked as VW-186-part-2.

## Rationale for key choices

**Onyx shim vs. custom clients**: Custom API clients for nine sources would require implementing pagination, rate-limit handling, incremental sync, and error classification from scratch. The Onyx connectors have solved these for production deployments. The shim boundary is explicit (`onyx_shim/onyx/connectors/`) and import-isolated via `sys.path` manipulation, so upgrading or forking specific connectors remains tractable.

**Kafka as the normalisation boundary**: Separating the producer (connector scheduler) from the consumer (indexer) allows each to scale, fail, and be redeployed independently. The `vault.connector-docs` topic also provides a durable audit trail — any sync can be replayed by seeking the consumer offset. This was critical during VW-188 where the consumer side (ConnectorIndexer missing `__main__` block, VW-232) was broken after the producer was already validated. The 231 Jira messages remained on the topic and were consumed automatically once the consumer was repaired.

**Single Qdrant collection `rootweaver_vault`**: Connector documents share the same collection as vault documents. The `source` metadata field and `vault_section` field enable post-retrieval filtering and faceted search without separate collections. This preserves cross-source ranking — a highly-relevant Jira comment can rank above a less-relevant vault note in the same result set.

**Credential-gated activation**: Connectors that lack credentials env vars return `None` from `create_connector()` and are silently skipped. This means the same connector framework codebase is deployable in environments where only some sources are configured (e.g. Jira-only for the initial activation) without conditional logic in the scheduler.

## Consequences

### Positive
- Jira issues are now indexable into Qdrant, making VW-* ticket content searchable via `auto_search_vault(source="jira")` — enabling queries like "what was the root cause of VW-186?" to return the actual Jira comments.
- The same framework extends to Confluence, GitLab, Slack, Gmail, Discord, web, and file sources without architectural change — only credentials and a registry entry per source.
- Connector documents share ranking with vault documents in a single Qdrant collection, enabling cross-source results in a single query.
- Watermark-based incremental sync is idiomatic (used by all major enterprise search platforms) and avoids full re-indexing on every run.
- `uuid5`-based point IDs guarantee upsert idempotency, so re-runs are safe.

### Negative / Open risks
- **Agent source filter not wired (VW-186-part-2)**: Queries routed through AUTO/FAST/DEEP agents ignore `source` filters. Only direct retriever calls (`mode="fast"` without agent routing) respect them. Until this is fixed, `auto_search_vault(source="jira")` may not filter as expected via the standard MCP path.
- **Consumer blocking issue (VW-232)**: The `ConnectorIndexer` class initially lacked a `__main__` entrypoint, causing KEDA ScaledJobs to spawn pods that exited silently without consuming any messages. This was a framework gap found post-producer-validation, not a design flaw, but it delayed end-to-end validation.
- **Confluence and other sources not yet activated**: The registry has entries for all nine sources, but only Jira credentials were provisioned during VW-188. Confluence, GitLab, Slack, etc. remain dormant until credentials are added via SealedSecrets.
- **HierarchyNode dropped**: Onyx connectors can yield `HierarchyNode` items representing parent-child relationships (e.g. Confluence page trees, Jira epic-story links). These are dropped at the bridge layer. If hierarchical navigation is needed in the future, a separate consumer or metadata enrichment step would be required.
- **Onyx shim version lock**: The shim is vendored at a point-in-time snapshot of the Onyx connector interfaces. Upstream Onyx interface changes (e.g. new checkpoint fields, changed `Document` schema) would require a manual shim update and re-test of all connectors. This is a maintenance cost not present with custom clients.
- **Kafka claim-check not implemented for connector docs**: Large connector documents (e.g. long Confluence pages, Jira issues with many comments) could exceed the Kafka message size limit. The `_produce_document` method in `OnyxBridge` logs errors on oversized messages but does not implement the claim-check pattern (strip content, write to object store, produce pointer). Tracked as future work.

## Alternatives considered

### Not using Onyx shim — building custom connector clients

Each connector would be a purpose-written Python class calling the relevant REST API. This would give full control over the integration surface and no dependency on a third-party codebase, but would require implementing incremental sync, pagination, rate-limiting, and error handling for each source independently. Estimated 2–3 engineer-weeks per source vs. the hours required to wrap an existing Onyx connector. Rejected in favour of the shim approach.

### Using Prefect scheduled flows (Path A)

A Prefect flow was prototyped (session 5f5aba56, 2026-04-15). It would have reused the existing Prefect work-pool infrastructure and kept the connector sync visible in the Prefect UI alongside other flows. Rejected for the reasons documented in Decision 4: unnecessary orchestration overhead for a fixed-cadence single-step job, and prior reliability concerns with the Prefect scheduler in this cluster. The K3s CronJob + KEDA ScaledJob pattern is simpler and consistent with the existing vault-indexer pipeline.

### Separate Qdrant collection per source

Each source (`jira`, `confluence`, `gitlab`, etc.) could have its own Qdrant collection. This would make source filtering trivial (query the collection) and avoid metadata-field-based filtering. Rejected because it prevents cross-source ranking in a single result set — a core requirement for the RAG use case where the best answer to a query might be split across a Jira comment, a vault note, and a Confluence design page.
