# ADR-012: Journal Pipeline Quality Overhaul

**Status**: Proposed
**Date**: 2026-02-08
**Jira**: VW-33
**Supersedes**: Partial aspects of ADR-010 (Streaming Journal System) and ADR-011 (Journal Consolidator LLM Generation Tuning)

---

## Context

The journal streaming pipeline (VW-6, VW-30) converts Claude Code OTEL events and session JSONL transcripts into automated journal entries that flow to the Obsidian vault. As of 2026-02-08 the pipeline is structurally operational — Kafka topics flow, boundary-detection triggers fire, and entries are written to `01-Journal/` — but **content quality has failed to reach publication standard**.

### The Fabrication Incident

Cross-referencing JSONL transcripts against OTEL-synthesised journal entries for session `session_20260207_163955` (vault evidence: `2026-02-08-ADR-012-Journal-Pipeline-Quality-Overhaul.md` cross-reference table, cited in blog episode S3E2) revealed the following:

| Journal Claim | Actual Session Content |
|---|---|
| `deploy_watchdog.sh` JSON conversion work | Writing Season 3 Episode 1 blog post |
| CI pipeline integration and Docker troubleshooting | Generating retro anime images via HuggingFace |
| Exit code debugging across four consecutive entries | Zero mentions of `deploy_watchdog.sh` in full transcript |

Every OTEL-path journal entry for that session was **entirely fabricated**. Voice and format were correct (the v8 prompt redesign, VW-30, had solved style), but factual content was fiction. This is not an edge case: the same structural conditions apply to every session processed through the OTEL synthesis path.

### Root Causes

Three independent failure modes compound to produce this result:

**1. OTEL events are structurally too sparse (richness: 2/10).**
ADR-010 proposed OTEL as the primary event source. Post-implementation analysis of 94 JSONL files (875 MB total, median session ~50 K tokens — VW-30 investigation) showed that OTEL events export only `[event_name] Tool: X (decision) @ HH:MM:SS`. No file paths, no command arguments, no conversation context, no reasoning chain. An LLM given 50 such events has no semantic ground to work from. No prompt redesign can compensate for absent information. The JSONL transcripts — also produced natively by Claude Code — carry richness score 9/10: every message, every tool call with arguments, every file path, the full reasoning chain.

**2. DeepSeek R1 8B is architecturally misaligned with summarisation.**
ADR-010 selected DeepSeek R1 8B (via vLLM) for journal synthesis as the largest model fitting an RTX 4080 with adequate KV-cache for 49 K context. R1 was trained via reinforcement learning for mathematical reasoning; it "overhelps" — when given sparse input it fills inferential gaps with plausible-sounding fiction. Vectara's Hallucination Leaderboard (2025) places the full R1 model at ~14.3% hallucination rate, roughly four times worse than DeepSeek V3. The 8B distillation performs at or below that benchmark. ADR-011's A/B temperature experiments (presets: default 0.7, focused 0.4, balanced 0.5, detailed 0.6, creative 0.9) produced structurally better output but could not fix the underlying hallucination ceiling: all five presets still operated through a model with fundamental overhelping behaviour.

**3. Single-shot generation with no verification.**
The pipeline currently produces content in one unguarded pass: generate, write to vault, advance. No output is checked against the source transcript. No quality metric is computed. No threshold gates auto-publication. Production content systems at every scale use multi-stage pipelines for exactly this reason; one-shot generation of factual content from a high-hallucination model is not a viable configuration.

**Supporting decisions from memory (Postgres facts layer, session `af5afaf7`, 2026-03-08):**
A subsequent session recorded the decision to swap from Qwen3-14B-AWQ to Qwen3-8B-AWQ, noting "the journal pipeline was non-functional, allowing for a model with a higher context capacity" — confirming the pipeline was still in flux and model selection remained an open concern even after this ADR was written.

### Strategic Context

The long-term vision is full automation: Claude Code session → journal entry → weekly blog post → LinkedIn posts. This pipeline is currently blocked. Human time spent manually re-writing fabricated journal entries defeats the purpose of the automation and delays publication of the Building-in-Public series.

## Decision

### 1. Deprecate the OTEL Synthesis Path

The OTEL synthesis path is structurally incapable of producing quality journal entries. No prompt instruction or quality gate applied downstream of sparse OTEL events can substitute for missing semantic content.

**Action**: Set `MIN_OTEL_EVENTS = 999` in the synthesiser configuration to effectively disable OTEL-triggered synthesis without removing the code. Add a deprecation warning log: `WARN: OTEL synthesis path disabled — richness score 2/10, superseded by JSONL path (ADR-012)`. Remove entirely in a subsequent clean-up version.

The OTEL boundary detector retains its value for triggering JSONL session processing (detecting session end, topic changes, significant milestones). It is the triggering mechanism that is preserved; only synthesis-from-OTEL is deprecated.

**New canonical data flow:**
```
OTEL events → boundary_detector → journal.triggers → session_parser
                 (trigger only;                          → JSONL file lookup
                  no synthesis)                          → E2A pipeline
```

### 2. Replace DeepSeek R1 8B with Qwen3-14B for the Extraction Stage

| Attribute | DeepSeek R1 8B | Qwen3-14B |
|---|---|---|
| Hallucination rate (Vectara, 2025) | ~14.3% | ~4–6% (estimated) |
| Training objective | RL for math/reasoning | General-purpose instruction following |
| Summarisation strength | Weak — "overhelps" | Strong |
| VRAM at Q4 quantisation | ~5 GB | ~9 GB |
| Fits RTX 4080 (16 GB) | Yes | Yes, with KV-cache headroom |
| Context window (RTX 4080) | 49 K tokens | ~25–30 K tokens |

Qwen3-14B fits on the existing RTX 4080 with room for meaningful KV cache. It is the highest-quality local option for the extraction stage given the current hardware constraint.

**Note on model availability (Postgres decision facts, 2026-03-08):** A March session subsequently swapped to Qwen3-8B-AWQ for higher context capacity at a cost of some quality. ADR-012 proposes Qwen3-14B as the extraction model; the March swap to 8B-AWQ is a post-hoc pragmatic adjustment that can be revisited once the E2A architecture is validated.

### 3. Implement Extract-then-Abstract (E2A) Two-Stage Pipeline

This is the single most impactful architectural change. Research (arxiv:2410.06520) demonstrates dramatically reduced hallucination when synthesis models work from structured extracted facts rather than raw transcripts. The extraction model cannot hallucinate a fact the synthesis model never receives.

```
Stage 1: EXTRACT (Local — Qwen3-14B on vLLM)
  Input: JSONL session transcript (formatted, chunked where > context window)
  Output: Structured JSON via vLLM guided_json
  Schema: {
    session_date, duration_estimate,
    primary_goals[],
    technical_decisions[{decision, reasoning, files_affected}],
    files_modified[{path, change_type, description}],
    problems_encountered[{problem, solution, resolved}],
    tools_and_services[],
    key_learnings[],
    collaboration_highlights[]
  }
  Temperature: 0.3   (factual; lower for grounding)
  Purpose: Extract only facts present in the transcript

Stage 2: SYNTHESISE (API — Claude Sonnet 4.5 Batch)
  Input: Extracted facts JSON + style prompt + previous journal entry
  Output: Publication-ready markdown journal entry in Ryan's voice
  Temperature: 0.5   (slight variation for natural prose)
  Purpose: Turn verified, structured facts into narrative
```

**Why E2A prevents hallucination**: The synthesis model (Claude) never sees the raw transcript. It works exclusively from the structured JSON produced by the extractor. If a fact is not present in the JSONL transcript the extractor cannot output it; if the extractor does not output it the synthesiser cannot include it. The fabrication vector is closed.

### 4. Add LLM-as-Judge Quality Gate

```
Stage 3: JUDGE (API — Claude Haiku 4.5)
  Input: Draft journal entry + extracted facts JSON
  Scores (0–10 each):
    - factual_grounding: every claim traces to an extracted fact
    - completeness: all significant session events covered
    - hallucination: binary — any invented content detected?
    - tone_and_style: appropriate for blog/LinkedIn audience
    - coherence: logical narrative flow
  Threshold: all scores >= 7 AND hallucination == 0
  Routing:
    PASS  → auto-publish to vault (01-Journal/)
    FAIL  → flag for human review queue OR trigger revision loop
```

LLM-as-judge achieves ~80% agreement with human evaluators at ~98% lower cost than human annotation (RAGAS paper, 2023). The judge stage provides a measurable, repeatable quality signal and creates the audit trail needed to detect future model regressions.

### 5. Adjust Inference Parameters

| Parameter | Current | Proposed |
|---|---|---|
| Temperature (synthesis) | 0.7 | 0.3 (extraction) / 0.5 (synthesis) |
| Model | DeepSeek R1 8B | Qwen3-14B local + Claude Sonnet 4.5 API |
| Max tokens (extraction) | 3 000 | 1 500 (extraction should be concise) |
| Max tokens (synthesis) | 3 000 | 2 048 |

The temperature reductions are justified by ADR-011's finding that `detailed` preset (0.6) outperformed `default` (0.7) on structure and by the general principle — confirmed in multiple summarisation studies — that 0.3–0.5 is optimal for factually-grounded content.

### 6. Retain OTEL Boundary Detector as Trigger Only

The boundary detector's session-end logic (time gaps, milestone detection, topic changes) is preserved unchanged. It writes to `journal.triggers`; the session parser consumes that topic to locate and process JSONL files.

## Consequences

### Positive

- Journal entries grounded in actual session content — the `deploy_watchdog.sh` fabrication class is closed by architecture, not prompt.
- Publication-quality output enables the downstream blog → LinkedIn pipeline; currently the entire content automation vision is blocked on this.
- LLM-as-judge produces measurable quality metrics (factual grounding, hallucination score) that can be tracked over time and alert on model regressions.
- E2A is composable: the same extracted JSON can feed blog synthesis, weekly summaries, LinkedIn post generation, and semantic search indexing as separate downstream consumers.
- Qwen3-14B fits the existing RTX 4080; no new GPU hardware required.

### Negative

- API dependency introduced for synthesis and judge stages: requires network connectivity and Anthropic uptime, and the Batch API can introduce up to 24-hour latency (mitigated by using the standard API for same-day publishing).
- Pipeline complexity increases from 1 stage to 3; operational failure surface widens correspondingly.
- Qwen3-14B requires a ~9 GB model download and slightly reduces the available KV cache window compared with the 8B model.
- Claude API creates a recurring monetary cost (~$0.78/month) where the current pipeline is $0, though this is immaterial at current volume.

### Risks

- **Qwen3-14B extraction quality unvalidated on production transcripts.** Mitigation: A/B test extraction output on 5 representative sessions before full rollout.
- **Batch API latency.** Mitigation: keep standard-API path as a `FAST_PUBLISH=true` environment flag for time-sensitive entries.
- **Claude API pricing changes.** Currently negligible; would need a >100× increase to affect decision economics.
- **Context window insufficiency for very long sessions.** Median session is ~50 K tokens; Qwen3-14B context is ~25–30 K at Q4 on the RTX 4080. Mitigation: chunk-and-extract with rolling summary across chunks, merge before synthesis.

## Alternatives Considered

### Alternative A: Keep DeepSeek R1 8B, Improve Prompts

This was the approach taken in VW-30 (v8 prompt redesign, grounding blocks). It achieved correct voice and format but did not reduce hallucination. The root cause is model architecture, not prompt design. A model with a ~14% hallucination rate trained for reasoning-task RL cannot be prompted into grounded summarisation behaviour. **Rejected.**

### Alternative B: Upgrade to a 70B+ Local Model

Qwen 2.5 72B or Llama 3.3 70B would reduce the hallucination rate further and eliminate the API dependency. Both require ~40 GB VRAM at Q4 quantisation; the RTX 4080 has 16 GB. Multi-GPU or a GPU upgrade would be required. ADR-011 already established the hardware constraint as a binding decision variable. **Deferred** — revisit if Qwen3-14B + API synthesis proves insufficient after six months of production data, or if hardware changes.

### Alternative C: Full API Pipeline (All Stages via Claude)

Replacing vLLM entirely with Claude API calls (extraction, synthesis, judge all via Anthropic) simplifies infrastructure — no local model management, no VRAM arithmetic. Quality ceiling is ~9/10 vs ~8/10 for hybrid. Monthly cost rises to ~$2.41. The hybrid approach saves ~$1.63/month and keeps the GPU useful. At current session volume this is not a significant cost difference in absolute terms, but the local extraction stage also provides auditability (the JSON is stored) and eliminates one API round-trip per session. **Viable but not chosen at this volume.** If session volume grows >10×, re-evaluate.

### Alternative D: Fine-Tune a Small Model on Ryan's Writing Style

Interesting long-term option. Requires a curated dataset of high-quality journal entries (which the hybrid E2A pipeline will produce). Cannot be pursued before the pipeline produces good output. **Deferred** — revisit after 3–6 months of production E2A entries as training data.

## Implementation Plan

### Phase 1 — Model Swap (1–2 hours)
- Pull Qwen3-14B to vLLM serving pod; update `VLLM_MODEL` environment variable via GitOps ConfigMap.
- Lower synthesis temperature to 0.3 in vLLM call parameters.
- Validate extraction quality on 3 representative JSONL transcripts before advancing.

### Phase 2 — E2A Pipeline (4–6 hours)
- Add extraction stage to `scripts/journal/streaming/synthesiser.py` (or equivalent): guided_json output schema as above.
- Add Anthropic SDK client for synthesis stage; wire extraction JSON → synthesis prompt.
- Wire extraction → synthesis → output flow.
- Test on 5 real JSONL sessions with manual cross-reference verification.

### Phase 3 — Quality Gate (2–3 hours)
- Add Haiku judge stage consuming synthesis output + extracted JSON.
- Define scoring rubric and pass/fail threshold (all >= 7, hallucination == 0).
- Add auto-publish vs human-review routing in Prefect flow.
- Instrument scores as Prometheus metrics for Grafana trend visibility.

### Phase 4 — OTEL Deprecation (30 minutes)
- Set `MIN_OTEL_EVENTS = 999` in synthesiser config.
- Add `WARN: OTEL synthesis path disabled (ADR-012)` log message.
- Verify OTEL boundary detector still writes to `journal.triggers` and the JSONL path picks up correctly.

### Phase 5 — Blog and Social Pipeline (4–6 hours)
- Weekly aggregation Prefect flow consuming daily E2A JSON extracts.
- Blog post synthesis via Claude Sonnet.
- LinkedIn post formatting.
