# ADR-012: Journal Pipeline Quality Overhaul — Model Selection & E2A Architecture

## Status

Proposed

## Context

The journal streaming pipeline is the first stage of Rootweaver's automated content funnel: Claude Code session → structured journal entry → blog episode → social post. If the journal stage fabricates content, every downstream artifact inherits the error. The current pipeline has three compounding failure modes that make its output unusable without heavy manual review.

### Problem 1: OTEL Event Sparsity

The pipeline's primary input path ingests OpenTelemetry spans emitted by Claude Code sessions. In practice, OTEL instrumentation captures only ~2 out of every 10 meaningful session events. The spans that do arrive carry metadata (tool name, duration, status code) but lack the content payload — the actual user message, assistant reasoning, and tool results that constitute the session's substance. The journal synthesizer receives a skeleton of timestamps and function names, then must infer what happened between them.

This sparsity was confirmed when the OTEL synthesis path was disabled (memory fact #2212, 2026-05-04): "The OTEL path was producing fabricated outputs due to insufficient grounding, leading to hallucinations." The subsequent decision was to "use JSONL transcripts for replay instead of the OTEL stream" (memory fact #2213) because "the OTEL stream lacks content payload, making it impossible to reprocess hallucinated entries accurately."

### Problem 2: DeepSeek R1 8B Hallucination Rate

The synthesizer uses DeepSeek-R1-Qwen3-8B-AWQ (switched from Qwen3.5-9B-AWQ per memory fact #3092, 2026-03-17) running on vLLM with GPU exclusivity. While DeepSeek-R1's chain-of-thought reasoning was chosen for "better performance for reasoning tasks," the 8B parameter count is insufficient for faithful long-form generation from sparse inputs. Measured hallucination rate is 14.3% — roughly one in seven journal entries contains fabricated narrative elements (tools invoked that were not, decisions attributed that were not made, timelines conflated across sessions).

This mirrors the platform's earlier experience with LLM-based NER tagging, where the hallucination rate reached 75% before the decision to replace it with spaCy + EntityRuler (memory fact #3125, 2026-03-19): "This change significantly reduced the hallucination rate from 75% to 0%, improving tagging accuracy." The NER precedent demonstrates that for extraction tasks on the Rootweaver platform, deterministic or tightly constrained approaches outperform unconstrained small-LLM generation.

A further data point: even after switching to Qwen3:8B with num_ctx=16384 for journal outputs, the model showed only a "62% increase in specificity" (memory fact #2196, 2026-05-05) — an improvement, but not a categorical fix. The problem is architectural, not just parametric.

### Problem 3: Single-Shot Generation at Temperature 0.7

The current pipeline makes a single LLM call per session: the full OTEL event stream (or JSONL transcript) is passed as context, and the model generates a complete journal entry in one pass. Temperature is set to 0.7, which encourages lexical diversity but also enables confident fabrication — the model fills gaps in sparse input with plausible-sounding but invented content. There is no verification step, no extraction pass to anchor claims to source material, and no structural constraint on what the model can assert.

The fabricated-narrative problem (memory fact #3854) was flagged but remains unresolved: "Fabricated narrative elements in the initial draft" — the interim workaround was to replace fabricated content with TODO placeholders, which defeats the purpose of automation.

### The Downstream Cost

The session → journal → blog → social funnel means every fabricated journal entry becomes a potential fabricated blog claim. Memory feedback entries document this risk from the blog side: concrete claims must be verified against live data before prose (feedback_verify_concrete_claims_before_prose.md), and a single draft audit caught "6 fabricated specifics in one draft" including wrong dates, inflated numbers, and compressed scope. The journal pipeline is the origin point for this class of error.

## Decision

We will overhaul the journal pipeline with two coordinated changes:

### 1. Model Selection: Replace DeepSeek-R1-Qwen3-8B with Qwen3-14B-AWQ for Extraction, Retain 8B for Abstraction

The pipeline will use a **two-model tiered approach** matched to task complexity:

- **Extraction pass (Qwen3-14B-AWQ via vLLM)**: The larger model handles the high-fidelity task — reading the full JSONL transcript and producing a structured JSON intermediate representation containing only claims directly attributable to source material. The 14B model was previously swapped out in favour of 8B for context capacity reasons (memory facts #4385, #3388, #4458, 2026-03-08), but the journal pipeline's non-functional state at that time meant the swap was made to serve architecture reviews, not journal quality. With the pipeline now being rebuilt, the 14B model's superior faithfulness justifies the VRAM cost for the extraction pass.

- **Abstraction pass (DeepSeek-R1-Qwen3-8B or Qwen3-8B via vLLM)**: The smaller model takes the verified structured JSON and generates the narrative journal entry. Because its input is already factually constrained — every claim in the intermediate JSON is anchored to a source line — the 8B model's tendency toward fabrication is bounded. It can only arrange and paraphrase claims that exist, not invent new ones. Temperature can remain at 0.5-0.7 for this pass since diversity in phrasing is acceptable when the factual substrate is locked.

**VRAM implications**: The 14B model requires ~15GB VRAM, matching the current vLLM allocation. The extraction and abstraction passes run sequentially, not concurrently, so only one model needs to be loaded at a time. vLLM's model-swapping capability (or a shared endpoint with two model aliases) handles this without additional GPU allocation. Ollama remains CPU-only per platform guardrails.

### 2. Architecture: Extract-then-Abstract (E2A) Pipeline Replacing Single-Shot Generation

The single-shot generation pattern is replaced with a three-stage pipeline:

**Stage 1 — Source Anchoring (deterministic)**
Parse the JSONL transcript into a structured timeline: each round (user message + assistant response + tool calls) becomes a timestamped event with verbatim content hashes. No LLM involved. This produces the source-of-truth document against which all downstream claims are validated.

**Stage 2 — Structured Extraction (Qwen3-14B, temperature 0.0)**
The 14B model receives the anchored timeline and a strict JSON schema. It extracts:
- `events[]`: What happened, with source line references (round index + message index)
- `decisions[]`: What was decided, with the reasoning quoted from the transcript
- `artifacts[]`: What was produced (files, commits, Jira transitions), with paths/IDs
- `problems[]`: What went wrong, with error messages verbatim
- `metrics{}`: Quantitative claims (durations, counts, sizes) with exact source values

Temperature 0.0 eliminates sampling randomness. The JSON schema acts as a structural constraint — the model cannot emit free-form narrative, only populate defined fields with attributed content. Each claim carries a `source_ref` pointing back to the anchored timeline.

**Stage 3 — Narrative Abstraction (8B model, temperature 0.5)**
The 8B model receives the structured extraction (not the raw transcript) and a journal template. It generates natural-language prose, but every factual claim must trace to an entry in the extraction JSON. A post-generation validation step checks that no claims appear in the narrative that lack a `source_ref` in the extraction — any unattributed claim is flagged or stripped.

**Input path change**: The pipeline's primary input switches from OTEL spans to JSONL session transcripts. This aligns with the prior decision (memory fact #2213) to use JSONL for replay. OTEL spans remain useful for latency/timing metadata but are no longer the content source.

**Temperature reduction**: Extraction at 0.0, abstraction at 0.5 (down from 0.7). The 0.7 setting was appropriate for creative generation but inappropriate for a pipeline whose primary quality metric is faithfulness.

### Kafka Integration

The existing Kafka consumer (`journal-session-parser`) retains its claim-check pattern for large payloads and manual offset commits. The E2A stages run within the consumer's processing loop — Stage 1 is pure Python (no LLM), Stages 2 and 3 are sequential vLLM calls. Total processing time per session is expected to increase from ~30s (single-shot) to ~90-120s (three-stage), which is acceptable for a non-real-time pipeline.

## Consequences

### Positive

- **Hallucination rate reduction**: The extraction pass at temperature 0.0 with JSON schema constraints eliminates the primary fabrication vector. Based on the NER precedent (75% to 0% hallucination rate when switching from unconstrained LLM to structured extraction), we expect the journal hallucination rate to drop from 14.3% to below 2%.

- **Auditability**: Every claim in the final journal entry traces through the extraction JSON back to a specific round in the JSONL transcript. When a claim is disputed (as happened with 6 fabricated specifics in the S3E6 blog draft), the audit trail is mechanical — check the `source_ref`, find the transcript line, confirm or reject.

- **Downstream quality**: The blog and social post generators receive journal entries with verified factual substrates. The manual verification step documented in feedback_verify_concrete_claims_before_prose.md becomes a spot-check rather than a full audit.

- **Model-appropriate tasking**: The 14B model handles the hard problem (faithful extraction from long, noisy input) while the 8B model handles the easier problem (narrative generation from clean, structured input). This matches each model's capability envelope rather than asking a small model to do both.

### Negative

- **Increased latency**: Three-stage processing (~90-120s) vs single-shot (~30s). Acceptable for daily journal generation but would block real-time use cases.

- **VRAM scheduling complexity**: If the extraction pass uses 14B and the abstraction pass uses 8B, model swapping on vLLM adds ~10-15s per transition. Alternatively, both passes could use the same model (14B for both), trading abstraction speed for simpler scheduling.

- **Intermediate format maintenance**: The extraction JSON schema becomes a contract between stages. Schema changes require coordinating the extraction prompt, the validation logic, and the abstraction prompt.

- **Reduced journal "voice"**: Temperature 0.5 and factual anchoring will produce more clinical, less narrative journal entries. This is a deliberate trade — the current "voice" is partly hallucinated colour, which is worse than clinical accuracy.

## Alternatives Considered

### Alternative A: Upgrade to a Larger Single Model (e.g., Qwen3-32B or 70B-AWQ)

A larger model in single-shot mode would have better faithfulness by default, reducing hallucination without architectural change.

**Rejected because**: (1) VRAM constraint — the RTX 4080 has 16GB, and 32B+ models require multi-GPU or heavy quantization that degrades quality. (2) Does not address the structural problem — even large models hallucinate when given sparse input with no verification step. The platform's NER experience showed that architectural constraints (spaCy rules) beat model scaling (larger LLM) for extraction fidelity. (3) Latency for 32B+ on single GPU would exceed 120s per session, negating the speed advantage of single-shot.

### Alternative B: Fine-Tune DeepSeek-R1-8B on Verified Journal Examples

Collect ~200 verified journal entries, fine-tune the 8B model to produce faithful outputs for this specific domain.

**Rejected because**: (1) The platform does not yet have 200 verified journal entries — the pipeline's quality problems mean the existing corpus is unreliable training data. (2) Fine-tuning requires ongoing maintenance as the journal format evolves. (3) The fundamental input sparsity problem (OTEL events lacking content) remains — a fine-tuned model still cannot extract information that is not in its input. (4) Fine-tuning on vLLM with the current single-GPU setup would require taking the inference endpoint offline, affecting all platform consumers.

### Alternative C: Abandon Local LLM, Use Claude API for Journal Generation

Route journal generation through the Claude API (Sonnet or Haiku) instead of local vLLM, leveraging superior faithfulness of larger frontier models.

**Rejected because**: (1) Cost — at ~$3/MTok input for Sonnet, processing 10-50 sessions/day with 50-200K tokens each would cost $15-300/month, which is significant for a personal platform. (2) Privacy — session transcripts contain tool outputs, file contents, and system paths that should not leave the local network. (3) Dependency — the platform's design principle is local-first inference (ADR-022 Local LLM Hybrid Routing), with cloud APIs as fallback, not primary path. (4) Latency and availability — Claude API rate limits and outages would make the journal pipeline non-deterministic.
