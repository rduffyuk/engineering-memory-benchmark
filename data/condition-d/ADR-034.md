# ADR-034: Data Source Connector Architecture

## Status

Proposed

## Context

Rootweaver's knowledge base is currently limited to a single data source: the Obsidian vault. The vault indexer (`neural-vault/vault_indexer.py`) watches for file changes via inotify, produces Kafka events, and consumers chunk, embed, and upsert documents into Qdrant (148K+ chunks from ~4K vault files, 1024-dimensional Qwen3-Embedding-0.6B vectors). This pipeline is mature and battle-tested, but it is hard-wired to local markdown files.

Meanwhile, significant organisational knowledge lives in systems that Rootweaver cannot search:

- **Jira** (VW project): 400+ issues with decisions, investigations, and context that only surface through MCP tool calls (`mcp__atlassian__jira_search`), never indexed into Qdrant.
- **Confluence** (VW space): Architecture pages, ADRs, runbooks, and design documents accessible only via `mcp__atlassian__confluence_search`, not available to the RAG agents.
- **GitLab**: Commit history, merge requests, and code review discussions from `gitlab.com:ryanduffy.uk/rootweaver-platform`.
- **Slack, Discord, Gmail, IMAP**: Communication channels containing decisions, troubleshooting threads, and tribal knowledge.

This gap blocks cross-source queries entirely. When a user asks "what decisions led to the Kafka consumer architecture?", the system can search vault documents but cannot correlate with the Jira issues (VW-232, VW-288), Confluence design pages, or GitLab commits that contain the actual decision trail. The typed memory system (`recall_decisions`, `find_problems`) captures some of this through session extraction, but it is incomplete -- it only records facts from Claude Code sessions, not from direct human activity in those tools.

Industry context reinforces the urgency. Glean indexes 100+ data sources out of the box. Onyx (formerly Danswer), an open-source enterprise search platform, supports 40+ connectors through a well-defined `BaseConnector` abstraction with polling-based incremental sync and normalised `Document` output. Rootweaver already carries an `onyx_shim` layer (vendored subset of Onyx's connector interfaces) and has partially implemented connectors for Jira, Confluence, GitLab, Slack, Discord, Gmail, IMAP, web, and file sources under `packages/rootweaver-connectors/`. However, this work is incomplete: the Jira connector CronJob has failed with `ImportError` due to missing symbols (typed memory fact #2386), the `OnyxBridge.incremental_sync()` method was broken and required a fix to `checkpoint_sync()` (facts #2461, #4714, VW-188), and interface mismatches between connector methods (`load_from_state` vs `load_from_checkpoint`) have caused runtime failures (fact #2481).

The question this ADR addresses is: what architecture should govern how Rootweaver ingests, normalises, and indexes documents from arbitrary external sources into a unified search corpus?

### Evidence from Typed Memory

The following platform decisions and problems, retrieved via `recall_decisions` and `find_problems`, directly inform this ADR:

1. **VW-184** (fact #4775): A Prefect scheduled flow was proposed to integrate Jira and Confluence updates using existing MCP tools. The alternative of "building a full connector framework" was explicitly rejected at that time due to complexity. This ADR revisits that trade-off now that partial connector infrastructure exists.

2. **CronJob over Prefect** (fact #2478): The connector scheduling was switched from Prefect to K8s CronJob because "the existing code had interface mismatches with Prefect, and no Redis was available for state management."

3. **Checkpoint sync fix** (facts #2461, #4714, VW-188): The bridge's `incremental_sync()` was broken; it was replaced with `checkpoint_sync()` to align with the `CheckpointedConnector` interface pattern from Onyx.

4. **Consumer split** (facts #2424, #4682, VW-232): The consumer logic was split into `indexer.py` (chunking + embedding + Qdrant upsert) and `indexer_runner.py` (Kafka consumption loop) to maintain single responsibility.

5. **Deterministic chunk IDs** (fact #4680, VW-232): Qdrant point IDs use `uuid5(NAMESPACE, source/source_id/chunk_index)` rather than `uuid4()` to ensure re-delivery idempotency.

6. **Jira CronJob ImportError** (fact #2386): The `connector-jira-scheduler` CronJob failed due to missing symbols in the shim layer, indicating the onyx_shim vendoring is fragile and under-tested.

7. **JiraConnector interface mismatch** (fact #2481): `JiraConnector` required `project_key` not `jira_project_url`; `bridge.py` called `load_from_state()`/`poll_source()` which needed replacing with `load_from_checkpoint()`.

## Decision

We adopt a **polling-based connector framework** built on the Onyx `BaseConnector` / `CheckpointedConnector` interface contract, with Rootweaver-specific adaptations for Kafka-first event transport, Qdrant-targeted indexing, and K8s CronJob scheduling.

### Architecture Components

**1. Connector Interface (`onyx_shim/`)**

Retain the vendored Onyx shim as the connector contract. Each connector implements one of:

- `CheckpointedConnector[CT]`: The primary interface. Yields `Document | ConnectorFailure | HierarchyNode` from `load_from_checkpoint(start, end, checkpoint)`. Checkpoint state is returned via `StopIteration.value`, enabling resumable pagination. This is the pattern already used by the Jira and Confluence connectors.
- `PollConnector`: Legacy interface for simpler sources. `poll_source(start, end)` returns an iterator of document batches. Used by sources without pagination (e.g., IMAP, file).
- `LoadConnector`: Full-load interface for sources without incremental support (e.g., web crawl).

**2. Normalised Document Model (`models.py`)**

`ConnectorDoc` is the platform's normalised document representation, decoupled from Onyx's internal `Document` model:

```
ConnectorDoc(source, source_id, title, content, url, author,
             created_at, updated_at, metadata)
```

The `OnyxBridge._convert()` method translates Onyx `Document` (with `sections`, `primary_owners`, `metadata`) into `ConnectorDoc`. This translation boundary ensures that downstream consumers (Kafka, indexer, Qdrant) never depend on Onyx internals.

**3. Bridge (`bridge.py` -- `OnyxBridge`)**

The bridge drives any connector to completion and produces `ConnectorDoc` messages to Kafka topic `vault.connector-docs`. Key behaviours:

- `checkpoint_sync(start_ts, end_ts)`: Loops `load_from_checkpoint()` until `has_more == False`, producing each `Document` to Kafka, counting failures, tracking `max_updated_at` for watermark advancement.
- Kafka flush in `finally` block ensures buffered messages are not lost on mid-loop exceptions.
- Returns `(produced, failures, max_updated_at)` tuple for caller to decide watermark update.

**4. Factory (`factory.py`)**

A static `CONNECTOR_REGISTRY` maps source names to `ConnectorMapping(module_path, class_name, config, credentials_env)`. `create_connector(source)` dynamically imports the connector class, instantiates with config, and calls `load_credentials()` from the environment variable's JSON value. This allows adding new sources by adding a registry entry and a credentials secret -- no code changes to the scheduling or indexing layers.

**5. Indexer (`indexer.py` -- `ConnectorIndexer`)**

Consumes `ConnectorDoc` from Kafka, chunks text (750 tokens, 75 overlap), embeds via the platform embedder, and upserts to Qdrant through `QdrantAdapter` (ensuring v1+v2 dual-write per VW-235). Deterministic `uuid5` point IDs guarantee idempotent re-indexing.

**6. Scheduling (`indexer_runner.py` + K8s CronJob)**

Each source runs as a K8s CronJob (not Prefect -- decision per fact #2478). The runner:

1. Reads the Postgres watermark for the source (last successful `max_updated_at`).
2. Calls `OnyxBridge.checkpoint_sync(watermark, now)`.
3. On success, advances the Postgres watermark.
4. On failure, leaves the watermark unchanged; next CronJob run retries the same window.

### Data Flow

```
[External Source API]
        |
        v
  Connector (Jira/Confluence/GitLab/...)
        |  yields Document / ConnectorFailure
        v
  OnyxBridge.checkpoint_sync()
        |  converts to ConnectorDoc
        v
  Kafka topic: vault.connector-docs
        |
        v
  ConnectorIndexer (Kafka consumer)
        |  chunks, embeds, deterministic IDs
        v
  QdrantAdapter.add_documents()
        |  dual-write v1 + v2
        v
  Qdrant collection: rootweaver_vault
        |
        v
  RAG agents (FastSearch / DeepResearch / Oracle)
```

### Metadata Contract

Every document indexed from a connector carries:

| Field | Purpose |
|-------|---------|
| `source` | Source identifier (e.g., "jira", "confluence") |
| `source_id` | Unique ID within the source |
| `source_url` | Link back to the original item |
| `type` | Always "connector" (distinguishes from vault docs) |
| `date_created` | Temporal filtering compatibility |
| `timestamp` | Unix epoch for range queries |
| `file_path` | `{source}/{source_id}` for Qdrant dedup |
| `meta_*` | Source-specific metadata (prefixed to avoid collisions) |

This aligns with the existing vault indexer's metadata schema, enabling unified temporal and source-filtered search across all data sources.

## Consequences

### Positive

1. **Cross-source search unlocked**: Users can ask "what decisions led to X?" and get results spanning vault documents, Jira issues, Confluence pages, and GitLab commits in a single query.

2. **Incremental by default**: The checkpoint/watermark pattern ensures each sync only fetches documents updated since the last successful run, keeping API call volume proportional to change rate rather than corpus size.

3. **Idempotent indexing**: Deterministic `uuid5` point IDs mean re-delivery or re-runs produce upserts, not duplicates. This is critical for CronJob-based scheduling where the same time window may be retried.

4. **Source isolation**: Each connector is a self-contained module. A Jira connector failure does not affect Confluence indexing. The factory pattern means adding a new source requires only a new `sources/<name>/connector.py` and a registry entry.

5. **Existing infrastructure reuse**: The architecture reuses the Kafka pipeline, Qdrant, `QdrantAdapter` dual-write, and the RAG agent stack without modification. No new databases, message brokers, or API gateways are introduced.

6. **Onyx ecosystem leverage**: By maintaining compatibility with the Onyx `BaseConnector` interface, Rootweaver can adopt upstream connector implementations for new sources (40+ available) with minimal adaptation -- primarily adding a shim for any missing Onyx utility dependencies.

### Negative

1. **Onyx shim maintenance burden**: The vendored `onyx_shim/` is a partial copy of Onyx's internal models and interfaces. Upstream changes (e.g., new fields on `Document`, changes to `CheckpointedConnector` signature) require manual propagation. The ImportError incident (fact #2386) demonstrates this fragility.

2. **No real-time ingestion**: CronJob-based polling introduces latency between a source update and searchability. For a 5-minute CronJob interval, a Jira issue created at T+0 is searchable at T+5 at best. Event-driven alternatives (webhooks) are not supported in this architecture.

3. **Credential management complexity**: Each source requires a JSON credentials blob in a K8s Secret, referenced by environment variable. Managing OAuth token refresh (e.g., Confluence, Gmail) across CronJob invocations requires external token management or persistent storage.

4. **Single collection mixing**: Connector documents are indexed into the same `rootweaver_vault` Qdrant collection as vault documents. While the `type: "connector"` and `source` metadata fields enable filtering, this increases collection size and may affect search relevance if connector documents have significantly different content characteristics (e.g., terse Jira issue titles vs. long vault articles).

5. **Testing gap**: The connector framework currently lacks integration tests. The Jira CronJob ImportError and interface mismatches were caught in production, not in CI. Each new connector adds surface area that must be tested against live or mocked API responses.

## Alternatives Considered

### Alternative 1: MCP-Only Ingestion via Prefect Flows

**Description**: Use Prefect scheduled flows to call existing MCP tools (`mcp__atlassian__jira_search`, `mcp__atlassian__confluence_search`) and write results directly to Qdrant, bypassing the connector framework entirely.

**This was the original VW-184 decision** (fact #4775): "Proposed a Prefect scheduled flow to integrate Jira and Confluence updates... Alternatives rejected: Building a full connector framework was considered but rejected due to higher complexity and longer implementation time."

**Why reconsidered**: The MCP-only approach has fundamental limitations:
- MCP tools return search results, not full document content. `jira_search` returns summaries; the full issue (description, comments, attachments) requires additional `jira_get_issue` calls per result.
- No pagination model: MCP tools return a fixed result set, not a scrollable cursor. Ingesting all 400+ VW issues requires crafting JQL queries with offset management outside the tool contract.
- No normalised document model: each MCP tool returns a different response shape, requiring per-source transformation logic in the Prefect flow.
- No incremental sync: without watermarks, every flow run must re-query and diff against existing Qdrant content to avoid duplicates.
- Prefect was already rejected for connector scheduling due to interface mismatches and missing Redis (fact #2478).

**Trade-off**: Lower initial complexity but fundamentally limited scalability. Acceptable for 2-3 sources; unsustainable for the target of 9+ sources.

### Alternative 2: Direct API Polling Without Onyx

**Description**: Build Rootweaver-native connectors from scratch, without the Onyx shim layer. Each connector would implement a simpler `RootWeaverConnector` interface:

```python
class RootWeaverConnector(ABC):
    def poll(self, since: datetime) -> Iterator[ConnectorDoc]:
        ...
    def load_credentials(self, creds: dict) -> None:
        ...
```

**Advantages**:
- Eliminates the onyx_shim maintenance burden entirely.
- Simpler interface (no `CheckpointedConnector`, `HierarchyNode`, `SlimDocument`, `ConnectorFailure` types).
- Direct `ConnectorDoc` output -- no conversion step.

**Why rejected**:
- Reimplements pagination, rate limiting, error handling, and API-specific quirks already solved in Onyx connectors. The Jira connector alone handles JQL pagination, attachment downloading, comment threading, and Atlassian Cloud rate limiting.
- The 9 existing connector implementations (`sources/jira/`, `sources/confluence/`, etc.) already use the Onyx interface. Migrating them to a new interface is more work than maintaining the shim.
- Forecloses the option of adopting additional Onyx connectors (40+ available) as the source list grows.

**Trade-off**: Cleaner abstraction at the cost of significant reimplementation effort and reduced future connector velocity.

### Alternative 3: Embed Onyx as a Sidecar Service

**Description**: Run Onyx (Danswer) as a separate service alongside Rootweaver. Onyx handles all connector management, document extraction, and indexing into its own Vespa/Postgres backend. Rootweaver queries Onyx's search API for non-vault sources.

**Advantages**:
- Zero connector code in Rootweaver.
- Onyx handles credential management, scheduling, error recovery, and UI for connector configuration.
- Automatic access to all 40+ Onyx connectors without any integration work.

**Why rejected**:
- Onyx requires PostgreSQL, Redis, Vespa, and a model server -- substantial infrastructure overhead for a single-node K3s cluster already running Qdrant, Kafka, and vLLM.
- Search results would come from two different backends (Qdrant for vault, Vespa for connectors), requiring a federation layer to merge and re-rank results.
- Embedding model mismatch: Onyx uses its own embedding model; Rootweaver uses Qwen3-Embedding-0.6B at 1024 dimensions. Cross-backend semantic similarity scores would not be comparable.
- Operational complexity of maintaining a second search platform with its own upgrade cycle, monitoring, and failure modes.

**Trade-off**: Maximum connector coverage at the cost of infrastructure doubling and search federation complexity.
