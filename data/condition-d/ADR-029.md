# ADR-029: KEDA-Based GPU Swap Controller for Embed/Inference Coexistence

## Status

Accepted

## Context

### The Single-GPU Constraint

The Rootweaver platform runs on a single NVIDIA RTX 4080 with 16 GB of VRAM, hosted on the desktop node (100.113.76.79) inside a K3s cluster. Three workloads compete for that GPU:

| Workload | VRAM Requirement | Runtime Pattern |
|----------|-----------------|-----------------|
| **vLLM** (Qwen3.5-9B-AWQ inference) | ~15.6 GB steady-state | Always-on; serves every RAG query via the MCP bridge |
| **TEI** (Qwen3-Embedding-4B) | ~8 GB transient | Burst; fires during vault-delta-indexer runs, connector ingestion, and Jira sync |
| **Code-graph embedder** (PDG / FalkorDB) | ~2-4 GB transient | Periodic; rebuilds the `code_knowledge` graph on code changes |

With vLLM consuming nearly all 16 GB, none of these workloads can coexist on the GPU simultaneously. This is not a theoretical concern — typed memory records four separate incidents where the constraint caused real outages:

1. **GPU contention caused vllm-0 to be Pending during initial checks** (fact 2131, session d1b33124). Resolution required manually pausing the TEI ScaledObject to free GPU resources.
2. **vLLM pod failed to become ready due to GPU contention** (fact 2183, session b6f9234f). TEI embedding had to be scaled down manually.
3. **vLLM crashed multiple times during bulk replay** (fact 2215, session ec1691bb). GPU memory contention from concurrent services was the root cause.
4. **Pending state of vLLM pod due to GPU scheduling conflict with TEI** (fact 3095, session 78d505fa). Resolved by deleting the TEI pod.

Additionally, the embedding workload has its own history of memory problems:

- **Retriever pod OOMKilled** loading Qwen3-Embedding-4B (fact 4052) — required switching to BF16 precision and raising memory limits.
- **TEI model dimension mismatch** between Qwen3-Embedding-0.6B and Qdrant's 2560d collection (fact 3944) — forced the move to the larger 4B model, which in turn increased VRAM demand from ~2 GB to ~8 GB.
- **MacBook embedding agent consuming 24 GB** of unified memory via MPS (VW-107, CLAUDE.md memory) — eventually decommissioned entirely (VW-203), consolidating all embedding onto the desktop GPU.

### Why Manual Swap Is Untenable

Before this ADR, GPU contention was resolved by an operator (Ryan or Claude Code) manually scaling TEI to zero, waiting for vLLM to reclaim the GPU, and reversing the process when embedding was needed. This worked when embedding was a nightly batch job. It does not work now because:

1. **Jira connector ingestion creates unpredictable bursts.** When a Jira sync completes, the vault-delta-indexer fires immediately, requiring TEI for embedding. These bursts can arrive at any time during the workday.
2. **The code-graph embedder runs on git push events.** Every merge to main triggers a PDG rebuild that needs GPU-accelerated embedding.
3. **vLLM prefix caching** (fact 2810, enabled in GitOps YAML to reduce TTFT by ~30%) means vLLM startup is expensive — cold-starting it after every embedding burst wastes 2-3 minutes of cache warm-up.

The operator overhead is ~5-10 minutes per swap, and with Jira connector bursts arriving 3-5 times per day, that represents 15-50 minutes of daily toil on a one-person platform.

### HAMi Ruled Out

HAMi (Heterogeneous AI computing Management Interface) was evaluated as a GPU-sharing solution that could partition VRAM between vLLM and TEI. It was rejected for two reasons:

1. **vLLM requires near-exclusive VRAM.** The Qwen3.5-9B-AWQ model with KV cache consumes ~15.6 GB. Even a 50/50 split (8 GB each) would prevent vLLM from loading the model at all.
2. **HAMi's overhead and complexity.** It introduces a device-plugin replacement, custom scheduler, and webhook that would be the most complex single component in the cluster — disproportionate for a single-GPU node.

No typed memory records exist for HAMi decisions, confirming it was never trialled in practice.

## Decision

Implement a **KEDA-based GPU swap controller** that orchestrates time-multiplexed access to the single GPU between vLLM (inference) and TEI (embedding) workloads using Kubernetes-native scaling primitives.

### Architecture

```
Kafka topic (vault-delta / code-change)
        │
        ▼
  KEDA ScaledObject (TEI)
  trigger: kafka lag > 0
        │
        ▼
  Pre-scale hook (Job):
    1. Scale vLLM replicas → 0 (paused-replicas annotation)
    2. Wait for GPU release (poll nvidia-smi or device-plugin)
    3. Allow TEI pod to schedule
        │
        ▼
  TEI processes embedding queue
        │
        ▼
  Post-scale hook (on scale-to-zero):
    1. TEI replicas → 0 (KEDA cooldown)
    2. Restore vLLM replicas → 1
    3. Wait for vLLM ready + prefix cache warm
```

### Key Design Choices

**1. KEDA `paused-replicas` annotation for vLLM scale-down.**

Typed memory (fact 2384, session 8111ef83) records the decision to use `paused-replicas` annotation instead of `fallback.replicas` for KEDA scaling, because the fallback mechanism does not force-scale when the scaler reports zero lag. The same principle applies here: vLLM's ScaledObject uses `paused-replicas: "0"` to guarantee scale-down, and the annotation is removed to restore it.

**2. Kafka lag as the embedding trigger.**

KEDA monitors Kafka consumer group lag on the `vault-delta` and `code-change` topics. When lag > 0, the swap sequence initiates. This aligns with the existing architecture where the vault-delta-indexer and code-graph embedder are Kafka consumers. The trigger is inherently bursty and self-draining — once the queue is empty, KEDA's cooldown period (default 300s) triggers the reverse swap.

**3. vLLM prefix cache warm-up budget.**

vLLM prefix caching (fact 2810) reduces TTFT by ~30% but requires warm-up after a cold start. The swap controller allocates a 60-second warm-up window after restoring vLLM to replicas=1, during which the MCP bridge returns a 503 with `Retry-After: 60` rather than routing queries to a cold model. This prevents degraded-quality responses during the transition.

**4. Priority: vLLM is the primary tenant.**

vLLM serves interactive RAG queries (P0 latency-sensitive). TEI serves batch embedding (P1 throughput-sensitive). If both need the GPU simultaneously, vLLM wins. The swap controller enforces this by:
- TEI can only acquire the GPU when KEDA triggers (queue backlog exists).
- TEI must release within `maxReplicaCount * cooldownPeriod` (bounded lease).
- An emergency pre-emption path allows vLLM to reclaim the GPU if TEI exceeds its lease (kill pod, KEDA re-queues).

**5. Code-graph embedder shares TEI's window.**

The code-graph embedder (PDG rebuild) is scheduled as a lower-priority Job that runs during TEI's GPU window. Since both need embedding (not inference), they share the same swap slot. The embedder uses TEI's HTTP API rather than loading its own model, eliminating a third contention party.

### GitOps Manifests

The swap controller is implemented as:

- `infrastructure/keda/tei-scaledobject.yaml` — KEDA ScaledObject with Kafka trigger
- `infrastructure/keda/vllm-scaledobject.yaml` — KEDA ScaledObject with `paused-replicas` annotation support
- `infrastructure/keda/gpu-swap-controller.yaml` — CronJob/Job template for pre/post-scale hooks
- `infrastructure/keda/gpu-swap-rbac.yaml` — ServiceAccount with permissions to patch Deployments and ScaledObjects

All changes go through GitOps (FluxCD reconciliation), consistent with platform guardrails.

## Consequences

### Positive

1. **Eliminates manual GPU swap toil.** The 15-50 minutes/day of operator intervention for GPU arbitration becomes zero.
2. **Enables Jira connector bursts without human intervention.** Embedding can fire automatically when Kafka lag appears, process the queue, and return the GPU to vLLM — all within minutes.
3. **Preserves vLLM prefix cache investment.** The bounded swap window (typically 2-5 minutes for a delta-indexer burst) minimizes cache invalidation. Cold restarts drop from several per day to the swap frequency.
4. **Uses existing KEDA infrastructure.** KEDA is already deployed for other ScaledObjects in the cluster. No new operators or CRDs required.
5. **Kubernetes-native.** The controller uses standard K8s primitives (Deployments, Jobs, annotations, RBAC) — no custom controllers or sidecars.

### Negative

1. **vLLM unavailability during swap windows.** Interactive RAG queries receive 503s for the duration of each swap (typically 2-5 minutes). Acceptable for a single-user platform; would need a queue/retry layer for multi-user.
2. **Increased vLLM cold-start frequency.** Each swap cycle requires vLLM to reload the model and warm prefix cache (~60-90 seconds). Mitigated by the 300s cooldown that batches multiple embedding messages into a single swap.
3. **Complexity in failure modes.** If the post-scale hook fails to restore vLLM, the platform has no inference capability until manual intervention. Mitigated by a watchdog CronJob that checks vLLM health every 5 minutes and force-restores if needed.
4. **KEDA cooldown tuning required.** Too short a cooldown (< 120s) causes thrashing between vLLM and TEI. Too long (> 600s) delays vLLM restoration. The initial 300s value is empirically chosen based on typical delta-indexer batch sizes (50-200 documents, ~2 minutes at TEI throughput).

### Neutral

- **Ollama remains CPU-only.** This decision does not affect Ollama's CPU-only configuration (`/etc/systemd/system/ollama.service.d/cpu-only.conf`). Ollama continues to serve auxiliary inference on system RAM.
- **TEI model choice unchanged.** The Qwen3-Embedding-4B model (fact 3942, required for 2560d Qdrant compatibility) remains the embedding model. The swap controller is model-agnostic.

## Alternatives Considered

### Alternative 1: Dedicated Embedding GPU

**Description:** Purchase a second GPU (e.g., RTX 3060 12 GB or RTX 4060 Ti 16 GB) dedicated to TEI and the code-graph embedder, leaving the RTX 4080 exclusively for vLLM.

**Pros:**
- Eliminates contention entirely — both workloads run simultaneously.
- No swap latency, no vLLM cold starts.
- Simpler operationally — no swap controller to maintain.

**Cons:**
- Hardware cost (GBP 250-400 for a suitable card).
- Desktop chassis may not have a second PCIe x16 slot or sufficient PSU headroom.
- Thermal concerns in a single-node setup running two GPUs continuously.

**Why rejected:** The KEDA swap controller is a zero-cost software solution that addresses the current burst pattern adequately. A second GPU becomes the correct investment if swap frequency exceeds ~10/day or if multi-user access requires zero-downtime inference. Filed as a future option under VW-304.

### Alternative 2: CPU-Only Embedding (Ollama or ONNX Runtime)

**Description:** Run embedding on CPU using Ollama (already deployed, CPU-only) or ONNX Runtime with the Qwen3-Embedding-4B model, avoiding GPU contention entirely.

**Pros:**
- No GPU contention at all — vLLM keeps exclusive access.
- No swap controller complexity.
- Ollama infrastructure already exists.

**Cons:**
- Massive throughput reduction. CPU embedding of Qwen3-Embedding-4B at BF16 on a Ryzen 9 would yield ~2-5 documents/second vs. ~50-100 documents/second on GPU via TEI. A 200-document delta batch would take 40-100 seconds on CPU vs. 2-4 seconds on GPU.
- The MacBook embedding agent was previously decommissioned (VW-203, fact 2683) partly because CPU/MPS embedding was too slow and resource-hungry for production use. Repeating that pattern on the desktop is a known anti-pattern.
- Typed memory records embedding agent memory leaks (fact 3029) and stuck states (fact 2685) from CPU-based embedding — the failure mode is well-documented.

**Why rejected:** The 10-50x throughput penalty makes CPU embedding impractical for the burst pattern created by Jira connector syncs, where hundreds of documents may arrive in a single batch. The swap controller preserves GPU-speed embedding while managing the contention.

### Alternative 3: Smaller Embedding Model (Qwen3-Embedding-0.6B)

**Description:** Downgrade from Qwen3-Embedding-4B (~8 GB VRAM) to Qwen3-Embedding-0.6B (~1.5 GB VRAM). At 1.5 GB, it might coexist with vLLM on the 16 GB card without swapping.

**Pros:**
- Potentially eliminates the need for swapping entirely if vLLM can fit in 14.5 GB.
- Simpler architecture — no swap controller.

**Cons:**
- **Dimension mismatch is a blocker.** Typed memory (fact 3944) records that Qwen3-Embedding-0.6B produces different-dimensional vectors than the 2560d Qdrant collection built with the 4B model. Switching would require re-embedding the entire 148K-chunk corpus — a multi-hour GPU-exclusive operation.
- vLLM with KV cache and prefix caching consumes ~15.6 GB. Even with a 1.5 GB embedding model, the combined 17.1 GB exceeds the 16 GB card. Coexistence is still impossible.
- Lower embedding quality would degrade retrieval precision across all RAG queries.

**Why rejected:** The dimension mismatch alone is a hard blocker without a full re-index. Even if dimensions matched, the VRAM arithmetic does not work.
