# ADR-043: Memory System Derived-Data GDPR Position

**Status**: Proposed
**Date**: 2026-05-17
**Deciders**: Ryan Duffy
**Jira**: VW-391

## Context

The Rootweaver memory system ingests Claude Code session transcripts and decomposes them into a multi-layer derived-data stack stored across Postgres (with pgvector), and FalkorDB. The storage layers, in order of derivation distance from source, are:

1. **Sessions** — top-level anchor row per conversation (`memory.sessions`).
2. **Rounds** — individual user/assistant turn pairs within a session (`memory.rounds`), including raw message UUIDs and compressed summaries.
3. **Facts** — structured extractions from rounds: decisions, learnings, problems, patterns (`memory.facts`), each with text, reasoning, entities, confidence scores, temporal validity window (`t_valid`, `t_invalid`), and a pgvector embedding column.
4. **Artifacts** — references to side-effects produced during a round: git commits, vault documents, file modifications (`memory.artifacts`).
5. **File snapshots** — point-in-time captures of file state referenced by a round (`memory.file_snapshots`).
6. **Fact corrections** — supersession records tracking when a later fact invalidated an earlier one (`memory.fact_corrections`).
7. **FalkorDB graph nodes** — entity and relationship nodes derived from facts and artifacts, powering `trace_decision_chain` and `graph_shortest_path` queries against the `code_knowledge` and `document_knowledge` graphs.
8. **Embeddings** — 1024-dimensional vectors (Qwen3-Embedding-0.6B) stored in `memory.facts.embedding` and in Qdrant collections for semantic recall.

A data subject exercising GDPR Article 17 (Right to Erasure) against session data creates a cascade question: which of these layers constitute "personal data" under the regulation, and which are sufficiently anonymised statistical aggregates exempt from the right?

This question is complicated by a genuine regulatory split:

- **Hamburg DPA position** (2023 guidance on AI-derived data): derived data that can be re-linked to a natural person retains its character as personal data and must be deleted upon valid erasure request. The test is whether the derived record, alone or combined with other data the controller holds, can identify the data subject.
- **EDPB Guidelines 05/2014 on Anonymisation Techniques**: data is anonymous only when re-identification is irreversible, considering "all means reasonably likely to be used." Statistical aggregates (e.g., average session length) may qualify; individual-level extractions (facts with entity references, timestamps, and reasoning chains) generally do not.
- **OWASP AI Security and Privacy Guide** (2024): recommends hard-delete as the default for AI training/inference data, noting that soft-delete and anonymisation approaches create audit surface area and re-identification risk that most organisations underestimate.

The project is intended for public release. A defensible, clearly documented position is required before the memory system ships outside the maintainer's personal infrastructure.

### Typed Memory Evidence

Two prior decisions were retrieved from the memory system's own `recall_decisions` tool, confirming directional consensus established during VW-391 implementation planning:

- **Fact 1721** (session `e0bfe7fb`, 2026-05-17, confidence 0.8, rerank 0.34): *"Adopt a hard-delete with ON DELETE CASCADE strategy for GDPR compliance."* Reasoning: *"This approach ensures that all derived data is removed in accordance with GDPR requirements, as validated by recent legal opinions."*
- **Fact 1747** (session `671c7bf9`, 2026-05-17, confidence 0.8, rerank 0.23): *"Use hard-delete-by-default with ON DELETE CASCADE for data retention."* Reasoning: *"This approach aligns with industry consensus on compliance with GDPR and right-to-be-forgotten requirements."*

No contradicting decisions or open problems were found via `recall_decisions(topic="memory retention")`, `recall_decisions(topic="personal data")`, or `find_problems(error_signature="data retention compliance")`. The directional signal is unanimous.

The `memory_forget.py` operator tool (already implemented under VW-391 Plan E T3) codifies this position: `DELETE FROM memory.sessions WHERE id = :sid` triggers `ON DELETE CASCADE` across rounds, facts, artifacts, file_snapshots, and fact_corrections. A tombstone-then-reap pattern provides a 7-day grace window before hard deletion.

## Decision

**We treat all layers of the memory system's derived-data stack as personal data for GDPR purposes and cascade hard-delete them when the source session is erased.**

Specifically:

### 1. Classification: All Derived Layers Are Personal Data

Every layer from sessions through fact_corrections contains or can be trivially re-linked to the data subject (the user whose session produced the data). Facts contain free-text reasoning that routinely includes the user's name, project names, file paths on their machine, and decision context. Embeddings encode semantic representations of that same content. FalkorDB graph nodes carry entity names and relationship labels derived from the same source. None of these layers pass the EDPB anonymisation test ("all means reasonably likely to be used" for re-identification).

We do not attempt to distinguish "more personal" from "less personal" layers. The Hamburg DPA position is the stricter reading, and adopting it eliminates the need for per-layer anonymisation audits that would be both expensive and fragile as the schema evolves.

### 2. Erasure Mechanism: ON DELETE CASCADE from sessions

The `memory.sessions` table is the cascade root. All child tables use foreign keys with `ON DELETE CASCADE`:

```
memory.sessions (root)
  └─ memory.rounds        (FK session_id → sessions.id, CASCADE)
       └─ memory.file_snapshots (FK round_id → rounds.id, CASCADE)
  └─ memory.facts         (FK session_id → sessions.id, CASCADE)
       └─ memory.fact_corrections (FK fact_id → facts.id, CASCADE)
  └─ memory.artifacts     (FK session_id → sessions.id, CASCADE)
```

A single `DELETE FROM memory.sessions WHERE id = :sid` removes all derived data in one atomic transaction. The `memory-forget` operator tool exposes this with `--immediate` (hard-delete now) and default tombstone mode (soft-delete with 7-day grace, reaped by the `memory-tombstone-reaper` CronJob).

### 3. FalkorDB Graph Nodes: Synchronous Purge

FalkorDB graph nodes derived from a session's facts and artifacts are not covered by Postgres CASCADE. The erasure path must include a synchronous `GRAPH.QUERY` call to delete nodes and edges tagged with the session ID. The `memory-forget` tool will be extended (if not already) to issue:

```cypher
MATCH (n {session_id: $sid}) DETACH DELETE n
```

against both the `code_knowledge` and `document_knowledge` graphs.

### 4. Embeddings: Covered by Fact Deletion

Embeddings stored in `memory.facts.embedding` (pgvector column) are deleted as part of the facts CASCADE. Embeddings replicated to Qdrant collections for the memory system's semantic recall must be purged by collection-point-delete keyed on fact ID. This is an implementation task downstream of this ADR.

### 5. Tombstone Grace Period: 7 Days Default

The default erasure path is tombstone (set `memory.sessions.tombstoned_at = now()`), with the `memory-tombstone-reaper` CronJob hard-deleting tombstoned sessions after 7 days. This grace period allows:

- Accidental-deletion recovery (user or operator error).
- Audit trail for the deletion event itself (the tombstone timestamp is the audit record; once hard-deleted, only the reaper's log retains evidence that deletion occurred).

The `--immediate` flag bypasses the grace period for GDPR-deadline scenarios (Article 17 requires erasure "without undue delay" — generally interpreted as within 30 days, but an immediate path is available for urgent requests).

### 6. No Anonymisation Alternative for Individual Records

We explicitly reject per-record anonymisation (e.g., stripping PII from facts while retaining the structural skeleton) as a substitute for deletion. Reasons:

- Facts contain free-text reasoning where PII is semantically entangled with the content. Redaction would destroy the utility of the record.
- Embeddings encode PII-bearing text; replacing the embedding with a redacted-text embedding changes the vector, invalidating downstream similarity queries.
- The anonymisation surface area grows with every schema change, creating an ongoing compliance maintenance burden disproportionate to the value of retaining anonymised skeletons.

## Consequences

### Positive

- **Regulatory clarity**: A single, unambiguous position ("everything cascades") eliminates per-layer classification debates during compliance reviews.
- **Implementation simplicity**: `ON DELETE CASCADE` at the database level means application code cannot accidentally retain orphaned derived data. The operator tool is 132 lines of Python.
- **OWASP alignment**: Hard-delete-by-default satisfies the OWASP AI Security and Privacy Guide's recommendation without requiring soft-delete audit machinery.
- **Public release readiness**: External users and contributors can understand the data lifecycle from one ADR rather than navigating per-layer retention policies.

### Negative

- **Irreversibility after reaper runs**: Once the 7-day grace period expires and the reaper hard-deletes, recovery requires Postgres point-in-time recovery (PITR) from backup. This is an operational cost accepted in exchange for compliance certainty.
- **Graph purge latency**: FalkorDB `DETACH DELETE` on a session with many derived nodes may take seconds. This is acceptable for an operator-initiated erasure path but would need batching if erasure volume ever scales (unlikely for a single-user system, but noted for public-release generality).
- **Loss of aggregate analytics**: Deleting all derived data means session-level analytics (e.g., "average facts per session over time") lose data points. If aggregate analytics become important, a separate, genuinely anonymous counter table (no session ID, no timestamps finer than month, no text) could be introduced as an additive change without weakening the erasure guarantee.

### Neutral

- **Qdrant sync**: The memory system's Qdrant integration for semantic recall is a replication target, not a source of truth. Purging Qdrant points by fact ID is a mechanical follow-on task, not an architectural decision.
- **Backup retention**: Postgres backups (PITR, pg_dump) will contain deleted data until the backup rotation window expires. This is standard for any deletion-based compliance approach and is addressed by backup retention policy (orthogonal to this ADR).

## Alternatives Considered

### Alternative A: Per-Layer Classification with Selective Anonymisation

Classify each layer individually: sessions and rounds as personal data (delete on erasure), facts and artifacts as "derived analytical data" (anonymise by stripping entity references and replacing text with hashes), embeddings as "statistical representations" (retain as-is on the theory that vectors alone cannot identify a person).

**Rejected because**:

- The EDPB "all means reasonably likely" test fails for our facts: they contain the user's name, project-specific terminology, and machine-local file paths. Hashing the text still allows re-identification via the entity array and temporal correlation with surviving session metadata.
- Embeddings of personal-data text are personal data under the Hamburg DPA interpretation. A 1024-dimensional embedding of "Ryan decided to use FalkorDB for the graph layer on 2026-03-15" encodes the who, what, and when. Nearest-neighbour search against a known corpus trivially re-identifies the subject.
- Maintaining per-layer anonymisation logic across schema migrations creates an ongoing compliance engineering burden. Every new column or table requires a classification decision and anonymisation implementation.
- The selective approach optimises for data retention at the cost of compliance confidence — the wrong trade-off for a project seeking public release.

### Alternative B: Soft-Delete Everywhere (No Hard Delete)

Mark sessions as deleted (`is_deleted = true`) but retain all data indefinitely. Exclude soft-deleted records from query paths. Provide an admin "purge" command for periodic hard-deletion if desired.

**Rejected because**:

- Soft-delete does not satisfy GDPR Article 17. The data remains in the database, accessible to anyone with database credentials or backup access. "Not showing it in the UI" is not erasure.
- Soft-delete creates a growing pool of "deleted but present" data that increases the blast radius of a data breach. If the database is compromised, the attacker gets all soft-deleted personal data — data the controller told the subject was erased.
- OWASP explicitly warns against soft-delete for AI system data, noting that it "creates a false sense of compliance while retaining the full risk surface."
- The only advantage of soft-delete (easy undo) is adequately served by the 7-day tombstone grace period before hard-delete.

### Alternative C: Event-Sourced Tombstones with Crypto-Shredding

Retain all derived data but encrypt it with a per-session key stored in a separate key management service. On erasure, delete the key ("crypto-shredding"), rendering the data unreadable without cryptanalytic effort exceeding the "all means reasonably likely" threshold.

**Rejected because**:

- Crypto-shredding is a valid GDPR compliance technique (acknowledged by the EDPB and UK ICO), but it introduces significant operational complexity: a KMS dependency, per-session key rotation, encryption-at-rest for every derived-data column, and a migration path for existing unencrypted data.
- For a single-user system with moderate data volume (thousands of sessions, not millions), the engineering cost of crypto-shredding exceeds the cost of hard-delete by an order of magnitude.
- FalkorDB does not natively support field-level encryption, so graph nodes would require application-level encrypt/decrypt on every read and write — a performance and complexity penalty.
- Crypto-shredding optimises for "retain structure, destroy content" — useful for systems that need to preserve referential integrity across deleted records (e.g., billing systems). The memory system has no such requirement; deleting a session's entire derived-data tree is semantically correct.

This alternative is noted as a viable future path if the system scales to multi-tenant use where per-tenant key management is already required for other reasons.
