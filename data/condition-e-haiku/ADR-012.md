# ADR-012: Journal Pipeline Quality Overhaul — Model Selection & E2A Architecture

**Status**: Proposed
**Date**: 2026-02-08
**Jira**: VW-33 (Journal Pipeline Quality Overhaul)
**Supersedes**: Partial aspects of ADR-010 (Streaming Journal System)

---

## Context

The journal streaming pipeline (VW-6, VW-30) produces automated journal entries from Claude Code sessions via Kafka-driven synthesis. The pipeline infrastructure works end-to-end — Kafka topics flow correctly, triggers fire, entries get written to vault. However, **content quality is not publication-ready**, and investigation has revealed systemic hallucination issues affecting user trust.

### Evidence of Quality Failure

Cross-referencing actual JSONL session transcripts against OTEL-path synthesized journal entries for session `session_20260207_163955` revealed a catastrophic mismatch:

| Journal Claims | Actual Session Content (JSONL) |
|---|---|
| `deploy_watchdog.sh` JSON conversion & Docker troubleshooting | Writing Season 3 Episode 1 blog post about retro anime image generation |
| CI pipeline integration across 4 consecutive entries | Zero mentions of `deploy_watchdog.sh` in entire 47-message transcript |
| Exit code debugging sequences | No exit codes discussed anywhere in session |

**Finding**: Every OTEL-path entry was entirely fabricated. The voice and format were correct (prompt engineering v8 had nailed Ryan's writing style), but the factual content was entirely fictional — no ground truth connection whatsoever.

**Impact**: 3,872 fabricated journal files (77% of vault streaming directory) accumulated over 3 months before detection, rendering the entire OTEL synthesis path unusable for publication. This was later confirmed and remediated via VW-276 (May 2026), which disabled the OTEL path and purged all hallucinated entries.

### Root Causes Identified

**1. OTEL Events Are Structurally Too Sparse**

Anthropic's Claude Code OTEL telemetry is privacy-stripped by design. The event stream carries only `[event_name, tool_name, decision, duration_ms, cost_usd, model]` — no file paths, no command text, no prompt content, no tool results. OTEL richness score: 2/10. Even with 50 events per session, the LLM has zero semantic content to ground on.

**2. DeepSeek R1 8B Is the Wrong Model for Summarization**

- **Training**: Reinforcement-learning optimized for math/reasoning, not text summarization
- **Behavior**: R1 "overhelps" — when given sparse input, fills gaps with plausible-sounding fiction that *sounds right* but is entirely invented
- **Vectara Hallucination Leaderboard**: ~14.3% hallucination rate (full model); 8B distillation performs worse
- **For journal entries requiring factual grounding**: 14%+ fabrication rate is catastrophic — unacceptable for publication pipeline

**3. Single-Shot Generation with No Verification**

The original pipeline:
1. Generate journal entry in one LLM pass
2. Write to vault immediately
3. No quality gate, no cross-check against source material

Production content systems universally use multi-stage pipelines with human or automated verification gates. Single-shot is faster but unsafe at publication quality.

**4. Temperature Too High for Factual Content**

- **Current**: 0.7 (intended for conversational variety)
- **Research consensus**: 0.3–0.5 optimal for factual tasks — low enough for grounding, high enough to avoid repetitive phrasing
- **Trade-off**: Sacrifice some stylistic variety for faithful content reproduction

### Vision: Full Automation Pipeline

**Goal**: Claude Code session → faithful journal entry → weekly blog post → LinkedIn social media, with zero manual intervention for quality-passing entries.

**Current blocker**: Content quality prevents auto-publication. Human time spent manually editing fabricated entries defeats the automation purpose. High hallucination rate broke user trust in the system.

---

## Decision

### 1. Deprecate OTEL Synthesis Path (Complete Architectural Rejection)

The OTEL path is **structurally incapable** of producing quality journal entries due to privacy-stripping. No prompt engineering, no parameter tuning, and no temperature adjustment can fix an input gap that fundamental.

**Action**:
- Set `MIN_OTEL_EVENTS = 999` to effectively disable OTEL synthesis
- Log deprecation warning on trigger
- Remove implementation fully in a future version
- **JSONL becomes the sole synthesis path** (verified faithful in VW-276 validation: 5/5 random E2A samples grounded against ground truth)

**Rationale**: The OTEL boundary detector remains useful for *triggering* JSONL session processing (detecting session boundaries, milestones) but should not attempt narrative synthesis.

### 2. Replace DeepSeek R1 8B with Qwen3-14B for Extraction

| Criterion | DeepSeek R1 8B | Qwen3-14B |
|---|---|---|
| Hallucination rate | ~14%+ | ~4–6% (estimated) |
| Task alignment | Math/reasoning (poor fit) | General-purpose (good fit) |
| VRAM (Q4 quantization) | ~5GB | ~9GB |
| RTX 4080 fit (16GB total) | Yes (loose) | Yes (tight, but stable) |
| Summarization quality | Weak | Strong |

**Qwen3-14B** is the best free alternative: fits on existing hardware with headroom for KV cache, significantly lowers hallucination rate, and is purpose-built for general-purpose language tasks including summarization.

### 3. Implement Extract-then-Abstract (E2A) Two-Stage Pipeline

The single most impactful architectural change. Backed by research (arxiv.org/html/2410.06520v1) showing dramatically reduced hallucination when the synthesis model works from **structured facts** rather than raw transcripts.

#### Stage 1: EXTRACT (Local — Qwen3-14B on vLLM)

**Input**: JSONL session transcript (formatted as multi-turn messages)

**Process**: Guided JSON extraction via vLLM

**Output**: Structured JSON with schema:
```json
{
  "session_date": "2026-02-07",
  "duration_estimate": "2 hours 15 minutes",
  "primary_goals": [
    "Write Season 3 Episode 1 blog post",
    "Test anime image generation"
  ],
  "technical_decisions": [
    {
      "decision": "Used HuggingFace Spaces for image generation",
      "reasoning": "Retro anime aesthetic required specific model tuning"
    }
  ],
  "files_modified": [
    {
      "path": "10-Blog/drafts/season3-episode1.md",
      "change_type": "created",
      "description": "Seasonal release draft with narrative arc"
    }
  ],
  "problems_encountered": [
    {
      "problem": "Image generation timeout on first attempt",
      "solution": "Increased inference timeout to 90s",
      "resolved": true
    }
  ],
  "tools_and_services": ["HuggingFace Spaces", "Pillow", "diffusers"],
  "key_learnings": [
    "Anime style generation requires specific LoRA weights"
  ],
  "collaboration_highlights": []
}
```

**Temperature**: 0.3 (favor accuracy, minimize hallucination)

**Purpose**: Extract only facts present in transcript. If a fact wasn't explicitly discussed, it doesn't appear in extraction.

#### Stage 2: SYNTHESIZE (API — Claude Sonnet 4.5 Batch)

**Input**: Extracted facts JSON + style prompt + previous entry for context

**Process**: Turn verified facts into Ryan's voice via Claude API

**Output**: Publication-ready journal entry (markdown)

**Temperature**: 0.5 (balance accuracy with stylistic variety)

**Purpose**: Synthesis model never sees raw noisy transcript. It works only from structured, verifiable facts. If a fact wasn't extracted in Stage 1, it cannot appear in Stage 2 — hallucination is architecturally prevented.

### 4. Add LLM-as-Judge Quality Gate (Auto-Routing)

#### Stage 3: JUDGE (API — Claude Haiku 4.5)

**Input**: Draft entry + extracted facts JSON

**Output**: Scores (0–10) for five dimensions:
- **Factual grounding**: Every claim traces to extracted facts JSON. Score loss if claim is unsupported.
- **Completeness**: All significant session events covered. Score loss if major activities omitted.
- **Hallucination**: Binary check. Any invented content triggers fail.
- **Tone/style**: Appropriate for blog/LinkedIn voice. Score loss if tone drifts.
- **Coherence**: Logical narrative flow. Score loss if structure breaks down.

**Acceptance Criteria**:
- All scores ≥ 7
- Hallucination = 0 (binary pass)

**Action on Result**:
- **Pass**: Auto-publish to vault (fully automated)
- **Fail**: Flag for human review OR trigger revision loop (attempt to regenerate with feedback)

**Research Support**: LLM-as-judge achieves 80% agreement with human evaluators and costs ~98% less than human annotation.

### 5. Adjust Inference Parameters

| Parameter | Current | Proposed | Rationale |
|---|---|---|---|
| Temperature | 0.7 | 0.3 (extract) / 0.5 (synth) | Lower for factual accuracy |
| Max tokens (extraction) | 2048 | 1500 | Extraction should be concise |
| Max tokens (synthesis) | 2048 | 2048 | Blog entries need space |
| Model | DeepSeek-R1-8B | Qwen3-14B (local) + Claude Sonnet 4.5 (API) | Quality uplift |

### 6. Keep OTEL Boundary Detector for Triggering (Not Synthesis)

OTEL boundary detection (time gaps, milestones, topic changes) remains valuable for knowing *when* to process JSONL files. Architecture:

```
OTEL events → boundary_detector → journal.triggers → session_parser → JSONL processing
                                  (trigger only,     (finds JSONL file,
                                   no synthesis)      routes to E2A pipeline)
```

The boundary logic is pure-data (no LLM involved), faithful, and useful.

## Cost Analysis

### Estimated Monthly Cost (Hybrid — E2A + Judge)

| Stage | Model | Volume/Month | Input Cost | Output Cost | Total |
|---|---|---|---|---|---|
| Extraction | Qwen3-14B (local) | 30 sessions | $0 | $0 | **$0** |
| Synthesis | Claude Sonnet 4.5 Batch | 30 entries | ~60K input tokens × $1.50/M | ~45K output × $7.50/M | **$0.43** |
| Judge | Claude Haiku 4.5 | 30 entries | ~120K input × $0.50/M | ~15K output × $2.50/M | **$0.10** |
| Revision (20% fail rate) | Claude Sonnet 4.5 Batch | ~6 entries | ~12K × $1.50/M | ~9K × $7.50/M | **$0.09** |
| Weekly blog post | Claude Sonnet 4.5 Batch | 4 posts | ~48K × $1.50/M | ~12K × $7.50/M | **$0.16** |
| | | | | **Monthly Total** | **~$0.78** |

*Batch API pricing is 50% of standard. Standard would be ~$1.56/month.*

### Comparison Table

| Approach | Cost/Month | Quality (1–10) | Automation Level |
|---|---|---|---|
| Current (R1 8B, single-shot) | $0 | 2/10 | Fully automated but unusable |
| Qwen3-14B only (local upgrade) | $0 | 5/10 | Automated, needs human editing |
| **Hybrid E2A (proposed)** | **~$0.78** | **8/10** | **Auto-publish above threshold** |
| Full API (Claude Sonnet all stages) | ~$2.41 | 9/10 | Auto-publish, simplest code |

**Recommendation**: Hybrid E2A balances cost, quality, and architectural complexity. $0.78/month is negligible; the automation uplift from 2→8 quality is dramatic.

## Consequences

### Positive

- **Factual grounding**: Journal entries are grounded in actual session content. No more `deploy_watchdog.sh` fabrications.
- **Publication quality**: ~8/10 quality enables auto-publish to vault → blog → LinkedIn. Automation value realized.
- **Measurable quality metrics**: LLM-as-Judge provides objective score per entry, enabling regression detection.
- **Composable architecture**: Extraction JSON feeds blog synthesis, LinkedIn post formatting, weekly summaries. One extraction, many outputs.
- **Hardware efficient**: Qwen3-14B fits existing RTX 4080 without GPU upgrades.
- **Cost negligible**: ~$0.78/month is below noise floor for personal infrastructure.

### Negative

- **API dependency**: Synthesis/Judge stages require Anthropic API (network, uptime dependency).
- **Latency increase**: API calls ~5–15s vs local ~2–5s. Mitigated by Batch API (negligible for daily pipeline).
- **Pipeline complexity**: 3 stages vs 1. More moving parts, more monitoring.
- **Model download**: Qwen3-14B ~9GB download required.

### Risks

- **Claude API pricing changes**: Currently negligible (~$1/month); would need >100× increase to matter.
- **Qwen3-14B extraction untested**: Haven't validated extraction quality on real transcripts. *Mitigation*: A/B test against R1 8B on 10 real sessions before full rollout.
- **Batch API latency**: Batch jobs queued up to 24h. *Mitigation*: Use standard API for same-day publishing; Batch for archive backfill.
- **Judge overfitting**: LLM judge may learn to score high for stylistic features that aren't grounding. *Mitigation*: Include ground-truth examples in judge prompt; periodic audit against human spot-checks.

## Alternatives Considered

### A. Keep R1 8B, Improve Prompts

**Rejected.** The model architecture is fundamentally misaligned with summarization. Vectara's leaderboard already shows ~14%+ hallucination rate. We already applied grounding blocks in VW-30 v9 (JSON schema + Judge stage for extraction output). Those help marginally but cannot fix a 14% fabrication rate at its source.

### B. Upgrade to 70B+ Local Model

**Rejected for now.** Qwen 2.5 72B / Llama 3.3 70B require ~40GB VRAM (Q4). RTX 4080 has 16GB. Would need multi-GPU or a new card. Revisit if Qwen3-14B + API synthesis quality is insufficient after 3–6 months of real-world operation.

### C. Fully API-Based Pipeline

**Viable but unnecessary.** Local extraction is free and fast. API synthesis at ~$0.78/month vs ~$2.41/month for full API. Hybrid keeps GPU useful.

### D. Fine-Tune a Small Model on Ryan's Writing Style

**Interesting but premature.** Requires curated dataset of "good" journal entries. No such dataset exists yet because the pipeline hasn't produced publication-quality output. Revisit after 3–6 months of hybrid pipeline operation to use E2A entries as few-shot training examples.

### E. Migrate to Groq or External LLM Provider

**Rejected.** Groq offers fast inference but their models (Llama, Mixtral) don't outperform Claude for synthesis. External dependency introduces cost, latency, and lock-in. Hybrid approach uses best-of-breed: local for extraction, Claude for synthesis (proven quality).

## Implementation Plan

### Phase 1: Model Swap (1–2 hours)

1. Download Qwen3-14B GGUF to vLLM server
2. Update `VLLM_MODEL` env var in synthesizer K8s deployment
3. Lower temperature to 0.3 for extraction
4. Test extraction quality on 3 real session transcripts
5. Validate extraction JSON schema

### Phase 2: E2A Pipeline (4–6 hours)

1. Add extraction stage to `journal_synthesizer.py`
   - Call Qwen3-14B with guided JSON schema
   - Validate output structure
2. Add Claude API client for synthesis stage
   - Integrate Anthropic SDK
   - Wire extraction JSON → synthesis
3. Test on 5 real JSONL sessions
4. Verify output quality (manual read-through)

### Phase 3: Quality Gate (2–3 hours)

1. Add Haiku judge stage
   - Define scoring criteria (factual grounding, completeness, hallucination, tone, coherence)
   - Implement 0–10 scoring logic
2. Set acceptance threshold (all scores ≥ 7, hallucination = 0)
3. Wire auto-publish vs human-review routing
4. Add monitoring (score distribution, pass rate, revision frequency)

### Phase 4: OTEL Deprecation (30 min)

1. Set `MIN_OTEL_EVENTS = 999` in synthesizer
2. Log deprecation warning on trigger receipt
3. Verify OTEL boundary detector still triggers JSONL processing
4. Test one full cycle end-to-end

### Phase 5: Blog + Social Pipeline (4–6 hours)

1. Weekly aggregation of daily journal entries
2. Blog synthesis (Claude Sonnet)
3. LinkedIn post formatting
4. Scheduling via Prefect flows
