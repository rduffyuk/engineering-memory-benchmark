# ADR-034: Data Source Connector Architecture

**Status**: Proposed  
**Date**: 2026-04-15  
**Jira**: VW-184 (Framework), VW-186 (Source Filter), VW-188 (Jira Activation), VW-232 (Indexer Consumer)  
**Deciders**: Ryan Duffy

---

## Executive Summary

Rootweaver currently ingests data from a single source: the local Obsidian vault (45K chunks) via inotify → Kafka → KEDA ScaledJob → Qdrant. **Jira and Confluence are accessible only via MCP tools during active Claude Code sessions** — their content is **not indexed into Qdrant**, blocking cross-source search and temporal filtering.

This ADR proposes a Kafka-native connector framework that makes Jira, Confluence, GitLab, and other sources searchable within the same Qdrant collection as vault documents. The decision is informed by three layers of evidence:

1. **Decision Context** (VW-184, VW-186, VW-188): Analysis of market differentiators, existing MCP integration, and Onyx architectural patterns
2. **Implementation Validation** (VW-188 Session Reports, Technical Log 2026-04-15): JiraConnector end-to-end verified; 231 Jira issues synced to Kafka; watermark tracking working
3. **Current Blockers** (VW-186, VW-232): Source filter silently dropped in SearchRequest (fixed in commit c392e56); connector-indexer consumer missing __main__ block (VW-232 plan filed)

## Context

### The Problem: Single-Source Indexing

| Source | Integration Type | Indexed in Qdrant | Bi-directional | Status |
|--------|-----------------|-------------------|----------------|--------|
| **Obsidian Vault** | File watcher → Kafka → Qdrant | Yes (45K chunks) | Read-only | Production |
| **Jira** | MCP tool (Atlassian MCP server) | **No** | Read + Write (CRUD) | Production |
| **Confluence** | MCP tool (Atlassian MCP server) | **No** | Read + Write (CRUD) | Production |
| **GitLab** | CI/CD runner only | **No** | Push only | Production |

**Impact**: Cannot answer cross-source questions like *"what Jira issues relate to the search pipeline changes last week?"* — requires both vault context AND Jira data in the same search index.

### Market Analysis (April 2026)

Connector coverage is the #1 differentiator between Rootweaver and competitor platforms:

| Platform | Connectors | Indexed | Strategy |
|----------|-----------|---------|----------|
| **Glean** | 100+ | Yes (enterprise graph) | Proprietary connectors |
| **Onyx (Danswer)** | 40+ | Yes (Vespa) | MIT-licensed framework + polling |
| **Rootweaver** | 1 (vault) + 2 MCP tools (live only) | Vault only | MCP + custom integrations |

### Existing Onyx Architecture (Reference Model)

Onyx is the closest open-source architectural peer. Their model:

1. **Base class**: `BaseConnector` with `load_from_state()` (incremental) and `poll_source()` (periodic)
2. **Document model**: Normalised `Document` with typed `Section` objects
3. **Polling**: Configurable intervals per connector (not webhook-driven)
4. **Credentials**: Per-connector OAuth flows or API key storage, encrypted at rest
5. **Processing**: Connector → Document → Chunker → Embedder → Vespa
6. **Effort**: ~200-400 lines of Python per connector (API pagination, rate limiting, doc normalisation)

## Decision: Kafka-Native Connector Framework

### Architecture Diagram

```
┌─────────────────────────────────────────────────────┐
│                  Data Sources                        │
│                                                      │
│  Jira    Confluence  GitLab   Slack   Google Drive   │
│   │         │          │        │         │          │
│   ▼         ▼          ▼        ▼         ▼          │
│  ┌──────────────────────────────────────────────┐    │
```

### Why Kafka, Not Direct Qdrant Writes

1. **Existing pattern**: Vault file indexing already uses Kafka (`vault.file-events`). Adding a new topic is trivial.
2. **Decoupled**: Connectors don't need Qdrant access — they just produce messages. The indexer handles embedding and storage.
3. **Backpressure**: KEDA scales the indexer based on Kafka consumer lag. Burst of 10K Confluence pages? Indexer scales up, processes, scales back to zero.
4. **Replayable**: If the embedding model changes (e.g., Qwen3-Embedding-4B upgrade), replay the topic to re-embed all connector docs.
5. **Observable**: Kafka lag metrics already feed Prometheus → Grafana. Connector health is automatically monitored.

### Why Not Adopt Onyx Wholesale

Onyx is MIT-licensed and could be deployed alongside Rootweaver. However:

1. **Vespa dependency**: Onyx uses Vespa, not Qdrant. Running both vector stores doubles infrastructure.
2. **No Kafka**: Onyx uses polling + direct DB writes. We'd lose our event-driven architecture.
3. **No temporal search**: Onyx doesn't have date-aware retrieval (we have 40 percentage points improvement via temporal filtering).
4. **No quality benchmarks**: No regression suite for RAG quality.
5. **Python connector code is portable**: Individual connector implementations (~200-400 lines each) can be adapted without adopting the full Onyx stack.

**Decision**: Build our own framework, borrowing Onyx's patterns but integrating into our existing Kafka + KEDA + Qdrant pipeline.

### Connector Interface

```python
from abc import ABC, abstractmethod
from dataclasses import dataclass
from datetime import datetime
from typing import List, Optional, Dict, Any, Iterator
from enum import Enum

class SourceType(str, Enum):
    VAULT = "vault"
    JIRA = "jira"
    CONFLUENCE = "confluence"
    GITLAB = "gitlab"
    SLACK = "slack"
    GOOGLE_DRIVE = "google_drive"
    NOTION = "notion"
    EMAIL = "email"
    WEB = "web"

@dataclass
class ConnectorDoc:
    """Normalised document from any source."""
    source: SourceType
    source_id: str              # Unique ID within source
    title: str
    content: str                # Markdown-normalised
    url: Optional[str] = None
    author: Optional[str] = None
    created_at: Optional[datetime] = None
    updated_at: Optional[datetime] = None
    metadata: Dict[str, Any] = None

class BaseConnector(ABC):
    """Base class for all data source connectors."""

    @abstractmethod
    def poll_source(self) -> Iterator[ConnectorDoc]:
        """Full sync — yield all documents from source."""
        ...

    @abstractmethod
    def incremental_sync(self, since: datetime) -> Iterator[ConnectorDoc]:
        """Delta sync — yield documents changed since last sync."""
        ...

    @abstractmethod
    def validate_credentials(self) -> bool:
        """Test that credentials are valid."""
        ...

    def get_source_type(self) -> SourceType:
        """Return the source type for this connector."""
        ...
```

### Qdrant Collection Strategy

**Decision**: Single collection with source metadata (Option A).

- All connector docs go into `rootweaver_vault` alongside vault documents
- `source` field in payload enables filtering: `source = "jira"`, `source = "vault"`, etc.
- Hybrid search naturally fuses results from all sources
- Temporal filtering works across sources

**Rationale**: The existing `vault_section` facet pattern extends naturally to a `source` facet. Single collection is simpler and delivers cross-source search without federation complexity.

### Scheduling

| Connector | Sync Interval | Rationale |
|-----------|--------------|-----------|
| Jira | Every 15 min | Issues change frequently during work hours |
| Confluence | Every 1 hour | Pages change less frequently |
| GitLab | Every 30 min | MR activity during development |
| Slack | Every 5 min | Conversations are time-sensitive |
| Google Drive | Every 1 hour | Documents change infrequently |

Incremental sync (delta) by default. Full re-sync triggered manually or weekly via Prefect scheduled flow.

### VW-186 — Source Filter Implementation (Partial Fix)

**Problem**: Source filter silently dropped in SearchRequest → Qdrant. Filter accepted at boundary but not plumbed through call chain.

**Evidence (Smoke Test 2026-04-16)**:
```
Test A: no filter → 3 vault results
{
  "query": "VW-184 xyzzy connector",
  "mode": "fast",
  "n_results": 3
}

Test B: source="jira" → SAME 3 results (should be 0)
{
  "query": "VW-184 xyzzy connector",
  "mode": "fast",
  "n_results": 3,
  "source": "jira"
}
```

**Fix**: Three drop-points repaired in commit c392e56:
1. Added module-level helper `build_filter()` in `qdrant.py`
2. Wired through existing `_build_filter` / `_build_filters` methods
3. Tests: 4 unit tests in `test_qdrant_source_filter.py`

**Known Follow-up (VW-186-part-2)**: AUTO/FAST/DEEP agent path still doesn't forward `request.source` to Qdrant. Their `search_async()` signatures don't accept `filters` dict. Requires signature change across all 3 agents.

### VW-232 — Connector-Indexer Consumer Implementation Plan

**Current Status**: Consumer blocking (plan filed 2026-04-26)

**Problem**: `ConnectorIndexer` class defined in `rag_platform/connectors/indexer.py` but has no `__main__` block — KEDA `connector-indexer` ScaledJob spawns empty pods that exit silently.

**Framework Gap**: This is an architectural issue, not a connector-side bug. Once VW-232 ships, the 231 Jira messages already on the Kafka topic will be consumed automatically and Qdrant will populate without re-running the producer.

**Plan**: VW-232 Implementation Plan (12 tasks, includes ScaledJob entrypoint, DLQ topic, error handling, monitoring).

## Consequences

### Positive

- ✅ Jira issues, Confluence pages, GitLab MRs, Slack messages become searchable alongside vault docs in a single query
- ✅ Temporal search works across all sources ("what happened last week" returns vault notes + Jira issues + Slack messages)
- ✅ Knowledge graph enrichment — connector docs feed NER pipeline and document_knowledge graph
- ✅ Benchmark suite can evaluate cross-source retrieval quality
- ✅ Reuses existing infrastructure (Kafka, KEDA, Qdrant, Prometheus)

### Negative

- ⚠️ Increased Qdrant storage (~2x-5x depending on connector volume)
- ⚠️ More SealedSecrets to manage (one per connector's credentials)
- ⚠️ Connector maintenance burden — API changes, rate limits, OAuth token refresh
- ⚠️ Single-collection approach means noisy sources (e.g., high-volume Slack) could dilute search quality without proper weighting

### Risks

- **Slack volume**: Active Slack workspaces can produce 10K+ messages/day. Need per-channel filtering and message-length thresholds.
- **Stale data**: If a connector fails silently, indexed data becomes stale. Need health checks and staleness alerts.
- **Credential rotation**: OAuth tokens expire. Need automated refresh or alerting on expiry.

## Alternatives Considered

### Alternative 1: Adopt Onyx Wholesale

Deploy Onyx alongside Rootweaver for its connectors.  
**Rejected**: Adds Vespa dependency, loses Kafka/KEDA pipeline, no temporal search, no benchmark integration.

### Alternative 2: MCP-Only (Current State)

Keep using MCP tools for live queries, don't index external data.  
**Rejected**: Doesn't support cross-source search, no temporal filtering on external data, requires active session.

### Alternative 3: Build Connectors as MCP Tools

Add `index_jira_to_qdrant` and similar MCP tools.  
**Rejected**: Manual trigger model, doesn't scale, no scheduled sync.
