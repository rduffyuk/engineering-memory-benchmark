## Executive Summary

This ADR documents the decision to implement KEDA (Kubernetes Event-Driven Autoscaling) as the control plane for CPU↔GPU workload swapping on the 16GB RTX 4080. Three GPU workloads (vLLM, embedding service, Prefect worker) cannot coexist; manual coordination is unsustainable. KEDA's event-driven model, already deployed in K3s (HLD-07 §7.1, helm-release v2.19), provides the necessary automation without custom controller code.

## Problem Statement

### Layer 1: Vault Architecture Evidence (ADR-030 §5, HLD-07)

The K3s deployment (rootweaver namespace) currently exposes KEDA in two locations:

1. **Namespace Table** (HLD-01 §6, HLD-07 §7.1 updated 2026-02-19):
   ```
   keda | 2 | Event-driven autoscaling | KEDA operator, metrics-apiserver
   ```

2. **HLD-07 Deployment Infrastructure** (confirmed 2026-02-19, not stale):
   - KEDA namespace exists with 2 pods: operator + metrics-apiserver
   - HelmRelease deployed at version 2.19 (upgraded from v2.13, supports K8s Resource Scaler, gRPC metrics)
   - Prometheus metrics enabled for both operator and metricServer

### Layer 2: Implementation Evidence (neural-vault gpu_monitor.py)

GPU contention is observable and quantified:

```python
# From gpu_monitor.py (lines 71–94):
- Memory info: pynvml.nvmlDeviceGetMemoryInfo(self.handle) 
  → tracks alloc/reserved/free VRAM in real-time
- Utilization: pynvml.nvmlDeviceGetUtilizationRates(self.handle)
  → GPU %, SM %, memory %, encoder %, decoder %
- Power: pynvml.nvmlDeviceGetPowerUsage(self.handle)
  → current draw vs. power_limit (can trigger throttling)
- Process list: pynvml.nvmlDeviceGetComputeRunningProcesses(self.handle)
  → identifies which workload consumes VRAM
```

**Current Symptom**: Three competing workloads trigger oversubscription:
- **vLLM** (DeepSeek-R1-0528-Qwen3-8B-AWQ): ~11–12GB during inference
- **Embedding service** (TEI or local model): ~2–3GB during indexing
- **Prefect worker** (GPU jobs): 1–4GB variable per flow

Total demand: 14–19GB on 16GB hardware → CUDA OOM or GPU throttling.

## Decision

Implement a **KEDA ScaledJob-based GPU swap controller** that:

1. **Monitors vLLM GPU utilization** via Prometheus (DCGM exporter or gpu_monitor.py)
2. **Triggers embedding/Prefect jobs only when GPU idle** (<10% utilization for 5 minutes)
3. **Temporarily suspends lower-priority workloads** when vLLM inference starts
4. **Restores suspended jobs** when GPU becomes available again

This is achieved via:
- **Custom Prometheus rule** to detect GPU idle state
- **ScaledJob trigger** on Prometheus metric (KEDA v2.13+ supports this natively)
- **Pod disruption budgets (PDBs)** to safely evict non-critical workloads
- **Workload priority classes** to define swap-out order

## Rationale

### Why KEDA Instead of Custom Controller?

| Aspect | Custom Controller | KEDA |
|--------|------------------|------|
| **Development** | 400–600 LOC, error-prone | 0 LOC; declarative CRD |
| **Maintenance** | VW team ownership | Upstream CNCF project |
| **Testing** | Integration tests required | kubectl apply --dry-run |
| **Observability** | Custom metrics | Prometheus + KEDA exporter |
| **Failure modes** | Unknown; potential cluster hang | Well-documented, tested |
| **Time to deploy** | 2–3 weeks | 2–3 days (CRDs + ConfigMap) |

### Why Not Manual Scheduling?

Manual coordination via Jira + Slack (current state):
- **Operator burden**: Every GPU job requires human approval
- **Latency**: Hours between request and approval
- **Error rate**: Humans forget to suspend jobs; OOM recurs
- **Scale**: Unworkable when jobs reach 10+ per day (projected by Q2 2026)

KEDA enables **predictable, sub-second responses** to GPU utilization changes.

### Why Prometheus Metrics?

- **Already deployed**: gpu_monitor.py exports Prometheus on port 9400
- **HLD-07 confirmed**: Prometheus is in monitoring namespace (port 30090)
- **Low overhead**: gpu_monitor.py samples every 5 seconds (negligible CPU)
- **Alert-ready**: Prometheus rules can trigger on sustained idle state, not on every fluctuation

## Technical Approach

### 1. Prometheus Rule (New)

```yaml
# rootweaver-gitops/infrastructure/monitoring/gpu-idle-rule.yaml
apiVersion: monitoring.coreos.com/v1
kind: PrometheusRule
metadata:
  name: gpu-idle-detector
  namespace: monitoring
spec:
  groups:
    - name: gpu
      interval: 30s
      rules:
        - alert: GPUIdle
          expr: |
            gpu_utilization_percent < 10
          for: 5m
          labels:
            severity: info
          annotations:
            summary: "GPU idle for 5 minutes"
        
        - alert: GPUUtilized
          expr: |
            gpu_utilization_percent >= 10
          for: 30s  # Exit idle quickly
          labels:
            severity: warning
          annotations:
            summary: "GPU utilization >10%"
```

### 2. KEDA ScaledJob (New)

```yaml
# rootweaver-gitops/infrastructure/keda/gpu-swap-dispatcher.yaml
apiVersion: keda.sh/v1alpha1
kind: ScaledJob
metadata:
  name: gpu-swap-dispatcher
  namespace: rootweaver
spec:
  jobTargetRef:
    template:
      spec:
        # Dispatcher logic: check GPU status, trigger low-priority jobs
        containers:
          - name: dispatcher
            image: rootweaver/gpu-swap-dispatcher:latest
            env:
              - name: PROMETHEUS_URL
                value: "http://prometheus.monitoring:9090"
              - name: GPU_IDLE_THRESHOLD
                value: "10"  # percent
              - name: IDLE_DURATION
                value: "300"  # seconds
  triggers:
    - type: prometheus
      metadata:
        serverAddress: "http://prometheus.monitoring:9090"
        metricName: "keda_gpu_idle"
        threshold: "1"
        query: |
          (gpu_utilization_percent < 10 and changes(gpu_utilization_percent[5m]) < 3)
```

### 3. Workload Priority Classes (New)

```yaml
# rootweaver-gitops/infrastructure/keda/priority-classes.yaml
apiVersion: scheduling.k8s.io/v1
kind: PriorityClass
metadata:
  name: gpu-critical
value: 1000
globalDefault: false
description: "vLLM inference — never evict"
---
apiVersion: scheduling.k8s.io/v1
kind: PriorityClass
metadata:
  name: gpu-batch
value: 100
globalDefault: false
description: "Embedding indexing, Prefect batch — can swap to CPU"
---
apiVersion: scheduling.k8s.io/v1
kind: PriorityClass
metadata:
  name: system-background
value: 10
globalDefault: false
description: "Background tasks — evict first"
```

### 4. Pod Disruption Budget (New)

```yaml
# rootweaver-gitops/infrastructure/keda/pdb-embedding.yaml
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: embedding-service-pdb
  namespace: rootweaver
spec:
  minAvailable: 0  # Allow full eviction when GPU needed
  selector:
    matchLabels:
      app: rag-retriever
      component: embedding
```

### 5. HAMi GPU Sharing (Optional Enhancement)

If VRAM pressure persists even with swapping, consider **HAMi (Heterogeneous AI computing Management Interface)** for static GPU slice allocation:

```yaml
# rootweaver-gitops/infrastructure/hami/device-plugin.yaml
# Allocates: vLLM 12GB, embedding 2GB, Prefect 2GB (static)
# Prevents oversubscription at pod-create time
```

*This is a future enhancement; ADR-029 focuses on dynamic swapping first.*

1. Annotate vLLM pods with `priorityClassName: gpu-critical`
2. Annotate embedding/Prefect pods with `priorityClassName: gpu-batch`
3. Create PDBs to allow safe eviction

### Phase 4: Testing & Observability (Week 2–3)
1. Simulate GPU contention: run vLLM + embedding indexing simultaneously
2. Verify Prefect job is suspended when GPU >10%
3. Add alerting in Grafana for GPU swap events
4. Document rollback procedure

### Phase 5: Production Deployment (Week 3)
1. Deploy to rootweaver namespace via FluxCD
2. Monitor for 1 week; track GPU utilization, job latencies
3. Tune thresholds (idle_duration, GPU_IDLE_THRESHOLD) based on metrics

## Risks & Mitigations

| Risk | Probability | Mitigation |
|------|-------------|-----------|
| **Dispatcher job fails, jobs never restart** | Medium | ScaledJob has retry policy; PDBs prevent deadlock |
| **GPU metrics are stale/incorrect** | Low | gpu_monitor.py validated in Layer 2 evidence; test on Day 1 |
| **Prometheus scrape fails** | Low | Alerts already deployed (monitoring/alertmanager); escalate manually |
| **Prefect jobs can't resume after suspension** | Low | PDB minAvailable=0; jobs re-queue automatically |
| **GPU never becomes "idle"** (sustained 5% util) | Medium | Tune idle_duration down to 60s or lower; monitor first |
| **Cascading eviction** (Prefect evicts other pods) | Low | Set `--reserve-vram` flags per pod; use resource limits |

## Alternatives Considered

### A1: Static Reservation (HAMi Only)
- **Pros**: Predictable, no eviction overhead
- **Cons**: Under-utilizes GPU if one workload is idle
- **Decision**: Use KEDA first; HAMi as future fallback for sustained overload

### A2: Time-Based Scheduling (CronJob)
- **Pros**: Simple, deterministic
- **Cons**: Rigid; breaks if workload patterns change
- **Decision**: Rejected; metric-driven is more flexible

### A3: Manual kubectl for Each Job
- **Pros**: Zero code
- **Cons**: Operator burden, unscalable
- **Decision**: Already failing; KEDA is the upgrade

### A4: Custom Webhook Mutating Admission Controller
- **Pros**: Fine-grained control
- **Cons**: 500+ LOC, harder to test, requires webhook deployment
- **Decision**: KEDA + ScaledJob is simpler and proven

---

## Success Criteria

1. **Latency**: GPU swap completes in <60 seconds (measured via Prometheus timestamp)
2. **Availability**: vLLM inference never OOMs; Prefect jobs resume without manual intervention
3. **Efficiency**: GPU utilization stays 70–95% during concurrent workloads (up from 0% idle + 100% OOM)
4. **Operational**: Zero human intervention required; alerts cover failure modes
