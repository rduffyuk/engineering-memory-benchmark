# ADR-012: Journal Pipeline Quality Overhaul — Model Selection & E2A Architecture

**Status**: Proposed
**Date**: 2026-02-08
**Jira**: VW-33 (to be created)
**Supersedes**: Partial aspects of ADR-010 (Streaming Journal System)

## Context

The journal streaming pipeline (VW-6, VW-30) automatically converts Claude Code sessions into daily journal entries. The pipeline's Kafka infrastructure is operational — OTEL events flow through topics, boundary detection triggers fire, and entries are written to the Obsidian vault at `01-Journal/Streaming/`. Mechanically, the system works. The problem is that the content it produces is fabricated.

### Evidence of Quality Failure

A manual audit cross-referencing JSONL session transcripts against OTEL-path journal entries for session `session_20260207_163955` revealed total grounding failure:

| Journal Claims | Actual Session Content (JSONL) |
|---|---|
| `deploy_watchdog.sh` JSON conversion | Writing Season 3 Episode 1 blog post |
| CI pipeline integration and Docker troubleshooting | Generating retro anime hero images via HuggingFace |
| Exit code debugging across 4 consecutive entries | Zero mentions of `deploy_watchdog.sh` anywhere in transcript |

Every OTEL-path entry was entirely fabricated. The voice and format were correct — the v8 prompt redesign (VW-30) produced plausible-sounding prose in the right register — but the substance was fiction. A journal entry that reads well but describes events that never happened is worse than no entry at all, because it creates false records.

### Root Cause Analysis

Four compounding factors produce this outcome:

**1. OTEL events are structurally too sparse (richness score: 2/10).** The events reaching the journal consumer carry only session-level metadata: `[event_name] Tool: X (decision) @ HH:MM:SS`. No file paths, no commands, no conversation context, no error messages. The boundary detector (`scripts/journal/streaming/synthesizer/journal_synthesizer.py`) triggers correctly on session boundaries, but the events it surfaces contain almost no semantic content. Even with 50 events, the LLM has nothing to ground its generation on. The sparsity is architectural, not a bug — OTEL was designed for observability telemetry, not content reconstruction.

**2. DeepSeek R1 8B is the wrong model for this task.** The current generation model (served via vLLM on the RTX 4080) is a reasoning-optimized distillation trained primarily for mathematical and logical tasks. Vectara's Hallucination Leaderboard measures the full DeepSeek R1 at a 14.3% hallucination rate — roughly 4x worse than DeepSeek V3. The 8B distillation performs worse still. When given a sparse prompt and asked to produce narrative text, R1 "overhelps" by injecting plausible-sounding technical details that are not present in the source material. The model is not defective; it is misapplied. A reasoning model optimized for chain-of-thought math does not transfer well to faithful summarization.

**3. Single-shot generation with no verification.** The pipeline currently fires one LLM call and writes the output to vault. There is no step that checks whether the claims in the generated entry are attested by the input. A hallucinated file path or a fabricated debugging scenario passes through unchallenged. Production content systems (news summarization, medical report generation, legal document drafting) universally use multi-stage pipelines with verification steps for exactly this reason.

**4. Temperature is too high.** Generation runs at 0.7, which is appropriate for creative writing but counterproductive for factual summarization. Research on summarization tasks consistently shows 0.3-0.5 as the optimal range — low enough for factual grounding, high enough to avoid repetitive phrasing. At 0.7, the model is encouraged to explore diverse completions, which in the absence of strong grounding signal means diverse fabrications.

### The Stakes

The journal pipeline is the first stage of an intended content cascade: Claude Code session -> journal entry -> weekly blog post -> LinkedIn/social media content. Every downstream artifact inherits upstream quality. Publishing fabricated technical claims under a real name, even on a personal blog, is a reputational risk. The pipeline must produce content that is verifiably grounded in what actually happened, or it must not produce content at all. The current output is not merely low-quality — it is actively harmful to the automation vision because it requires full manual rewriting, which defeats the purpose of automation.

### Existing Work

The codebase already contains a `verified_insight_generator.py` (v3 journal redesign) implementing ensemble hallucination detection based on SemEval-2025 Mu-SHROOM approaches (Semantic Entropy Probes, Context Sensitivity Ratio). This validates that hallucination detection is a recognized need in the project. However, detection after generation is less effective than prevention through architecture — the E2A approach proposed below prevents hallucination by construction rather than detecting it after the fact.

## Decision

We adopt a five-part overhaul: deprecate the OTEL synthesis path, replace the generation model, implement an Extract-then-Abstract (E2A) two-stage pipeline, add an LLM-as-Judge quality gate, and adjust inference parameters.

### 1. Deprecate OTEL Synthesis Path

The OTEL path is structurally incapable of producing grounded journal entries. The events lack semantic content, and no amount of prompt engineering can extract facts that are not present in the input.

**Action**: Set `MIN_OTEL_EVENTS = 999` in the synthesizer configuration to effectively disable OTEL synthesis without removing code. Add a deprecation log message. The OTEL boundary detector remains useful for triggering JSONL session processing (detecting session boundaries and milestones via time gaps and topic changes), but it should no longer attempt content synthesis from OTEL events alone.

JSONL becomes the sole synthesis input path. The JSONL transcripts contain the full conversation — tool invocations, file paths, error messages, reasoning — and provide the rich grounding signal that OTEL events lack.

### 2. Replace DeepSeek R1 8B with Qwen3-14B for Extraction

The extraction stage needs a model that is good at reading comprehension and structured extraction, not mathematical reasoning. Qwen3-14B is a general-purpose model with strong summarization capability.

| Property | DeepSeek R1 8B | Qwen3-14B |
|---|---|---|
| Hallucination rate (Vectara) | ~14%+ (full R1; distillation worse) | ~4-6% estimated (general-purpose family) |
| Writing quality for summarization | Poor (reasoning/math optimized) | Strong (general-purpose, instruction-tuned) |
| VRAM at Q4 quantization | ~5GB | ~9GB |
| Fits RTX 4080 (16GB) | Yes | Yes, with KV cache headroom |
| guided_json support via vLLM | Yes | Yes |

Qwen3-14B fits on the existing RTX 4080 GPU without requiring new hardware. The `VLLM_MODEL` environment variable in the streaming synthesizer (currently defaulting to `qwen3-8b` in `daily_aggregator.py` and `qwen3.5-9b` in `vllm_worker.py`) will be standardized to `qwen3-14b`.

### 3. Implement Extract-then-Abstract (E2A) Two-Stage Pipeline

This is the single most impactful architectural change. The E2A pattern (see arxiv.org/html/2410.06520v1) dramatically reduces hallucination by ensuring the synthesis model never sees raw noisy transcript. Instead, it works only from structured, verifiable facts.

```
Stage 1: EXTRACT (Local — Qwen3-14B on vLLM)
  Input:  Formatted JSONL session transcript
  Output: Structured JSON via vLLM guided_json
  Schema: ExtractedFacts {
    session_date, duration_estimate,
    primary_goals[],
    technical_decisions[{decision, reasoning}],
    files_modified[{path, change_type, description}],
    problems_encountered[{problem, solution, resolved}],
    tools_and_services[],
    key_learnings[],
    collaboration_highlights[]
  }
  Temperature: 0.3
  Purpose: Extract only facts present in transcript — no narrative, no embellishment

Stage 2: SYNTHESIZE (API — Claude Sonnet 4.5 via Batch API)
  Input:  Extracted facts JSON + style prompt + previous journal entry (for continuity)
  Output: Publication-ready journal entry (markdown)
  Temperature: 0.5
  Purpose: Turn verified facts into Ryan's voice and narrative style
```

**Why E2A prevents hallucination by construction**: The synthesis model (Claude Sonnet) never sees the raw transcript. It receives only the structured `ExtractedFacts` JSON from Stage 1. If a fact was not extracted — because it was not in the transcript — it cannot appear in the final output. The extraction schema (already implemented at `scripts/journal/streaming/synthesizer/extraction_schema.py`) uses vLLM's `guided_json` feature to constrain Stage 1 output to valid JSON matching the `EXTRACTION_JSON_SCHEMA`, preventing free-form hallucination in the extraction phase itself.

The `facts_to_prompt_context()` function in `extraction_schema.py` formats the structured facts into a readable prompt block for Stage 2, replacing the raw transcript entirely.

### 4. Add LLM-as-Judge Quality Gate

A third stage evaluates the synthesized entry against the extracted facts before publication. Research shows LLM-as-judge achieves approximately 80% agreement with human evaluators at roughly 98% lower cost than human annotation.

```
Stage 3: JUDGE (API — Claude Haiku 4.5)
  Input:  Draft journal entry + extracted facts JSON
  Output: Structured verdict via JUDGE_JSON_SCHEMA:
    - factual_grounding (0-10): Every claim traces to extracted facts
    - completeness (0-10): All significant session events covered
    - hallucination_detected (boolean): Any fabricated content found?
    - tone_style (0-10): Matches Ryan's voice and blog register
    - coherence (0-10): Logical narrative flow
  Threshold: All scores >= 7 AND hallucination_detected == false
  Actions:
    - Pass  -> auto-publish to vault (01-Journal/)
    - Marginal (scores 5-6, no hallucination) -> one revision attempt
    - Fail  -> flag for human review, do not publish
```

The judge schema is already defined at `scripts/journal/streaming/synthesizer/judge_schema.py` with the `JudgeVerdict` dataclass and `parse_judge_verdict()` parser. The threshold is configurable via `MIN_SCORE_AUTO_PUBLISH` (default 7) and `MIN_SCORE_REVISE` (default 5) environment variables.

The judge's `overall_pass` field is recomputed from actual scores rather than trusting the model's self-assessment — a defensive design choice visible in the existing `parse_judge_verdict()` implementation (line 148: "Recompute overall_pass from actual scores — don't trust model's self-assessment").

### 5. Adjust Inference Parameters

| Parameter | Current | Proposed | Rationale |
|---|---|---|---|
| Temperature | 0.7 | 0.3 (extraction) / 0.5 (synthesis) | Lower for factual accuracy; slightly higher for prose quality |
| Max tokens | 2048 | 1500 (extraction) / 2048 (synthesis) | Extraction should be concise structured JSON |
| Model (extraction) | deepseek-r1-0528-qwen3-8b | Qwen3-14B (local vLLM) | Better summarization, lower hallucination |
| Model (synthesis) | same single-shot model | Claude Sonnet 4.5 (Batch API) | Publication-quality prose from verified facts |
| Model (judge) | none | Claude Haiku 4.5 | Cheapest model adequate for binary quality assessment |

## Cost Analysis

### Estimated Monthly Cost (Hybrid Pipeline)

| Stage | Model | Volume/Month | Input Cost | Output Cost | Total |
|---|---|---|---|---|---|
| Extraction | Qwen3-14B (local) | ~30 sessions | $0 (GPU already allocated) | $0 | **$0** |
| Synthesis | Claude Sonnet 4.5 Batch | ~30 entries | ~60K tokens @ $1.50/M | ~45K tokens @ $7.50/M | **$0.43** |
| Judge | Claude Haiku 4.5 | ~30 entries | ~120K tokens @ $0.50/M | ~15K tokens @ $2.50/M | **$0.10** |
| Revision (~20% fail rate) | Claude Sonnet 4.5 Batch | ~6 entries | ~12K @ $1.50/M | ~9K @ $7.50/M | **$0.09** |
| Weekly blog synthesis | Claude Sonnet 4.5 Batch | 4 posts | ~48K @ $1.50/M | ~12K @ $7.50/M | **$0.16** |
| | | | | **Monthly total** | **~$0.78** |

Batch API pricing is 50% of standard API. At standard rates the total would be approximately $1.56/month. Both figures are negligible.

### Approach Comparison

| Approach | Monthly Cost | Estimated Quality (1-10) | Automation Level |
|---|---|---|---|
| Current (R1 8B, single-shot, OTEL) | $0 | 2/10 | Fully automated but output is fiction |
| Qwen3-14B only (local model upgrade, no E2A) | $0 | 5/10 | Better but still single-shot, needs editing |
| **Hybrid E2A (this proposal)** | **~$0.78** | **8/10** | **Auto-publish above threshold** |
| Full API (Claude Sonnet for all stages) | ~$2.41 | 9/10 | Simplest architecture, highest per-unit cost |

## Consequences

### Positive

- Journal entries are grounded in actual session content, not hallucinated scenarios. The fabricated `deploy_watchdog.sh` class of error is eliminated by construction.
- Publication-quality output for under $1/month enables the full content cascade: session -> journal -> blog -> LinkedIn.
- The LLM-as-Judge stage provides measurable quality metrics (factual grounding, completeness, hallucination rate) that can be tracked over time and used to detect regressions.
- The E2A pattern is composable. The same `ExtractedFacts` JSON that feeds journal synthesis can feed blog post generation, weekly summaries, LinkedIn posts, and brag document compilation without re-processing raw transcripts.
- Qwen3-14B fits on the existing RTX 4080 with headroom for KV cache. No new hardware required.
- The extraction schema (`extraction_schema.py`) and judge schema (`judge_schema.py`) already exist in the codebase, reducing implementation effort.

### Negative

- API dependency for synthesis and judge stages introduces a network requirement and dependence on Anthropic API availability. If the API is down, entries queue rather than publish.
- Pipeline complexity increases from 1 stage to 3 (extraction -> synthesis -> judge), with associated debugging surface area.
- Batch API latency can be up to 24 hours for batch jobs. If same-day publishing is required for a specific entry, the standard (non-batch) API must be used at 2x cost.
- Model download required: Qwen3-14B at Q4 quantization is approximately 9GB.

### Risks

- **Claude API pricing changes**: Currently negligible at $0.78/month. Would need a >100x price increase to become material, which is unlikely given competitive dynamics.
- **Qwen3-14B extraction quality is untested on actual JSONL transcripts**: Mitigation is to A/B test extraction quality against the existing R1 8B on 3-5 real transcripts before full cutover.
- **Batch API latency variance**: Anthropic states up to 24h for batch completion. For the journal use case (daily entries, not real-time), this is acceptable. The daily consolidator (`scripts/journal/streaming/daily_consolidator/daily_consolidator_claude.py`) already runs on a schedule.
- **LLM-as-Judge calibration**: The judge thresholds (>=7 on all dimensions) are initial estimates. They may need adjustment after observing real score distributions — too strict wastes revision cycles, too lenient lets marginal entries through.

## Alternatives Considered

### A. Keep DeepSeek R1 8B, Improve Prompts Only

Rejected. The model's architecture is fundamentally misaligned with faithful summarization. VW-30 already applied grounding instructions (v8/v9 prompt redesigns) — they improved voice and format but could not prevent content fabrication because the hallucination rate is a property of the model, not the prompt. A 14%+ baseline hallucination rate cannot be prompt-engineered to acceptable levels for auto-publication, especially when the input signal (OTEL events) is too sparse to ground on.

### B. Upgrade to a 70B+ Local Model

Rejected for now. Qwen 2.5 72B and Llama 3.3 70B would offer better extraction quality but require approximately 40GB VRAM at Q4 quantization. The RTX 4080 has 16GB. Running a 70B model would require either multi-GPU (not available) or a new GPU purchase. The hybrid approach (local 14B extraction + API synthesis) achieves publication quality at negligible cost without hardware changes. If Qwen3-14B extraction proves insufficient after testing, this alternative can be revisited — but it should be the fallback, not the first move.

### C. Fully API-Based Pipeline (All Stages via Claude)

Viable but unnecessary at this stage. Using Claude Sonnet for extraction would cost approximately $2.41/month versus $0.78 for the hybrid approach. The $1.63 difference is negligible in absolute terms, but the hybrid approach keeps the local GPU useful, avoids full API dependency, and exercises the vLLM infrastructure that serves other workloads. If operational simplicity becomes more important than cost (e.g., the local vLLM instance becomes unreliable), migrating extraction to the API is a straightforward configuration change — the `llm_backends.py` already abstracts the backend behind a common interface with fallback chains (vLLM -> Groq -> Gemini -> OpenRouter).

### D. Fine-Tune a Small Model on Ryan's Writing Style

Interesting but premature. Fine-tuning requires a curated dataset of high-quality journal entries written in the target voice. That dataset does not exist yet because the pipeline has not produced publication-quality output. After 3-6 months of E2A pipeline operation, the accumulated entries (reviewed and approved by the judge gate) would form a natural training set. At that point, a fine-tuned 7-8B model could potentially replace both the extraction and synthesis stages entirely, eliminating the API dependency. This is a future optimization, not a current solution.

## Implementation Plan

1. **Phase 1 — Model Swap** (1-2 hours): Download Qwen3-14B to vLLM, update `VLLM_MODEL` environment variable across streaming components, lower extraction temperature to 0.3, verify extraction quality on 3 real JSONL transcripts.

2. **Phase 2 — E2A Pipeline Wiring** (4-6 hours): Connect the existing extraction schema (`extraction_schema.py`) and synthesis stage into the journal synthesizer's JSONL processing path. Add Claude API client for the synthesis stage. Wire the `facts_to_prompt_context()` output into the synthesis prompt. Test end-to-end on real sessions.

3. **Phase 3 — Quality Gate** (2-3 hours): Wire the existing judge schema (`judge_schema.py`) into the pipeline after synthesis. Configure Haiku as the judge model. Implement the pass/revise/fail routing logic. Connect to Prefect flow for scheduling.

4. **Phase 4 — OTEL Deprecation** (30 minutes): Set `MIN_OTEL_EVENTS = 999`, add deprecation warning in logs. Verify the OTEL boundary detector still triggers JSONL processing correctly without attempting OTEL-path synthesis.

5. **Phase 5 — Blog and Social Pipeline** (4-6 hours, future): Weekly aggregation from daily entries, blog post synthesis via Claude Sonnet, LinkedIn post formatting, Prefect scheduling. This phase depends on Phase 1-4 producing reliable daily entries.
