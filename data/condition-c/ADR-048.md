# ADR-048: Egress Control Strategy — Forward Proxy Now, CNI Migration Later

**Status**: Proposed
**Decision date**: 2026-05-21
**Jira**: VW-409 (parent), VW-425 (Squid Phase 1), VW-426 (Cilium Phase 2)

---

## 1. Context

### 1.1 The triggering incident

During VW-409 memory-pipeline backfill work (2026-05-21), `memory-synthesize` Jobs needed to call OpenAI's `gpt-4o-mini` API to extract facts from 12 backlogged Claude Code sessions. The Job pods carry the `app=memory-writer` or `app=memory-backfill` label so they can share the existing `memory-writer-policy` NetworkPolicy for Postgres, Kafka, and FalkorDB access. The empirical finding was clear:

```
Before patch (no internet:443 rule):
  call 0: OK 1195ms          ← kube-router policy-enforcement race
  call 1: FAIL Connection refused
  call 2: FAIL Connection refused
  call 3: FAIL Connection refused
  call 4: FAIL Connection refused
```

The first-call success is a kube-router timing artefact: iptables rules take several hundred milliseconds to attach after pod start, creating a window where traffic passes unchecked. Every subsequent call hits the default-deny egress. This signature is repeatable and confirms the policy is effective once loaded.

The `memory-writer-policy` (`rootweaver-gitops/infrastructure/rootweaver/network-policies/memory-writer-policy.yaml`) permits egress only to:
- Kafka 9092 (journal.sessions consumer, DLQ producer)
- vault-postgres 5432 (memory schema writes)
- FalkorDB 6379 (graph fanout)
- vllm-embedding 8001 (in-cluster embedding)
- OTel/Tempo 4317 (tracing)
- kube-system DNS 53

There is no rule for internet:443 because the live `memory-writer` Deployment embeds via in-cluster vLLM and has never needed external API access. Only the synthesize step, which calls OpenAI for fact extraction, requires outbound HTTPS.

### 1.2 The temporary patch

To unblock the backfill, we added a temporary `0.0.0.0/0:443 except RFC1918` egress rule to `memory-writer-policy`. The backfill succeeded (12 sessions, 107 facts, 62,265 tokens, $0.023). But the patch widened the blast radius: **every** pod matching `app in (memory-writer, memory-backfill)` gained unrestricted internet:443 egress, including the live Deployment that does not need it.

### 1.3 What Kubernetes NetworkPolicy provides

Vanilla Kubernetes NetworkPolicy operates at OSI L3/L4 only: IP CIDR and port. **There is no native FQDN selector.** The official Kubernetes documentation confirms this. Any FQDN-aware egress enforcement requires infrastructure beyond the built-in API (a proxy, a CNI with DNS-aware policy, or a service mesh).

### 1.4 Why this matters beyond the immediate fix

The platform already has services with external API dependencies (rag-gateway calls Perplexity/Serper; Open WebUI may call upstream model providers). Future work under VW-402 and VW-394 will add more. Without a principled egress strategy, each new external integration either gets the wide-open `0.0.0.0/0:443` treatment or requires bespoke NetworkPolicy rules that cannot distinguish between `api.openai.com` and any other IP on port 443.

### 1.5 Cluster constraints

- **Single-node K3s homelab** on Ubuntu, 64 GB RAM, NVIDIA RTX 4080
- **CNI**: kube-router (flannel-mode), in use since cluster inception
- **GitOps**: FluxCD watches `rootweaver-gitops` repo, auto-reconciles
- **Operational posture**: solo operator, simplicity and auditability trump architectural perfection

---

## 2. Decision drivers

1. **Homelab scale, single operator** — operational simplicity is weighted heavily; a half-day CNI migration carries real downtime risk for a cluster that runs production-grade personal infrastructure
2. **Existing CNI is kube-router** — disrupting it has cascading implications for every NetworkPolicy in the cluster (15+ policies across 8 namespaces)
3. **Narrow use case today** — only a handful of LLM provider FQDNs need outbound access: `api.openai.com`, `api.anthropic.com`, `generativelanguage.googleapis.com`
4. **Auditability** — a single chokepoint where every external API call is logged (caller, destination, timestamp) enables both security review and cost attribution
5. **Future multi-tenant readiness** — VW-402/VW-394 may introduce more services with distinct external API needs; the pattern should scale without per-service bespoke rules
6. **Defense in depth** — the RFC1918-exclusion-only approach leaves a data-exfiltration vector open if any pod with the `memory-writer` label is compromised

---

## 3. Considered alternatives

### 3.1 Option A: Accept the wide-open rule (status quo via temporary patch)

Keep `0.0.0.0/0:443 except RFC1918` permanently on `memory-writer-policy`.

- **Pros**: Zero work; already in place
- **Cons**: Permanently widens blast radius for the live `memory-writer` Deployment, which embeds via in-cluster vLLM and never needs internet access. No audit trail of external API calls. Any future pod sharing the `app=memory-writer` label inherits unrestricted outbound HTTPS. Violates principle of least privilege.
- **Verdict**: Acceptable as a 1-week bridge; rejected as the durable answer

### 3.2 Option B: Squid forward proxy with FQDN allowlist (chosen for Phase 1)

Deploy a single Squid pod in a dedicated `egress-proxy` namespace. Squid's configuration is an explicit FQDN allowlist loaded from a ConfigMap. The Squid pod is the **only** pod in the cluster with broad `0.0.0.0/0:443` egress. All other pods that need external HTTPS set `HTTPS_PROXY=http://squid.egress-proxy.svc.cluster.local:3128` and route through the proxy. Squid logs every CONNECT request, providing a complete audit trail.

- **Pros**: Single chokepoint, fully auditable via `kubectl -n egress-proxy logs deployment/squid`. ~80 lines of new YAML; ~30 MB image. Does not touch existing CNI. Allowlist managed in ConfigMap, extensible via gitops commit + pod restart. httpx, urllib, and requests all honour `HTTPS_PROXY` natively (zero application code changes).
- **Cons**: Single point of failure for all external API calls (acceptable at homelab scale). Compromised proxy = total egress takeover (low likelihood in a single-tenant homelab). Pods must explicitly opt in via env var; a missed `HTTPS_PROXY` setting results in silent connection failure. Non-HTTPS external use cases need separate plumbing.

### 3.3 Option C: Cilium CNI replacement with FQDN-native policy (chosen for Phase 2)

Replace kube-router with Cilium. Cilium intercepts DNS responses and maps resolved IPs to FQDN labels, enabling policies like `egress: [{toFQDNs: [{matchName: "api.openai.com"}]}]`. No proxy, no `HTTPS_PROXY` env var, transparent enforcement.

- **Pros**: Most elegant — transparent, kernel-level (eBPF). Catches IP-direct bypass (a compromised pod cannot skip DNS to reach arbitrary IPs). Unlocks L7-aware policies (HTTP method/path restrictions), eBPF observability, Gateway API, and service-mesh-lite capabilities.
- **Cons**: CNI migration is a half-day operation with real downtime risk. Requires draining workloads, swapping CNI, and revalidating every NetworkPolicy. Cilium agent uses ~200 MB RAM per node. Some kube-router behaviours (egress NAT, certain NetworkPolicy edge cases) may change. Migration plan must include etcd snapshot and rollback path.

### 3.4 Option D: Monzo egress-operator (Envoy-based, CRD-driven)

Same proxy concept as Squid but uses Envoy sidecars and a custom operator that creates per-FQDN Deployment+Service pairs automatically from CRD definitions. Requires a custom CoreDNS image rebuild with the operator's DNS plugin.

- **Verdict**: Good fit for a 50-engineer team with dedicated platform engineers. Over-engineered for a solo homelab with 3 allowlisted FQDNs.

### 3.5 Option E: Istio service mesh egress gateway

Full sidecar-injected mesh with VirtualService FQDN routing via a dedicated egress gateway pod.

- **Verdict**: ~100 MB/pod overhead for sidecar injection, a full Istiod control plane, and significant operational complexity. Massive overkill for this need.

## 4. Decision

**Two-phase migration, starting with the lowest-risk, highest-auditability option and graduating to CNI-native enforcement when stability is proven and the FQDN list grows.**

### Phase 1 — Squid forward proxy (now, VW-425)

Deploy the Squid forward proxy as a dedicated chokepoint. The implementation consists of six manifests in `rootweaver-gitops/infrastructure/egress-proxy/`:

| File | Purpose |
|------|---------|
| `namespace.yaml` | Dedicated `egress-proxy` namespace with ADR-048 annotation |
| `squid-configmap.yaml` | Squid configuration + FQDN allowlist (`api.openai.com`, `api.anthropic.com`, `generativelanguage.googleapis.com`) |
| `squid-deployment.yaml` | Single-replica Squid pod (`ubuntu/squid:5.2-22.04_beta`, 64Mi-256Mi, 50m-500m) |
| `squid-service.yaml` | ClusterIP service on port 3128 |
| `squid-networkpolicy.yaml` | Ingress from `rootweaver` namespace on 3128; egress to `0.0.0.0/0:443 except RFC1918` + kube-system DNS |
| `kustomization.yaml` | Kustomize resource list with operational comments |

The FluxCD Kustomization at `rootweaver-gitops/clusters/homelab/egress-proxy.yaml` watches `./infrastructure/egress-proxy` for auto-reconciliation.

On the consumer side:
1. **Revert** the temporary `0.0.0.0/0:443` rule from `memory-writer-policy`
2. **Add** a narrow egress rule to `memory-writer-policy` permitting traffic to `squid.egress-proxy:3128` only, scoped to the `egress-proxy` namespace and `app: squid` pod selector (see `rootweaver-gitops/infrastructure/rootweaver/network-policies/memory-writer-policy.yaml`, lines 84-98)
3. **Set** `HTTPS_PROXY`, `HTTP_PROXY`, and `NO_PROXY` env vars on the backfill Job template (`rootweaver-gitops/infrastructure/memory-backfill/manual-job-template.yaml`, lines 76-85). `NO_PROXY` includes `.cluster.local`, `.svc`, RFC1918 ranges, and localhost to keep all in-cluster traffic direct
4. The live `memory-writer` Deployment does **not** get proxy env vars — it has no external API need and should remain unable to reach the internet

### Phase 2 — Cilium CNI migration (long-term, target Q3 2026, VW-426)

Migrate from kube-router to Cilium after Phase 1 has been stable for at least 30 days. Drivers for promoting Phase 2:
- FQDN allowlist grows past ~10 entries (ConfigMap sprawl)
- A second cluster node is added (Cilium scales better than a single-proxy bottleneck)
- L7-aware policies are needed (HTTP method/path restrictions)
- eBPF observability becomes desirable for broader platform work

Migration plan: snapshot etcd, drain workloads, swap CNI, restore, validate every NetworkPolicy, confirm Squid is redundant, retire it.

## 5. Consequences

### 5.1 Positive

- **Least-privilege restored**: The live `memory-writer` Deployment loses the internet:443 privilege it never needed. Only pods that explicitly opt in via `HTTPS_PROXY` can reach external APIs, and only through the Squid chokepoint.
- **Single audit point**: Every external API call is logged in Squid's access log with timestamp, source IP, destination FQDN, HTTP method, and response size. `kubectl -n egress-proxy logs deployment/squid` provides a complete external-call audit trail. This log could be shipped to Loki in a future iteration.
- **Low operational cost for new integrations**: Adding a new external API endpoint requires one line in `squid-configmap.yaml` (the `allowlist.txt` section), a gitops commit, and `kubectl -n egress-proxy rollout restart deployment/squid`. No NetworkPolicy changes needed.
- **No CNI disruption**: Phase 1 runs entirely within the existing kube-router CNI using standard Kubernetes NetworkPolicy primitives. No cluster-wide risk.
- **Minimal resource footprint**: Squid runs at 64Mi request / 256Mi limit, 50m / 500m CPU. The `ubuntu/squid:5.2-22.04_beta` image is ~30 MB compressed.
- **Transparent to application code**: httpx, urllib3, and the `requests` library all honour the `HTTPS_PROXY` environment variable without code changes. The OpenAI Python SDK, which uses httpx internally, routes through Squid automatically.

### 5.2 Negative

- **Single point of failure**: If the Squid pod is down, all external API calls from proxy-configured pods fail. At homelab scale with a single operator this is acceptable — the pod auto-restarts via K8s and the use case is batch Jobs, not latency-critical user-facing traffic.
- **Extra network hop**: Adds ~1 ms latency per external call. Negligible against typical LLM API call latency (800-1600 ms observed in VW-409 backfill).
- **Opt-in discipline required**: Developers must remember to set `HTTPS_PROXY` + `NO_PROXY` on any new pod that needs external API access. A missed `HTTPS_PROXY` results in silent connection refusal (the NetworkPolicy blocks direct internet:443). This is by design — it makes internet access an explicit, auditable decision — but it is a footgun for someone unfamiliar with the pattern.
- **Allowlist drift**: Over time, FQDNs may be added to `allowlist.txt` without corresponding removal of no-longer-needed entries. Mitigation: periodic review of Squid access logs to identify unused allowlist entries.
- **CONNECT-only enforcement**: Squid sees the FQDN during the CONNECT handshake but cannot inspect the encrypted payload. A malicious actor who controls a pod could tunnel arbitrary data through an allowlisted FQDN. This is a known limitation of any L4 proxy; Cilium Phase 2 does not solve it either (it also operates at the DNS/IP level, not within TLS).

### 5.3 Risks and mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Squid pod crashes during active backfill | Low | Medium — backfill retries | Liveness probe restarts pod; Job has per-session retries |
| Allowlist too narrow for new provider | Low | Low — gitops fix is <5 min | Proactively added Anthropic + Gemini alongside OpenAI |
| `NO_PROXY` misconfigured, in-cluster traffic routes through Squid | Low | Medium — Squid denies it | Template includes comprehensive `NO_PROXY` covering `.cluster.local`, `.svc`, RFC1918 |
| Squid image supply-chain compromise | Very low | High — proxy sees all external calls | Pin to specific tag; image is Canonical-maintained Ubuntu LTS |

## 6. Acceptance criteria

### Phase 1 (VW-425)

- `egress-proxy` namespace + Squid Deployment + Service + ConfigMap deployed via gitops
- Squid allowlist contains: `api.openai.com`, `api.anthropic.com`, `generativelanguage.googleapis.com`
- `memory-writer-policy` has no `0.0.0.0/0:443` rule
- `memory-writer-policy` has egress to `squid.egress-proxy:3128` scoped to `app: squid` pod selector
- Synthesize Job template sets `HTTPS_PROXY`, `HTTP_PROXY`, and `NO_PROXY`
- Smoke test: `memory-synthesize` Job produces facts identically via Squid to the direct-path baseline
- Squid access log captures each external API call with timestamp, source, and destination FQDN

### Phase 2 (VW-426)

- Migration plan written as a vault document
- Cilium validated in a separate test namespace or lab cluster
- Production cutover executed with etcd snapshot rollback path
- Every existing NetworkPolicy validated under Cilium semantics
- Squid retired after Cilium is stable for 30 days
