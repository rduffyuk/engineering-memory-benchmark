# ADR-043: Memory System Derived-Data GDPR Position

## Status

Proposed

## Context

### The Memory Data Pipeline

The Rootweaver long-term memory system (VW-391) ingests Claude Code session JSONL transcripts via a Kafka consumer (`rootweaver_memory.consumer`) and produces multiple layers of derived data across two storage backends:

1. **`memory.sessions`** — session-level metadata: timestamps, branch, Jira refs, commits, a summary embedding (pgvector `VECTOR(1024)`), and a `tombstoned_at` column used for soft-delete grace periods. Primary key is the session UUID.

2. **`memory.rounds`** — per-turn verbatim user/assistant text, tool call JSONB, file-touch arrays, and a per-round embedding. Foreign key to `sessions(id)` with `ON DELETE CASCADE`. The `assistant_thinking` column stores redacted reasoning traces (see layer 7).

3. **`memory.facts`** — LLM-extracted structured knowledge (decisions, problems, learnings, goals, file changes, tool uses, collaborations). Each fact carries `text`, `reasoning`, entity tags, a pgvector embedding, and bi-temporal validity columns (`t_valid`/`t_invalid`). Foreign keys to both `rounds(id)` and `sessions(id)`, both `ON DELETE CASCADE`. (`packages/rootweaver-memory/src/rootweaver_memory/tables.py`, lines 87-130.)

4. **`memory.artifacts`** — references to external objects produced during sessions: git commits, diff hunks, vault docs, Confluence pages, Jira issues/comments/transitions, PRs, file creates/modifies/deletes, K8s applies. Foreign keys cascade from `rounds`, `facts`, and `sessions`. (`tables.py`, lines 164-231.)

5. **`memory.fact_corrections`** — negation records (do_not_do, is_not_true, superseded_by, misunderstood) linked to parent facts via `ON DELETE CASCADE`. (`tables.py`, lines 132-162.)

6. **`memory.file_snapshots`** — point-in-time captures of file state during sessions. Cascades from `sessions`.

7. **`assistant_thinking` column on `memory.rounds`** — raw extended-thinking content from Claude, scrubbed at INSERT time by `rootweaver_memory.redactor.Redactor` (regex pass for API keys, bearer tokens, emails, UK/international phones, credit card numbers; optional spaCy NER pass for PERSON/ORG entities). This column is explicitly NEVER embedded — documented in `models.py` line 69 and `tables.py` line 69 as an industry-convergence decision per the VW-391 spec.

8. **FalkorDB `rootweaver_memory` graph** — Session, Round, Fact, Artifact, and Entity nodes with CONTAINS, GENERATED, PRODUCED, MENTIONS, and CO_OCCURS edges. Written asynchronously by `FalkorDBFanout` after the Postgres transaction commits. Failures route to a `memory-writer.graph-retry` Kafka topic for weekly reconciliation. (`falkordb_fanout.py`; schema in `schema/falkordb_bootstrap.cypher`.)

9. **pgvector embeddings** — `VECTOR(1024)` columns co-located on `sessions.summary_embedding`, `rounds.embedding`, and `facts.embedding`. These are generated at write time by `EmbeddingClient` (via vLLM) and are fail-soft (NULL if vLLM is unreachable).

### The GDPR Question

When a data subject invokes the right to erasure under GDPR Article 17, which of these layers must be deleted? The answer depends on whether derived and transformed data qualifies as "personal data."

### Regulatory Landscape (as of May 2026)

**Hamburg DPA discussion paper (July 2024)**: Argued that LLM model parameters are not personal data because individual data points cannot be meaningfully extracted from them. This position was narrowly scoped to model weights — not to structured derived stores like our facts table — but has been cited broadly by the permissive-interpretation camp.

**EDPB Opinion 28/2024 (December 2024)**: The European Data Protection Board rejected the Hamburg position's broad implications. EDPB held that if personal data can be extracted from a system — regardless of the transformation applied — the data remains personal data subject to Article 17. This is the currently authoritative EU-wide interpretation.

**EDPB CEF 2025**: The Coordinated Enforcement Framework specifically targeting right-to-erasure compliance across all EU regulators launched in early 2025, with a report expected February 2026. This signals that erasure compliance is an active enforcement priority, not a theoretical concern.

**OWASP AISVS C08**: The AI Security Verification Standard explicitly mandates hard-delete capability plus tombstoning for right-to-be-forgotten (RTBF) in vector stores and memory systems. Soft-delete alone is explicitly non-compliant. This standard is increasingly referenced in enterprise procurement and security audits.

### Why This Matters for Rootweaver

Ryan personally is not subject to GDPR for his own private data on his own infrastructure — there are no third-party data subjects. However, the project is planned for public release. Downstream deployers may operate it in multi-user or service-with-customers contexts where GDPR applies in full. Architectural decisions made now determine whether GDPR compliance is achievable without a rewrite. The conservative position costs little in implementation effort (the cascade infrastructure already exists in the schema) and avoids a class of regulatory and reputational risk for every future deployer.

### The Core Tension

The data layers form a derivation chain: JSONL → rounds → facts → embeddings → graph nodes. Under the Hamburg position, deleting the source rounds might suffice; the derived facts and embeddings could arguably persist as non-personal transformed data. Under the EDPB position, a fact reading "Ryan decided to migrate from ChromaDB to Qdrant on 2026-02-10" is personal data regardless of the extraction step that produced it — the name, decision, and date are extractable. Similarly, the 1024-dimensional embedding of that fact encodes its semantic content and may be approximately invertible via nearest-neighbour reconstruction attacks (Song & Raghunathan 2020; Carlini et al. 2021).

Our FalkorDB graph makes this even more concrete: a `(:Entity {name: "Ryan Duffy"})` node with `(:Fact)-[:MENTIONS]->(:Entity)` edges is not a diffuse statistical representation — it is a structured record that plainly constitutes personal data under any interpretation.

## Decision

**We adopt EDPB Opinion 28/2024 as the governing interpretation and cascade-delete all derived data on session erasure.**

### 1. ON DELETE CASCADE from sessions

The existing schema already implements this. Every table in the `memory` schema (`rounds`, `facts`, `fact_corrections`, `artifacts`, `file_snapshots`) has foreign keys to `memory.sessions(id)` with `ON DELETE CASCADE`. A single `DELETE FROM memory.sessions WHERE id = $1` removes the complete derived graph in one Postgres transaction. This was a deliberate design choice validated during the VW-391 schema design — see the migration at `schema/migrations/versions/0001_initial_schema.py`, lines 119-300, where every `REFERENCES memory.sessions(id)` carries `ON DELETE CASCADE`.

### 2. pgvector Embeddings Deleted Alongside Parent Rows

Because pgvector columns (`VECTOR(1024)`) live ON the same tables as the source text — `sessions.summary_embedding`, `rounds.embedding`, `facts.embedding` — the cascade is automatic. We deliberately did NOT split embeddings into sibling vector tables. This avoids the class of bugs where langchain+pgvector deployments retain orphaned embeddings after source deletion (cf. langchain discussion #17499). The AGENTS.md for rootweaver-memory explicitly documents this invariant: "pgvector columns are co-located on row tables (cascade-delete safety) — do NOT split into sibling vector tables."

### 3. FalkorDB Graph Reconciliation

FalkorDB nodes and edges are reconstructed from Postgres source-of-truth. After Postgres cascade-deletes fire, the weekly `memory-graph-reconcile` CronJob (draining the `memory-writer.graph-retry` topic) removes corresponding FalkorDB nodes. Nodes that aggregate information across multiple sessions (e.g., an Entity node for "Qdrant" referenced by many sessions) retain their edges from surviving sessions; only edges tracing to the deleted session are removed. If all provenance for a node is deleted, the node itself is deleted.

### 4. 7-Day Grace Tombstone

Before hard-delete fires, `sessions.tombstoned_at` is set to the current timestamp. A cron job (`memory-tombstone-reaper`) runs `DELETE FROM memory.sessions WHERE tombstoned_at < now() - interval '7 days'`, triggering the cascade. All MCP tools (`recall_decisions`, `find_problems`, `recall_session`, `recall_state_at`, `recall_origin`) apply an implicit `WHERE s.tombstoned_at IS NULL` filter (verified in `memory_tools.py`), so tombstoned sessions are functionally invisible from the moment of marking.

### 5. --immediate Flag for Deadline Compliance

An `--immediate` flag on the `memory-forget` operator script bypasses the tombstone grace period and fires `DELETE` directly. GDPR Article 12 allows one calendar month for response, but some operators may need faster execution.

### 6. Redaction at Insert for assistant_thinking

The `Redactor` class (`redactor.py`) scrubs `assistant_thinking` content at INSERT time — before the data touches persistent storage. This addresses the documented 19-78% sensitive-value leakage rate in chain-of-thought traces (Leaky Thoughts, arXiv:2506.15674). The regex layer catches API keys (OpenAI sk-, Anthropic sk-ant-, AWS AKIA, GitHub PAT ghp_/gho_/ghs_/ghr_/ghu_), bearer tokens, emails, credit-card-shaped numbers, UK and E.164 phone numbers. The optional spaCy NER layer scrubs PERSON and ORG entities. Insert-time redaction is permanent and non-reversible — this is intentional.

### 7. Source JSONL Files Are NOT Auto-Deleted

Source JSONL files under `conversation-archives/raw-jsonl/` are user-owned filesystem artifacts. `memory-forget` does not delete them. This is documented in the operator handbook as a conscious separation of concerns: the memory system governs its own derived stores; JSONL retention is an explicit user/operator decision.

## Consequences

### Positive

- **Defensible against regulator inquiry.** Aligned with EDPB Opinion 28/2024 (current most-authoritative interpretation) and OWASP AISVS C08 (industry standard control framework). The CEF 2025 enforcement wave targets exactly this area; the system is ready.
- **Public release is GDPR-safe by default.** Downstream deployers do not have to revisit the retention strategy to comply. The strict position is the shipped default.
- **Single-command erasure.** `memory-forget --session-id <uuid>` sets the tombstone; the reaper job completes the cascade. `--immediate` bypasses the grace period. No multi-system coordination required — the `memory` schema namespace means one Postgres DELETE sweeps everything.
- **No orphaned vectors.** Co-locating embeddings on row tables guarantees that cascade-delete removes them. This is architecturally enforced, not dependent on application-layer cleanup.
- **Co-location simplifies erasure rather than complicating it.** The initial intuition that separate databases provide cleaner GDPR isolation is wrong in practice: federated stores require coordinating multi-system DELETEs. A single `DELETE FROM memory.sessions WHERE id = ?` against a co-located schema is one transaction with deterministic cascade. The 0001 migration's `DROP SCHEMA IF EXISTS memory CASCADE` downgrade path further validates this — one statement cleans up every table, index, and ENUM.
- **Insert-time redaction removes the most dangerous content class.** Reasoning traces cannot leak what was scrubbed before persistence.

### Negative

- **Cascade cost at scale.** A session with hundreds of rounds, each with facts and artifacts, triggers a large cascading DELETE including pgvector index updates. At current single-user scale this is negligible. At production scale (millions of facts), partition-aware or batched deletion strategies may be needed. Out of scope for v1.
- **Implicit tombstone filter on every query.** All MCP tools and dashboard queries must include `WHERE tombstoned_at IS NULL`. Mitigated by centralising query logic in `memory_tools.py` rather than exposing raw SQL to callers, but raw `psql` operators must remember.
- **Re-ingestion produces new UUID.** A deleted session that is later re-ingested from its JSONL gets a new `sessions.id`. Forensic continuity from old git-commit trailers (VW-393) to the new ingestion is broken by design — correct for GDPR (erasure is meant to be permanent), but a forensic-recovery cost.
- **Source JSONL not auto-deleted.** Some GDPR interpretations would require deletion of the original record too. We surface this clearly as an operator decision rather than automating it, because JSONL files may be backed up, replicated, or governed by separate retention policies.
- **Entity-level cascade not in v1.** "Forget everyone named X across all sessions" requires an entity-tag index that does not yet exist. Per-session deletion is the v1 scope.

## Alternatives Considered

### Alternative 1: Hamburg Position — Retain Derived Facts and Embeddings

Under Hamburg DPA's July 2024 interpretation, only verbatim rounds and directly-identifying facts would require deletion. Embeddings and graph nodes, as transformed representations, would be retained on the grounds that individual source text cannot be reliably extracted from them.

**Why rejected.** (a) The EDPB explicitly repudiated this interpretation in Opinion 28/2024, and the CEF 2025 enforcement framework targets erasure compliance. Building on a contested-and-losing regulatory position creates legal risk for downstream deployers. (b) Our derived stores are not analogous to model weights. A FalkorDB `(:Entity {name: "Ryan Duffy"})` node is a structured record, not a diffuse statistical representation. A `memory.facts` row reading "Ryan decided to use HAMi for GPU sharing" is plainly personal data regardless of the LLM extraction step that produced it. (c) Retaining embeddings of deleted text serves no retrieval purpose — the source row they index is gone.

### Alternative 2: Anonymise Rather Than Delete

Replace personal identifiers in facts, graph nodes, and artifact metadata with pseudonyms (e.g., "User_001") rather than deleting records. This preserves knowledge graph structure and embedding space while removing identifying information.

**Why rejected.** (a) Anonymisation of free-text is unreliable. NER-based scrubbing misses oblique references, contextual identifiers, and rare names at rates that vary by language and domain. Our facts contain highly specific technical context ("Ryan decided to use HAMi for GPU sharing on the K3s cluster") where removing "Ryan" still leaves a quasi-identifier given the project's uniqueness. (b) GDPR Article 17 grants a right to erasure, not anonymisation. While anonymised data falls outside GDPR scope, the anonymisation must be provably complete — a bar difficult to meet for unstructured text containing domain-specific context. (c) Embeddings of anonymised text still encode semantic proximity to the original, requiring re-embedding to be consistent — at which point the cost advantage over deletion vanishes.

### Alternative 3: Soft-Delete Only (No Hard-Delete Path)

Mark records as deleted via a `deleted_at` timestamp without physically removing them. Enforce access controls so that deleted records are excluded from all query paths.

**Why rejected.** Soft-delete alone is explicitly non-compliant per OWASP AISVS C08 and the Brandur Leach general argument ("Soft Deletion Probably Isn't Worth It"). The data continues to exist and remains accessible to database administrators, backup systems, and any future code path that forgets the filter clause. We do use a tombstone as a 7-day grace period before hard-delete — this is a temporal buffer against operator error, not a substitute for physical deletion.
