# ADR-034: Data Source Connector Architecture

## Status

Proposed

## Context

Rootweaver's knowledge base contains approximately 148,000 chunks in Qdrant, but these originate from a single data source: the Obsidian vault. The vault indexer (`neural-vault/vault_indexer.py`) watches the filesystem, chunks Markdown files, embeds them via vLLM, and upserts to Qdrant. This pipeline is mature and battle-tested but fundamentally limited to one input channel.

Meanwhile, organisational knowledge is fragmented across systems that are accessible but not indexed:

- **Jira** (VW project, ~400+ issues) is queryable only via the Atlassian MCP tools (`mcp__atlassian__jira_search`). Issue descriptions, comments, and resolution context are invisible to semantic search.
- **Confluence** (VW space, ~50 pages) is similarly MCP-only. Architecture decisions, runbooks, and post-incident reviews cannot be retrieved by the RAG agents.
- **GitLab** (3 repositories) contains commit messages, merge request discussions, and CI logs that encode design rationale not captured elsewhere.
- **Slack**, **Discord**, **Gmail/IMAP**, and **web pages** each hold knowledge that could enrich cross-source queries.

The consequence is that cross-source queries are structurally blocked. A question like "What was the root cause of the Kafka consumer crash-loop and what Jira issues tracked it?" requires the human to manually bridge vault search results with Jira search results. The RAG agents (FastSearch, DeepResearch, Oracle) operate on a single Qdrant collection and cannot synthesise across source boundaries because non-vault data never enters that collection.

Enterprise knowledge platforms have solved this problem at scale. Glean indexes over 100 SaaS connectors. Onyx (formerly Danswer), an open-source alternative, provides 40+ connectors through a well-defined `BaseConnector` abstraction with polling-based incremental sync, normalised `Document` output, and checkpoint-based resumption. Rootweaver's codebase already contains an Onyx-derived shim layer (`packages/rootweaver-connectors/src/rootweaver_connectors/onyx_shim/`) that reproduces the core Onyx interfaces, and nine source-specific connectors have been ported: Jira, Confluence, GitLab, Discord, Slack, Gmail, IMAP, file, and web.

The missing piece is not individual connector code but the end-to-end architecture that orchestrates connectors, normalises their output, routes documents through a shared ingestion pipeline, and lands them in Qdrant alongside vault content in a way that preserves source provenance and supports incremental sync.

## Decision

We adopt a connector architecture modelled on the Onyx `BaseConnector` pattern, with three key adaptations for Rootweaver's single-tenant, Kafka-native infrastructure. The design has five layers.

### Layer 1: Connector Abstraction (Onyx Shim)

Each data source is represented by a class that extends one of the Onyx connector interfaces defined in `packages/rootweaver-connectors/src/rootweaver_connectors/onyx_shim/onyx/connectors/interfaces.py`:

- **`PollConnector`** — incremental sync via `poll_source(start, end)`. Used for sources with reliable update timestamps (Jira, Confluence, GitLab).
- **`LoadConnector`** — full-load via `load_from_state()`. Used for sources without incremental capability (static file sets, web crawls).
- **`CheckpointedConnector`** — resumable sync via `load_from_checkpoint(start, end, checkpoint)`. Used for large sources where a single poll window may exceed memory or API rate limits (Jira Cloud v3 with enhanced search).
- **`EventConnector`** — push-based via `handle_event(event)`. Reserved for future webhook-driven sources.

All connectors yield Onyx `Document` objects (defined in `onyx_shim/onyx/connectors/models.py`), which carry structured sections (`TextSection`, `ImageSection`, `TabularSection`), metadata dictionaries, ownership information (`BasicExpertInfo`), and a `DocumentSource` enum tag. This normalised output means downstream layers never interact with source-specific data shapes.

The connector registry (`packages/rootweaver-connectors/src/rootweaver_connectors/factory.py`) maps source names to `ConnectorMapping` entries that bind a module path, class name, default config, and credentials environment variable. Nine sources are registered: `jira`, `confluence`, `gitlab`, `discord`, `slack`, `gmail`, `imap`, `web`, and `file`. Adding a new source requires one registry entry and one connector class.

### Layer 2: Bridge (Onyx-to-Kafka Translation)

The `OnyxBridge` class (`packages/rootweaver-connectors/src/rootweaver_connectors/bridge.py`) is the adapter between connector output and the Kafka ingestion pipeline. It accepts any connector instance, drives it through its sync method (checkpoint-based or poll-based), and for each yielded `Document`:

1. Converts it to a `ConnectorDoc` dataclass (`packages/rootweaver-connectors/src/rootweaver_connectors/models.py`) — a flat, JSON-serialisable representation with `source`, `source_id`, `title`, `content`, `url`, `author`, `created_at`, `updated_at`, and `metadata`.
2. Produces the serialised `ConnectorDoc` to the `vault.connector-docs` Kafka topic, keyed by `{source}:{source_id}` to enable log compaction.

`ConnectorDoc` is intentionally simpler than Onyx's `Document`. It strips sections into plain text, collapses ownership to a single author string, and flattens metadata to `dict[str, Any]`. This normalisation boundary ensures the downstream indexer never depends on Onyx internals, making it possible to replace or upgrade the shim without touching the consumer.

The bridge also tracks `ConnectorFailure` items emitted by the connector, logging them without halting the sync. `HierarchyNode` items are dropped (no consumer exists for hierarchy data yet). The bridge calls `producer.flush()` in a `finally` block to prevent message loss on mid-loop exceptions.

### Layer 3: Watermark-Based Scheduling

Each connector runs on a schedule (currently a K8s CronJob every 15 minutes for Jira). The scheduler (`packages/operator-tools/src/operator_tools/connector_jira_sync.py`) implements a watermark pattern:

1. Read the `last_watermark` for `(source, scope)` from the `connector_sync_state` Postgres table.
2. Compute the poll window as `[last_watermark - overlap_minutes, now()]`.
3. Drive `bridge.checkpoint_sync(start_ts, end_ts, connector)`.
4. On success, upsert the new watermark (the maximum `doc_updated_at` seen) and `status=ok`.
5. On failure, upsert `status=error` with the previous watermark unchanged, so the next run retries the same window.

The overlap parameter (default 2 minutes) handles clock skew and eventual consistency in source APIs. Postgres is the state store because it already hosts MCP memory tables and provides transactional guarantees; a lightweight upsert on a single-row table adds negligible load.

### Layer 4: Kafka Consumer and Qdrant Indexing

The `ConnectorIndexer` (`packages/rootweaver-connectors/src/rootweaver_connectors/indexer.py`) consumes `ConnectorDoc` messages from Kafka, chunks them (750 tokens with 75-token overlap), embeds via vLLM HTTP endpoint, and upserts to Qdrant through the `QdrantAdapter`. Point IDs are deterministic UUID5 hashes of `{source}/{source_id}/{chunk_index}`, making upserts idempotent — re-processing the same document overwrites the same points.

The consumer runner (`packages/rootweaver-connectors/src/rootweaver_connectors/indexer_runner.py`) implements production-grade Kafka consumption:

- **Manual offset commit** after successful processing (no auto-commit).
- **Retry with exponential backoff** (default 3 attempts, 1s base delay) for transient failures (Qdrant timeouts, vLLM errors).
- **Dead-letter queue** (`vault.connector-docs.dlq`) for terminal failures, with base64-encoded original message for replay.
- **Graceful shutdown** on SIGTERM via threading.Event, completing the current message before exiting.
- **Dual-write to Qdrant v1 and v2 collections** via `QdrantAdapter` (VW-235), ensuring connector data is visible to the AUTO/FAST/DEEP read path that uses v2's named vectors and BM25 sparse index.

Connector-sourced chunks land in the same `rootweaver_vault` collection as vault content, distinguished by `metadata.source` (e.g., `"jira"`, `"confluence"`) and `metadata.type = "connector"`. This means existing RAG agents query both vault and connector data in a single Qdrant call with no code changes.

### Layer 5: Source Provenance and Cross-Source Queries

Each chunk carries provenance metadata:

- `source`: the connector name (`jira`, `confluence`, `gitlab`, etc.)
- `source_id`: the source-native identifier (Jira issue URL, Confluence page ID, etc.)
- `source_url`: a clickable link back to the original item
- `vault_section`: set to the source name (enabling existing vault-section filters to work)
- `date_created`, `timestamp`, `year`, `month`, `day`: temporal metadata for date-filtered search

This metadata enables the RAG agents to:

1. **Filter by source** — a query can restrict results to `source=jira` for project management queries.
2. **Attribute results** — every search result carries a link to the original document in its native system.
3. **Merge across sources** — a cross-source query returns vault notes, Jira tickets, and Confluence pages ranked together by semantic similarity.

## Consequences

### Positive

- **Cross-source search is unblocked.** Queries that span vault, Jira, Confluence, and other sources return unified results from a single Qdrant call.
- **Incremental sync minimises API load.** Watermark-based polling fetches only documents updated since the last run, keeping API calls within free-tier rate limits.
- **Adding a new source is low-cost.** A new connector requires one class (extending `PollConnector` or `CheckpointedConnector`), one factory registry entry, one credentials environment variable, and one CronJob manifest. No changes to the bridge, indexer, or RAG agents.
- **Existing RAG agents work without modification.** Connector data lands in the same collection with compatible metadata, so FastSearch, DeepResearch, and Oracle see it automatically.
- **Failure isolation is strong.** A broken connector does not affect vault indexing or other connectors. Kafka decouples the producer (bridge) from the consumer (indexer). DLQ captures terminal failures for later replay.
- **Idempotent upserts prevent duplicates.** Deterministic UUID5 point IDs mean re-processing a document overwrites the same chunks, not appends new ones.

### Negative

- **Onyx shim is a maintenance burden.** The shim (`onyx_shim/`) reproduces ~35 files from the Onyx codebase. Upstream changes (Onyx releases new connector features or breaking API changes) must be manually ported. The shim is intentionally thin — it provides interfaces, models, and utilities but not the full Onyx runtime — but it is still third-party code embedded in the repository.
- **Kafka dependency deepens.** The connector pipeline requires Kafka for the producer-consumer decoupling. This is acceptable given Kafka is already deployed for the vault delta pipeline and DLQ processing, but it increases the blast radius of Kafka outages.
- **Single Qdrant collection may hit scale limits.** Mixing vault and connector data in `rootweaver_vault` works at current scale (~150K chunks) but may require partitioning (separate collections per source, or Qdrant's upcoming partition feature) if connector data grows by an order of magnitude.
- **Credential management is per-connector.** Each connector reads credentials from a dedicated environment variable (`CONNECTOR_JIRA_CREDENTIALS`, etc.), which must be provisioned in K8s secrets and mounted into CronJob manifests. There is no unified credential store.

### Neutral

- **Connector activation is opt-in.** Only connectors with configured credentials are instantiated (`factory.py` returns `None` for unconfigured sources). This means deploying the framework does not immediately increase API calls or Qdrant storage.
- **The Onyx `Document` model is richer than `ConnectorDoc`.** Sections, hierarchy nodes, and permission sync data are deliberately dropped at the bridge layer. If future requirements need these (e.g., per-document ACLs), the bridge conversion would need to be extended rather than the downstream pipeline.

## Alternatives Considered

### Alternative A: Deploy Onyx as a Standalone Service

Run the full Onyx (Danswer) application as a separate K8s deployment, using its built-in connector framework, scheduling, and Vespa-backed search. Rootweaver would query Onyx's API instead of Qdrant for non-vault sources.

**Rejected because:**

- Onyx requires Vespa, Postgres, Redis, and its own API server — a substantial infrastructure addition for a single-tenant homelab deployment already running Qdrant, Kafka, and Postgres.
- Search results would be split across two backends (Qdrant for vault, Vespa for connectors), requiring a federation layer in the RAG agents. The current architecture's single-collection model is simpler and produces better-ranked results from a unified embedding space.
- Onyx's multi-tenant permission model and admin UI add complexity with no benefit for a single-user system.

### Alternative B: Direct API Polling Without Kafka

Each connector writes directly to Qdrant (chunk, embed, upsert in the same process), bypassing Kafka entirely. The CronJob runs the connector and the indexer in a single invocation.

**Rejected because:**

- **No failure isolation.** A Qdrant timeout during upsert would lose all documents fetched in the current poll window. With Kafka, the producer succeeds independently of the consumer; failed upserts are retried or routed to DLQ.
- **No backpressure.** If a connector produces documents faster than Qdrant can index them (e.g., initial full load of thousands of Jira issues), the process would either OOM or need complex internal buffering. Kafka provides natural backpressure via consumer lag.
- **No replay.** Without Kafka's retention, a bug in the chunking or embedding logic requires re-fetching from the source API. With Kafka, the raw `ConnectorDoc` messages can be replayed from the topic.
- **Tighter coupling.** The connector process would need direct access to the embedding service and Qdrant, adding environment variables and network dependencies to every CronJob. The current separation keeps the connector CronJob lightweight (only needs Kafka and source API access).

### Alternative C: File-Based Staging (Write Markdown, Let Vault Indexer Handle It)

Each connector writes its output as Markdown files to a staging directory inside the vault. The existing vault indexer picks them up, chunks, embeds, and indexes them like any other vault file.

**Rejected because:**

- **Impedance mismatch.** Jira issues and Confluence pages do not naturally map to Markdown files. Converting to Markdown loses structured metadata (issue type, status, priority, assignee) that is valuable for filtered search. The `ConnectorDoc` model preserves this metadata as first-class fields.
- **Filesystem pollution.** Thousands of auto-generated Markdown files in the vault would clutter the Obsidian UI and bloat Syncthing sync between desktop and MacBook.
- **No incremental delete.** If a Jira issue is deleted or a Confluence page is unpublished, the corresponding Markdown file would remain in the vault indefinitely. The watermark-based pipeline can detect missing documents via tombstone messages or full-sync reconciliation.
- **Vault indexer exclusion list complexity.** The vault indexer's exclusion logic (`shared_constants.py`) would need careful extension to avoid indexing staging files prematurely, reintroducing the data-leak risk class from VW-107.
