# ADR-029: KEDA-Based GPU Swap Controller for Embed/Inference Coexistence

**Status**: Accepted (retroactive — ratifies the deployed VW-233 design)
**Date**: 2026-04-29
**Jira**: [VW-233](https://ryanduffyuk.atlassian.net/browse/VW-233) (controller design), [VW-237](https://ryanduffyuk.atlassian.net/browse/VW-237) (health alerts), [VW-232](https://ryanduffyuk.atlassian.net/browse/VW-232) (scaling target rewire to TEI)
**Related ADRs**: ADR-034 (Data-Source Connector Architecture — Kafka lag signal source), ADR-036 (Vault Structured Index Postgres — dual-write tier whose embedding model alignment drives the choice of TEI)

---

## Context

### Hardware constraint

The Rootweaver platform runs on a single-node K3s cluster with one NVIDIA RTX 4080 (16 GB VRAM). Three GPU workloads contend for this single card:

1. **vLLM inference** (DeepSeek-R1-0528-Qwen3-8B-AWQ) — the primary LLM serving agent routing, OracleAgent synthesis, SRE Brain triage, and the architecture-review scoring pipeline. Its VRAM footprint is approximately 15.6 GB: model weights occupy ~5 GB (AWQ 4-bit), with the remainder consumed by a 115K-token KV cache and PyTorch's CUDA allocator overhead (`gpu_memory_utilization=0.95`). This pins essentially the entire GPU.

2. **TEI embedding** (Qwen3-Embedding-4B, 2560-dimensional) — the production embedder whose vector dimension matches the Qdrant `rootweaver_vault` collection. TEI requires approximately 8 GB VRAM transiently when loading the model and serving batches. It has two trigger patterns: the deterministic `vault-nightly-reindex` CronJob (daily 03:00–05:00 UTC) and the stochastic `connector-indexer` consumer (VW-232), which fires whenever Jira deltas land on the `vault.connector-docs` Kafka topic.

3. **vllm-embedding** (jina-embeddings-v4-vllm-code) — a code-oriented embedder retained for the FalkorDB code-knowledge graph. Currently dormant (FluxCD pins replicas to 0), but shares the same PriorityClass as TEI so it can be reactivated without manifest changes.

The first two **cannot coexist** on 16 GB. vLLM's allocator claims the full card at startup; TEI cannot load even a partial model into the remaining scraps of VRAM. This is not a scheduling problem — it is an arithmetic one.

### HAMi ruled out

HAMi (Heterogeneous AI computing Management Interface) was evaluated and rejected during VW-233 design work (2026-04-27). HAMi implements fractional GPU sharing via CUDA MPS or kernel-level time-slicing, both of which require headroom for concurrent tenants. With vLLM consuming ~15.6 of 16 GB, there is no fraction to allocate to a second process. HAMi remains suitable for smaller co-tenants (it is what allocates the 16 GB to vLLM today), but it cannot solve the embed/inference coexistence problem on a 16 GB card. Reconsideration requires either (a) shrinking vLLM's footprint by >5 GB (e.g., a smaller model or aggressive KV-cache pruning) or (b) upgrading to a ≥24 GB GPU.

### Manual swap and its failure modes

The historical workaround was a manual operator choreography: before a nightly reindex or ad-hoc embedding burst, scale vLLM to 0, scale TEI to 1, wait for model load, process the work, reverse. This pattern had two recurring production failures:

- **Forgotten scale-down**: TEI left running after an embedding window, vLLM unable to reclaim the GPU. The RAG gateway returned 503s for all inference-dependent requests (OracleAgent, SRE Brain, architecture-review) until the operator (Ryan) noticed and intervened. Median detection time: ~45 minutes, driven by when the next interactive query hit the system.

- **Ad-hoc ingest blocked**: The Jira connector (VW-188, go-live 2026-04-26) produces embedding requests in bursts whenever `connector-jira-scheduler` syncs ticket deltas. With TEI scaled to 0 outside the nightly window, these requests queued indefinitely on `vault.connector-docs`. The operator had to notice the Kafka lag, manually execute the swap, and reverse it. Round-trip: 3–5 minutes per burst.

VW-188's go-live transformed ingest bursts from a quarterly manual event into a routine occurrence (every 15 minutes when Jira has changes). The manual model became untenable.

### Requirements driving the decision

1. **Automatic** — no operator intervention for routine embedding bursts.
2. **Mutually exclusive** — vLLM and TEI must never hold the GPU simultaneously (CUDA OOM crashes both).
3. **Bounded swap latency** — embedding requests should begin processing within 90 seconds of trigger.
4. **Observable** — swap events, durations, and stuck states must surface in Prometheus/Grafana.
5. **Reversible** — the mechanism must be removable when a second GPU becomes available.

## Decision

Deploy a **KEDA + Kubernetes PriorityClass GPU swap controller** — not a custom controller, but a composition of stock Kubernetes primitives orchestrated by KEDA and FluxCD:

### Components

| Object | Kind | GitOps location | Role |
|--------|------|-----------------|------|
| `vllm-embedding-priority` | PriorityClass (value 1000) | `infrastructure/vllm/vllm-priorityclasses.yaml` | High priority, `PreemptLowerPriority`. Assigned to TEI Deployment and the dormant `vllm-embedding` StatefulSet. |
| `vllm-llm-priority` | PriorityClass (value 100) | (same file) | Low priority, `PreemptLowerPriority`. Assigned to `vllm-0` (LLM inference). |
| `tei-embedding` | KEDA ScaledObject | `infrastructure/rootweaver/tei-embedding-scaledobject.yaml` | Scales TEI Deployment between 0 and 1 replica. `pollingInterval=30s`, `cooldownPeriod=300s`, `fallback.replicas=0` after 3 consecutive trigger failures. |
| GPU swap alerts | PrometheusRule | `infrastructure/monitoring/prometheus-rules.yaml` | VW-237 alert family: `KEDATEIStuckWhileKafkaLag`, `KEDATEIScalerFailure`, `vLLMSwapNotRecovering`. |

### Trigger configuration (OR semantics)

The `tei-embedding` ScaledObject scales TEI to 1 replica if **either** trigger fires:

1. **Kafka lag** — consumer-group `connector-indexer` on topic `vault.connector-docs` exceeds `lagThreshold: 50`. This drives ad-hoc embedding bursts, most commonly from `connector-jira-scheduler` producing batches of Jira issue updates every 15 minutes.

2. **Cron window** — daily 03:00–05:00 UTC, matching the `vault-nightly-reindex` CronJob schedule. This guarantees TEI is available during the bulk vault re-embedding window even when there is no Kafka lag signal.

Both triggers OR together. The `fallback.replicas: 0` setting means that if KEDA cannot reach Kafka (e.g., the bootstrap broker is down), TEI stays scaled to 0 rather than preempting vLLM unnecessarily.

### Swap state machine

```
Idle state:
  vllm-0 (priority=100) holds GPU. TEI Deployment at 0 replicas.

Trigger fires (Kafka lag > 50 OR cron window):
  KEDA scales TEI 0 → 1.
  TEI pod enters Pending, requests nvidia.com/gpu.
  Scheduler sees vllm-0 (priority 100) holding the GPU.
  TEI (priority 1000, PreemptLowerPriority) outranks → vllm-0 evicted.
  TEI scheduled, loads Qwen3-Embedding-4B (~30s model load).
  TEI serves embedding traffic on port 8090.

Trigger clears (Kafka lag returns to 0 AND outside cron window):
  cooldownPeriod (300s) elapses with no re-trigger.
  KEDA scales TEI 1 → 0. GPU released.
  vllm-0 rescheduled, reloads DeepSeek-R1-AWQ (~60s cold start).
```

No custom controller binary is involved. The preemption ordering is stock Kubernetes scheduler behaviour driven by PriorityClass values. KEDA manages only the replica count; the scheduler handles eviction.

### FluxCD coexistence

TEI's Deployment carries the annotation `kustomize.toolkit.fluxcd.io/ssa-ignore: "spec.replicas"`, which tells FluxCD to skip server-side-apply on the replicas field. Without this, FluxCD's 5-minute reconciliation loop would fight KEDA's scaling decisions — resetting TEI to the manifest-declared 0 replicas even while KEDA has it at 1.

### Retry coupling with connector-indexer

The `connector-indexer` ScaledJob is configured with `RETRY_TRIES=5` and `RETRY_BASE_DELAY=2s`, yielding approximately 30 seconds of exponential backoff. This budget is deliberately sized to absorb TEI's ~30s cold-start model load. The first few embedding HTTP calls after a scale-from-zero will get connection-refused or timeout errors; the retry loop absorbs them until TEI's readiness probe passes. Future tuning of either the retry budget or TEI's startup time must consider the coupling.

### NetworkPolicy surface

TEI ingress on port 8090 must be reachable from three consumers: `connector-indexer`, `arch-review-extractor`, and `vault-nightly-reindex`. VW-233 updated the NetworkPolicy manifests (commit `19d74f0`) to add explicit egress allow rules for each consumer targeting TEI. This coupling is load-bearing — repointing a KEDA scaler to a different embedding backend always implies a NetworkPolicy audit (a lesson learned the hard way: the initial VW-232 repointing from `vllm-embedding` to TEI left the old egress rules pointing at the wrong pod identity, causing silent embed failures until the policy was fixed).

## Consequences

### Positive

- **No idle VRAM cost.** TEI runs at 0 replicas when there is no embedding work. ~8 GB VRAM reclaimed for inference 90%+ of the day.

- **Bounded inference outage.** The only unavoidable cost is vLLM's ~60s cold start when embedding bursts arrive. SRE Brain and agent routing fall back to Ollama on CPU during the gap, so the system remains nominally available for lower-fidelity inference. The `vault-nightly-reindex` window swaps twice per day (once to TEI, once back); ad-hoc bursts cluster temporally, so the swap fires once per burst, not per request.

- **Workload-driven, not operator-driven.** Eliminates the "forgot to scale down" failure mode. The Kafka lag trigger reacts within 30–60 seconds of new embedding work arriving (KEDA poll interval + scheduler latency). The cron trigger guarantees the nightly window without operator scheduling.

- **Compatible with existing KEDA practice.** The cluster already uses KEDA ScaledJobs for `connector-indexer`, `vault-delta-indexer`, `ner-consumer`, and the `arch-review-*` fleet. The ScaledObject for TEI is the same pattern extended to a Deployment rather than a Job.

- **Reversible.** When a second GPU becomes available, the PriorityClass annotations can be removed and TEI pinned permanently to the embedding GPU. No code changes required, only manifest edits.

### Negative / accepted tradeoffs

- **vLLM cold-start latency on every swap (~60s).** Acceptable because: (a) embedding bursts cluster temporally, (b) the nightly window swaps at most twice, (c) Ollama provides a CPU fallback, and (d) the 300s cooldown prevents thrash on bursty Kafka traffic.

- **Single point of GPU contention.** By design, only one of {vLLM, TEI, vllm-embedding} can hold the GPU at any time. Adding a fourth GPU workload would require a second physical GPU or a more sophisticated scheduling layer.

- **KEDA as a critical dependency.** If the KEDA operator pod is down, TEI never scales up and embedding requests queue indefinitely. Mitigated by the `KEDATEIScalerFailure` alert (fires when `keda_scaler_errors` increases) and by the fact that manual `kubectl scale` on the ScaledObject's `fallback.replicas` field remains available as an escape hatch.

- **Observability coupling.** The VW-237 alert family (3 alerts: `KEDATEIStuckWhileKafkaLag`, `KEDATEIScalerFailure`, `vLLMSwapNotRecovering`) depends on kube-state-metrics and the KEDA metrics server being healthy. If `dcgm-exporter` is in CrashLoopBackOff (which it was as of 2026-04-29), swap state is observable only via Kafka consumer lag — not GPU metrics directly.

- **NetworkPolicy maintenance.** Every new consumer of TEI must have its egress policy updated. This is a recurring cost, not a one-time setup.

### Operational notes

- **Manual override**: `kubectl scale` on TEI is discouraged — KEDA will revert within one polling interval. To force TEI up for an extended period, edit the ScaledObject's `fallback.replicas` field or temporarily widen the cron trigger window.

- **Model alignment is load-bearing**: TEI must serve the same embedding model as the Qdrant `rootweaver_vault` collection (currently Qwen3-Embedding-4B, 2560d). Changing one without the other silently produces dimensionality-mismatched vectors. This is why TEI replaced the prior `vllm-embedding` ScaledObject for VW-232: the Qdrant collection was migrated to 2560d, requiring TEI's Qwen3-Embedding-4B rather than the old jina-v4-code model.

## Alternatives Considered

### A. HAMi fractional GPU sharing

**Rejected.** vLLM's ~15.6 GB footprint and TEI's ~8 GB requirement sum to ~23.6 GB — exceeding the 16 GB card by 47%. HAMi cannot conjure VRAM that does not exist. Neither CUDA MPS nor kernel time-slicing helps: MPS memory pinning is advisory (PyTorch ignores it and OOMs), and time-slicing introduces latency jitter that degrades both vLLM token generation (p99 target <200ms) and TEI embedding throughput. HAMi was evaluated during VW-233 design (2026-04-27) and ruled out for this specific pair. It remains appropriate for smaller co-tenants.

### B. Pure cron scheduling (no Kafka trigger)

**Rejected.** Adequate for the deterministic nightly reindex, but VW-232's `connector-indexer` produces ad-hoc bursts whenever Jira ticket deltas arrive. With a 24-hour cron grain, embedding requests from the Jira connector would queue for hours between windows. The Kafka lag trigger provides sub-minute reactivity to actual workload demand.

### C. Run TEI on CPU

**Rejected.** Qwen3-Embedding-4B at 2560d on CPU is approximately 10x slower than GPU (measured during the 2026-03-18 model migration; reference: `2026-03-18-VLLm-Model-Swap-DeepSeek-R1` vault doc). At CPU speed, the nightly reindex of ~148K chunks would take approximately 6–7 hours, far exceeding the 2-hour window. On-demand Jira connector latency would degrade from ~2s/document to ~20s/document, making interactive workflows impractical. A smaller CPU-feasible model (e.g., nomic-embed-text at 768d) was also rejected: benchmark Run-38 showed Qwen3-Embedding-4B outperforming it by +0.12 context precision and +0.08 MRR on the platform's evaluation suite. CPU embedding remains the DLQ-processor's fallback path, but not the primary one.

### D. Add a second GPU

**Rejected (for now).** The natural long-term solution, but the desktop is a single-slot consumer motherboard. A second GPU would require a PCIe riser (thermal and stability risk) or a hardware upgrade. The KEDA+PriorityClass design is deliberately reversible: when a second GPU arrives, the priority annotations are dropped and TEI is pinned to the dedicated embedding card. No code changes required.

### E. Inline embedding inside vLLM (model swap within the same pod)

**Rejected.** vLLM's runtime is optimised for serving a single model with a persistent KV cache. Hot-swapping weights in-process would not materially reduce the ~60s cold-start cost (dominated by model deserialization and CUDA memory allocation), and it would couple two workload lifecycles — an embedding failure would take down inference, and vice versa.
